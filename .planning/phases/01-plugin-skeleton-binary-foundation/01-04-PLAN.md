    - Wenn noch kein SpotOn librespot-Fork existiert: git clone https://github.com/librespot-org/librespot.git /tmp/librespot-spoton und checkout tag v0.8.0
    - HINWEIS: Das vollstaendige Portieren von spotty.rs nach spoton.rs ist Rust-Arbeit die ausserhalb dieses Plans liegt (per RESEARCH.md: "1-2 Tage Rust-Arbeit"). Fuer Phase 1 ist ein Minimal-Binary ausreichend, das den --check-Vertrag erfuellt.
    - Wenn der SpotOn-Fork noch keinen --check Support hat, erstelle ein Minimal-Rust-Programm das den --check-Vertrag erfuellt:
      - Bei Aufruf mit --check: Gibt "ok spoton v0.1.0" auf der ersten Zeile aus, gefolgt von einem Newline und einem JSON-Objekt mit mindestens {"version":"0.1.0","lms-auth":false,"ogg-direct":false}
      - Bei Aufruf mit -n: Akzeptiert den Parameter (ignoriert ihn)
      - Sonst: Gibt Usage-Info aus
    - cargo build --release --target x86_64-unknown-linux-musl
    - Kopiere das gebaute Binary nach Plugins/SpotOn/Bin/x86_64-linux/spoton
    - chmod 755 Plugins/SpotOn/Bin/x86_64-linux/spoton
    - Verifiziere: Plugins/SpotOn/Bin/x86_64-linux/spoton --check gibt "ok spoton v..." + JSON aus
    - .gitkeep in Plugins/SpotOn/Bin/x86_64-linux/ entfernen (Binary ersetzt Placeholder)

    TEIL B: GitHub Actions Workflow fuer ARM-Cross-Compilation

    Erstelle .github/workflows/build-librespot.yml:

    Name: "Build librespot binaries"
    Trigger: workflow_dispatch (manuell) + push auf Tags "v*"

    Matrix-Strategy mit 4 Targets:
    - aarch64-unknown-linux-musl -> Bin/aarch64-linux/spoton
    - armv7-unknown-linux-musleabihf -> Bin/armhf-linux/spoton
    - arm-unknown-linux-musleabi -> Bin/arm-linux/spoton
    - i686-unknown-linux-musl -> Bin/i386-linux/spoton

    Steps pro Target:
    1. actions/checkout@v4
    2. dtolnay/rust-toolchain@stable
    3. cross-rs/cross installieren (cargo install cross)
    4. cross build --release --target ${{ matrix.target }}
    5. Binary umbenennen auf 'spoton' und in Bin/{arch}/ ablegen
    6. chmod 755
    7. actions/upload-artifact@v4 mit Binary als Artifact

    Zusaetzlicher Job: release (needs: build)
    - Nur bei Tag-Push
    - Download aller Artifacts
    - Upload als GitHub Release Assets

    Der Workflow setzt voraus, dass der Rust-Source fuer das spoton-Binary im Repository liegt oder als Submodule referenziert wird. Workflow-Pfade fuer Source muessen an die tatsaechliche Repo-Struktur angepasst werden.
