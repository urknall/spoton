<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| TokenManager -> Spotify API | Outbound HTTPS POST to accounts.spotify.com/api/token; untrusted response |
| TokenManager -> Slim::Utils::Cache | PKCE state + verifier stored temporarily; access tokens stored with TTL |
| TokenManager -> Slim::Utils::Prefs | Refresh tokens stored persistently on disk (YAML) |
| Browser -> LMS | OAuth flow initiated from Settings page (covered in Plan 03) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.1-01 | Spoofing | PKCE code_verifier | mitigate | Use Crypt::OpenSSL::Random::random_bytes(32) for cryptographic randomness; never use rand() or srand() |
| T-02.1-02 | Spoofing | OAuth state parameter | mitigate | Generate state with random_bytes(16); cache with TTL=600s; validate and consume on callback (one-time use) |
| T-02.1-03 | Information Disclosure | Token logging | mitigate | Never log access_token, refresh_token, or authorization code values; log only account IDs and status codes per T-02-10 convention |
| T-02.1-04 | Information Disclosure | Refresh token storage | mitigate | Store refresh_token in Slim::Utils::Prefs (YAML on disk, LMS-managed permissions); NOT in Cache (TTL eviction would cause user logout per Pitfall 4) |
| T-02.1-05 | Tampering | Token exchange response | mitigate | Validate response contains access_token field before storing; handle missing/malformed JSON gracefully |
| T-02.1-06 | Information Disclosure | code_verifier in cache | mitigate | Cache with TTL=600s; remove after consumption in callback handler |
| T-02.1-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this phase — all modules are LMS-bundled Perl core |
</threat_model>
