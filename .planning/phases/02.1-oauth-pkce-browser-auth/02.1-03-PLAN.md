<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Settings.pm | User submits Client-ID and startOAuth form; untrusted input from POST params |
| Settings.pm -> Spotify | HTTP 302 redirect to accounts.spotify.com; auth URL contains client_id and PKCE params |
| Plugin.pm -> Callback.pm | Route initialization at startup; no untrusted data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.1-11 | Tampering | Client-ID input | accept | Client-ID is not a secret in PKCE (by design); stored in Prefs on disk. User controls their own Spotify App. No validation beyond non-empty needed. |
| T-02.1-12 | Spoofing | startOAuth action | mitigate | Settings handler is protected by protectURI (CSRF via LMS); only authenticated LMS sessions can trigger startOAuth |
| T-02.1-13 | Information Disclosure | Redirect URI exposure | accept | Redirect URI is http://127.0.0.1:{port}/... — loopback only, not externally routable. Displayed to user for Spotify App registration. |
| T-02.1-14 | Tampering | removeAccount path traversal | mitigate | Existing WR-03 validation retained: removeId must match /\A[0-9a-f]{8}\z/ and exist in accounts hash |
| T-02.1-15 | Tampering | switchAccount injection | mitigate | Existing WR-04 validation retained: switchId must exist in accounts hash |
| T-02.1-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan |
</threat_model>
