# Phase 02.1 Plan 04: Gap Closure — Display Name + REQUIREMENTS.md Alignment Summary

**One-liner:** OAuth PKCE callback success page now shows user display name via Prefs accounts hash lookup; REQUIREMENTS.md updated from Keymaster/login5 to OAuth 2.0 PKCE.

## What Was Built

### Task 1: Display name in success callback page (SC-2)

`Plugins/SpotOn/Settings/Callback.pm` was updated to show the authenticated user's Spotify display name on the success page:

1. Added `use Slim::Utils::Prefs` and `my $prefs = preferences('plugin.spoton')` at module level.
2. Extended `_renderResult` signature from 7 to 8 parameters: added `$displayName` between `$response` and `$success`.
3. Success HTML: conditionally renders `<p>Verbunden als: <strong>Test User</strong></p>` when `$displayName` is defined and non-empty. XSS protection via `_html_escape()` (T-02.1-GC-01 mitigated).
4. Success branch of `exchangeCode` callback: reads `$prefs->get('accounts')->{$accountId}{displayName}` and passes it to `_renderResult`.
5. All error-path `_renderResult` calls updated to pass `undef` as `$displayName`.

`t/10_pkce_callback.t` updated with 3 new assertions (15 total, up from 12):
- TokenManager stub seeds `accounts` pref with `displayName => 'Test User'` before calling `$cb`.
- Test 4 (SC-2): asserts `like($$html_ref, qr/Verbunden als:.*Test User/s)`.
- Test 7 (new): asserts success page omits "Verbunden als:" when displayName is empty string.
