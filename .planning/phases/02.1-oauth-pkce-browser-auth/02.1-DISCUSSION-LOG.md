## Motivation & Koexistenz

### Ausloeser

| Option | Description | Selected |
|--------|-------------|----------|
| Keymaster-Probleme | login5/Keymaster ist unzuverlaessig oder wird von Spotify blockiert | ✓ |
| Nutzer-Wunsch | User wollen kein Passwort eingeben | |
| Strategische Absicherung | PKCE als Fallback fuer Zukunftssicherheit | |

**User's choice:** Keymaster-Probleme
**Notes:** login5 akzeptiert kein Username/Passwort mehr — verifiziert mit echten Credentials.

### Koexistenz-Modell

| Option | Description | Selected |
|--------|-------------|----------|
| PKCE ersetzt Keymaster | Keymaster-Code wird entfernt | ✓ |
| PKCE primaer, Keymaster Fallback | Beide Wege, Keymaster fuer headless | |
| Beide gleichberechtigt | User waehlt in Settings | |

**User's choice:** PKCE ersetzt Keymaster
**Notes:** Sauberer Schnitt — kein Dual-Auth-System.

### Binary-Rolle

| Option | Description | Selected |
|--------|-------------|----------|
| Binary nur fuer Playback | Kein --authenticate, kein --get-token | |
| --get-token bleibt fuer Connect | PKCE fuer Web-API, Binary fuer Connect | |
| Noch unsicher | Research klaert | |

**User's choice:** Freetext — "Accounts werden ueber Username/Passwort in Settings hinzugefuegt -> das hat nicht funktioniert weil das login5 Protokoll kein Username Passwort akzeptiert"
**Notes:** Bestaetigt dass das Problem technisch ist, nicht UX-getrieben. Research soll klaeren ob --get-token noch nutzbar ist.

### Verifikation: login5 wirklich kaputt?

| Option | Description | Selected |
|--------|-------------|----------|
| Mit echten Credentials getestet | Passwort stimmt, aber "Bad credentials" | ✓ |
| Noch nicht selbst getestet | Basiert auf Berichten | |
| Jetzt testen | Live-Test durchfuehren | |

**User's choice:** Mit echten Credentials getestet
**Notes:** Binary-Test vor Ort lieferte "Authentication failed: Permission denied { Login failed with reason: Bad credentials }" — login5 ist definitiv kaputt.
