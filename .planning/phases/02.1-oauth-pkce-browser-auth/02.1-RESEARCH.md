## Summary

Phase 02.1 replaces the non-functional Keymaster/login5 auth (login5 is deactivated by Spotify; verified by actual test) with a standard OAuth 2.0 Authorization Code + PKCE flow. The core challenge is browser round-trip integration within an LMS plugin: the user must be redirected to `accounts.spotify.com`, authenticate, then Spotify redirects back to LMS carrying the authorization code.

**Redirect URI strategy (D-06):** LMS itself can serve as the callback target. `Slim::Web::Pages->addPageFunction('plugins/SpotOn/settings/callback', \&handler)` registers a route at `http://127.0.0.1:{httpPort}/plugins/SpotOn/settings/callback`. Spotify allows HTTP for loopback addresses (`127.0.0.1`, not `localhost`). This avoids any external proxy service, which Spotty requires but SpotOn does not need. [VERIFIED: Spotify docs, LMS source]

**Callback capture (D-08):** A dedicated `addPageFunction` route (not the Settings handler) receives the `?code=...&state=...` redirect from Spotify. The Settings handler cannot receive this redirect because it is protected by CSRF (`protectURI`) and its URL does not match the registered redirect URI. The callback route is GET-only, has no CSRF requirement, and immediately exchanges the code for tokens via `SimpleAsyncHTTP`. [VERIFIED: LMS source]

**Token storage (D-12):** Refresh tokens go into `Slim::Utils::Prefs` (YAML on disk, persistent across LMS restarts). Access tokens go into `Slim::Utils::Cache` (SQLite, also persistent, with TTL). This is the correct architectural split: refresh tokens must survive restarts; access tokens are short-lived and benefit from TTL eviction. [VERIFIED: LMS source]

**D-03 / --get-token fate:** The `--get-token` flag reads cached librespot credentials from disk. After Spotify deactivated login5, there are no valid credentials for it to read. The flag is non-functional as an auth mechanism. However, the binary retains its Playback role (`--single-track`, Connect daemon), which does not require credentials from LMS — Connect daemons authenticate independently through Spotify's ZeroConf/mDNS flow. The `--get-token` path can simply be deleted from TokenManager. [CITED: librespot CLI docs, Spotty-Plugin reference]

**Primary recommendation:** Implement PKCE fully in Perl with LMS-bundled modules (`Digest::SHA`, `MIME::Base64::encode_base64url`, `Crypt::OpenSSL::Random`). Use `http://127.0.0.1:{httpPort}/plugins/SpotOn/settings/callback` as redirect URI. Register callback with `Slim::Web::Pages->addPageFunction`. Store refresh token in Prefs; access token in Cache.

### Pattern 1: LMS Route Registration for OAuth Callback

**What:** Register a custom URL handler that receives Spotify's redirect (GET with `?code=&state=`).
**When to use:** Any page not managed by `Slim::Web::Settings` — i.e. a callback that Spotify (not the user) navigates to.

```perl
# Source: /usr/share/perl5/Slim/Web/Pages.pm — addPageFunction
# In Settings/Callback.pm (called from Settings.pm::new() or initPlugin)

use constant CALLBACK_PATH => 'plugins/SpotOn/settings/callback';

sub init {
    Slim::Web::Pages->addPageFunction(CALLBACK_PATH, \&handler);
}

# Handler signature matches what LMS passes to all page functions:
# ($client, $paramRef, $callback, $httpClient, $response)
# $paramRef contains all GET query parameters (code, state, error)
sub handler {
    my ($client, $paramRef, $callback, $httpClient, $response) = @_;

    my $code  = $paramRef->{code}  // '';
    my $state = $paramRef->{state} // '';
    my $error = $paramRef->{error} // '';

    # ... validate state, exchange code for tokens ...
}
```

**Key insight:** `addPageFunction` registers a regexp match on the path. The handler receives `$paramRef` with all GET parameters already parsed by LMS. No manual URL parsing needed.

### Pattern 2: PKCE Code Verifier + Challenge Generation

**What:** Generate cryptographically random code_verifier; derive SHA-256 code_challenge.
**When to use:** Once per OAuth flow initiation, before redirecting to Spotify.

```perl
# Source: verified via direct execution on target system
# Modules: Crypt::OpenSSL::Random, MIME::Base64, Digest::SHA (all LMS-bundled)

use Crypt::OpenSSL::Random;
use MIME::Base64 qw(encode_base64url);
use Digest::SHA qw(sha256);

sub _generatePkce {
    # Generate 32 random bytes -> 43-char base64url verifier (within PKCE spec 43-128)
    my $random_bytes  = Crypt::OpenSSL::Random::random_bytes(32);
    my $code_verifier = encode_base64url($random_bytes);
    $code_verifier    =~ s/=+$//;  # strip base64 padding per PKCE spec

    # SHA-256 of verifier, URL-safe base64 encoded
    my $code_challenge = encode_base64url(sha256($code_verifier));
    $code_challenge    =~ s/=+$//;

    return ($code_verifier, $code_challenge);
}
```

**Verified output:** verifier length 43, challenge length 43, charset `[A-Za-z0-9-_]` only. [VERIFIED: tested on target system]

### Pattern 3: OAuth State Parameter (CSRF Protection)

**What:** Generate a random state value, cache it server-side, verify on callback.
**When to use:** Every OAuth flow initiation.

```perl
sub _generateState {
    my $raw   = Crypt::OpenSSL::Random::random_bytes(16);
    my $state = encode_base64url($raw);
    $state    =~ s/=+$//;
    return $state;
}

# Cache verifier + state together, keyed by state (TTL = 10 minutes)
$cache->set("spoton_pkce_$state", {
    code_verifier => $code_verifier,
    account_id    => $accountId,    # if re-auth for existing account
    client_id     => $clientId,
}, 600);

# In callback handler:
my $pkce_data = $cache->get("spoton_pkce_$state");
unless ($pkce_data) {
    # State invalid or expired — reject
}
$cache->remove("spoton_pkce_$state");  # consume once
```

### Pattern 4: Redirect URI Construction

**What:** Build the redirect URI using LMS's own server address and httpport.
**When to use:** Once per OAuth flow. Must match exactly what is registered in the Spotify Developer App.

```perl
# Source: /usr/share/perl5/Slim/Utils/Network.pm::serverURL()
# serverURL() returns http://serverAddr():httpport — but serverAddr() may return
# the LAN IP (e.g. 192.168.x.x), which Spotify will reject for HTTP.
# Always use 127.0.0.1 explicitly for the redirect_uri.

use Slim::Utils::Prefs;

sub _buildRedirectUri {
    my $httpPort = preferences('server')->get('httpport') || 9000;
    return "http://127.0.0.1:${httpPort}/plugins/SpotOn/settings/callback";
}
```

**Why 127.0.0.1, not `serverURL()`:** Spotify's 2025 policy prohibits `localhost` and requires loopback IP literals. `serverURL()` may return the LAN IP (which requires HTTPS) or `localhost` (now prohibited). Using `127.0.0.1:{httpPort}` is always valid. [CITED: developer.spotify.com/documentation/web-api/concepts/redirect_uri]

**What the user must register in Spotify Developer Console:** `http://127.0.0.1:9000/plugins/SpotOn/settings/callback` (with their actual LMS port). The Setup Wizard must show this exact string with the port filled in.

### Pattern 5: Spotify Auth URL Construction

```perl
# Source: developer.spotify.com/documentation/web-api/tutorials/code-pkce-flow
# URI module available (v5.27 verified on target)
use URI;
use constant SPOTIFY_AUTH_URL => 'https://accounts.spotify.com/authorize';
use constant SPOTIFY_TOKEN_URL => 'https://accounts.spotify.com/api/token';
use constant REQUIRED_SCOPES => join(' ', qw(
    user-read-playback-state
    user-modify-playback-state
    user-read-currently-playing
    user-read-recently-played
    user-read-private
    playlist-read-private
    user-library-read
    user-top-read
    streaming
));

sub _buildAuthUrl {
    my ($client_id, $redirect_uri, $code_challenge, $state) = @_;

    my $uri = URI->new(SPOTIFY_AUTH_URL);
    $uri->query_form(
        response_type         => 'code',
        client_id             => $client_id,
        redirect_uri          => $redirect_uri,
        code_challenge_method => 'S256',
        code_challenge        => $code_challenge,
        scope                 => REQUIRED_SCOPES,
        state                 => $state,
    );
    return $uri->as_string();
}
```

### Pattern 6: Token Exchange (POST to Spotify)

```perl
# Source: developer.spotify.com/documentation/web-api/tutorials/code-pkce-flow
# SimpleAsyncHTTP post() with body: last odd argument is content body
# Verified pattern from /usr/share/perl5/Slim/Networking/SimpleHTTP/Base.pm

use URI;

sub _exchangeCode {
    my ($class, $code, $verifier, $client_id, $redirect_uri, $cb) = @_;

    my $body_uri = URI->new('', 'https');
    $body_uri->query_form(
        grant_type    => 'authorization_code',
        code          => $code,
        redirect_uri  => $redirect_uri,
        client_id     => $client_id,
        code_verifier => $verifier,
    );
    my $body = $body_uri->query();

    Slim::Networking::SimpleAsyncHTTP->new(
        sub {
            my $http   = shift;
            my $result = eval { from_json($http->content) };
            $cb->($result, undef);
        },
        sub {
            my ($http, $error) = @_;
            $cb->(undef, $error);
        },
        { timeout => 30 }
    )->post(
        SPOTIFY_TOKEN_URL,
        'Content-Type' => 'application/x-www-form-urlencoded',
        $body,
    );
}
```

### Pattern 7: Token Refresh

```perl
# Source: developer.spotify.com/documentation/web-api/tutorials/refreshing-tokens
# grant_type=refresh_token, refresh_token=..., client_id=...
# Spotify MAY return a new refresh_token — if not present, keep existing one.

sub refreshToken {
    my ($class, $accountId, $cb) = @_;

    my $accounts     = $prefs->get('accounts') || {};
    my $account      = $accounts->{$accountId} || {};
    my $refreshToken = $account->{refreshToken};
    my $clientId     = $prefs->get('clientId');

    unless ($refreshToken && $clientId) {
        $cb->(undef);
        return;
    }

    my $body_uri = URI->new('', 'https');
    $body_uri->query_form(
        grant_type    => 'refresh_token',
        refresh_token => $refreshToken,
        client_id     => $clientId,
    );

    Slim::Networking::SimpleAsyncHTTP->new(
        sub {
            my $http   = shift;
            my $result = eval { from_json($http->content) };
            if ($result && $result->{access_token}) {
                # Update refresh token if Spotify issued a new one
                if ($result->{refresh_token}) {
                    $accounts->{$accountId}{refreshToken} = $result->{refresh_token};
                    $prefs->set('accounts', $accounts);
                }
                $class->_cacheToken($accountId, $result->{access_token}, $result->{expires_in});
                $cb->($result->{access_token});
            } else {
                $cb->(undef);
            }
        },
        sub { $cb->(undef) },
        { timeout => 30 }
    )->post(
        SPOTIFY_TOKEN_URL,
        'Content-Type' => 'application/x-www-form-urlencoded',
        $body_uri->query(),
    );
}
```

### Pattern 8: Callback Response (HTML page from addPageFunction)

The callback handler does NOT use `SUPER::handler` (no settings form, no template scaffolding). It builds its own HTML response:

```perl
# Source: Slim::Web::HTTP handler pattern — page functions return body string
# OR call the $callback with prepared $response and $body

sub handler {

    # In the SimpleAsyncHTTP success callback:

    $response->content_type('text/html');
    my $html = '<html><body><script>window.location="'
        . $paramRef->{webroot}
        . 'settings/plugin/spoton.html";</script>'

```

### Anti-Patterns to Avoid

- **Using `serverURL()` for redirect_uri:** Returns LAN IP or `localhost` — both problematic for Spotify. Always use `127.0.0.1:{httpPort}` explicitly.
- **Storing refresh_token in Slim::Utils::Cache:** Cache has TTL eviction; refresh tokens must persist indefinitely. Use Prefs.
- **Storing refresh_token as a top-level pref:** It belongs inside the per-account hash so multi-account works: `$accounts->{$accountId}{refreshToken}`.
- **Using `Slim::Web::HTTP::CSRF->protectURI` on the callback route:** The callback is called by Spotify's redirect, which has no CSRF token. Protecting it blocks all OAuth flows.
- **Blocking token exchange:** `SimpleAsyncHTTP` is mandatory. No `LWP` or backtick spawning for token exchange — LMS is single-threaded and any blocking call freezes audio.
- **Using `localhost` in redirect_uri:** Spotify prohibited this in November 2025. Only `127.0.0.1` works.
- **Caching the code_verifier in Slim::Utils::Cache without TTL:** Use a TTL (600s is sufficient for user interaction time). Unbounded storage is a mild info-leak risk if the server is compromised.
- **Not consuming the state after callback:** Remove `$cache->remove("spoton_pkce_$state")` after use to prevent replay.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| URL encoding for POST body | Manual string concatenation | `URI->new('', 'https'); $uri->query_form(...)` | Handles special characters, percent-encoding, `+` vs `%20` |
| URL construction for auth URL | String interpolation | `URI->new(SPOTIFY_AUTH_URL); $uri->query_form(...)` | Auto-encodes redirect_uri and scope with special chars |
| SHA-256 hashing | Manual bit operations | `Digest::SHA::sha256($input)` | Cryptographic correctness; available in LMS env |
| Cryptographic random bytes | `rand()` or `srand()` | `Crypt::OpenSSL::Random::random_bytes(N)` | PKCE spec requires cryptographically random; `rand()` is not |
| Base64url encoding | Manual `tr` on encode_base64 output | `MIME::Base64::encode_base64url($input)` | Correct padding removal; available v3.16+ |
| Async HTTP POST | Synchronous LWP or backtick | `Slim::Networking::SimpleAsyncHTTP->post()` | LMS single-threaded; any blocking HTTP freezes playback |

**Key insight:** The PKCE algorithm is not complex, but every step has a footgun. The modules available in LMS handle all of it correctly with less code than hand-rolling.

## D-03 Resolution: librespot --get-token Future

**Research finding:** [VERIFIED based on Spotty-Plugin reference and librespot docs]

`--get-token` reads cached librespot credentials from `{cache_dir}/credentials.json`. These credentials are the login5/Shannon-protocol session tokens that librespot establishes during the `--authenticate` flow. Since Spotify deactivated login5 authentication (`"Bad credentials"` on valid accounts), no valid `credentials.json` can be created. Even if old credentials exist on disk from a previous installation, they are likely invalidated server-side.

**Conclusion for D-03:**
- `--get-token` is non-functional and should be removed from TokenManager entirely.
- librespot's Playback role (`--single-track`) does NOT require credentials from LMS. In `--single-track` mode, librespot authenticates through a different mechanism tied to the Spotify session initiated by the Connect flow or via a token passed directly to the binary.
- Connect daemons authenticate independently via ZeroConf/mDNS discovery; LMS does not inject auth tokens into the Connect daemon invocation.
- [ASSUMED] Whether `--single-track` can accept an externally-provided access token (e.g., from PKCE) via a command-line flag needs verification against the actual binary. The Spotty reference implementation passes `--access-token` to the helper after code exchange — this pattern may be available in librespot 0.8.0.

## Common Pitfalls

### Pitfall 1: Redirect URI Mismatch
**What goes wrong:** OAuth flow returns `INVALID_CLIENT: Invalid redirect URI` even though the URI looks correct.
**Why it happens:** Spotify does an exact-string match. `http://127.0.0.1:9000/plugins/SpotOn/settings/callback` and `http://127.0.0.1:9000/plugins/SpotOn/settings/callback/` (trailing slash) are different. Also: port mismatch if the user changed the LMS HTTP port.
**How to avoid:** Show the exact redirect URI in the Setup Wizard with the actual port. Include a "Copy" button or direct link. Document that the string must be registered exactly.
**Warning signs:** `error=invalid_client` on callback, `state` present in callback URL.

### Pitfall 2: state Parameter Not Validated
**What goes wrong:** An attacker sends a forged callback to LMS with a valid `code` from a different OAuth session, injecting credentials for an account they don't own.
**Why it happens:** Skipping the `state` round-trip check.
**How to avoid:** Always validate `$paramRef->{state}` against the cache-stored state. Reject with 400 if not found.
**Warning signs:** Callback completes without a matching prior `startOAuth` action.

### Pitfall 3: Blocking Token Exchange
**What goes wrong:** LMS freezes for 1-3 seconds during code exchange; audio stutters or drops.
**Why it happens:** Using `LWP::UserAgent->post()` or a blocking socket in the callback handler.
**How to avoid:** All HTTP to `accounts.spotify.com` must go through `SimpleAsyncHTTP`.
**Warning signs:** LMS log shows warning about blocking operations; music pauses during auth.

### Pitfall 4: Refresh Token Eviction
**What goes wrong:** User is logged out after ~48 hours (Slim::Utils::Cache default cleanup cycle).
**Why it happens:** Storing refresh_token in Cache (SQLite with TTL) instead of Prefs (YAML, no expiry).
**How to avoid:** Refresh token goes in `$prefs->set('accounts', {accountId => {refreshToken => '...'}})`. Access token goes in Cache with TTL.
**Warning signs:** Users report needing to re-authenticate every 1-2 days.

### Pitfall 5: LMS Port in Redirect URI
**What goes wrong:** Redirect URI built with `Slim::Utils::Network::serverURL()` gives `http://192.168.1.x:9000/...`. This is a non-loopback HTTP URL — Spotify requires HTTPS for non-loopback.
**Why it happens:** Using `serverURL()` instead of hardcoding `127.0.0.1`.
**How to avoid:** Always build redirect URI as `"http://127.0.0.1:" . preferences('server')->get('httpport') . "/plugins/SpotOn/settings/callback"`.
**Warning signs:** `error=invalid_client` only when LMS is on a non-localhost IP.

### Pitfall 6: CSRF Protection on Callback Route
**What goes wrong:** Spotify's redirect is blocked with 403 Forbidden.
**Why it happens:** Registering the callback path with `protectURI()`, which marks it as a dangerous command requiring a CSRF token. Spotify's redirect doesn't carry LMS's CSRF token.
**How to avoid:** Use `addPageFunction` directly (not `protectURI`). The state parameter IS the CSRF protection for the OAuth flow.
**Warning signs:** Callback always returns 403; LMS log shows "failed CSRF Referer/token test".

### Pitfall 7: Refresh Token Not Updated
**What goes wrong:** Token refresh stops working after Spotify rotates the refresh token.
**Why it happens:** Spotify's refresh endpoint MAY return a new `refresh_token` (documented: "when a refresh token is not returned, continue using the existing token"). If a new one is returned and not stored, the next refresh fails.
**How to avoid:** Always check `$result->{refresh_token}` in the refresh response and update Prefs if present.
**Warning signs:** 400 Bad Request on refresh after weeks of working auth.

## Code Examples

### Complete PKCE Flow Initiation (Settings.pm side)

```perl
# Source: Spotify PKCE docs + LMS source verified on target system

use Crypt::OpenSSL::Random;
use MIME::Base64 qw(encode_base64url);
use Digest::SHA qw(sha256);
use URI;

sub startOAuth {
    my ($class, $paramRef) = @_;

    my $clientId = $prefs->get('clientId') or return (undef, 'No client ID configured');
    my $httpPort = preferences('server')->get('httpport') || 9000;
    my $redirectUri = "http://127.0.0.1:${httpPort}/plugins/SpotOn/settings/callback";

    # PKCE
    my $verifierBytes  = Crypt::OpenSSL::Random::random_bytes(32);
    my $codeVerifier   = encode_base64url($verifierBytes);
    $codeVerifier      =~ s/=+$//;
    my $codeChallenge  = encode_base64url(sha256($codeVerifier));
    $codeChallenge     =~ s/=+$//;

    # State (OAuth CSRF nonce)
    my $stateBytes = Crypt::OpenSSL::Random::random_bytes(16);
    my $state      = encode_base64url($stateBytes);
    $state         =~ s/=+$//;

    # Cache verifier under state key (TTL 10 min)
    $cache->set("spoton_pkce_$state", {
        code_verifier => $codeVerifier,
        client_id     => $clientId,
        redirect_uri  => $redirectUri,
    }, 600);

    # Build auth URL
    my $authUri = URI->new('https://accounts.spotify.com/authorize');
    $authUri->query_form(
        response_type         => 'code',
        client_id             => $clientId,
        redirect_uri          => $redirectUri,
        code_challenge_method => 'S256',
        code_challenge        => $codeChallenge,
        scope                 => REQUIRED_SCOPES,
        state                 => $state,
    );

    return ($authUri->as_string(), undef);
}
```

### Callback Handler Skeleton (Settings/Callback.pm)

```perl
# Source: LMS source — addPageFunction pattern
package Plugins::SpotOn::Settings::Callback;

use constant CALLBACK_PATH => 'plugins/SpotOn/settings/callback';

sub init {
    Slim::Web::Pages->addPageFunction(CALLBACK_PATH, \&handler);
}

sub handler {
    my ($client, $paramRef, $callback, $httpClient, $response) = @_;

    my $code  = $paramRef->{code}  // '';
    my $state = $paramRef->{state} // '';
    my $error = $paramRef->{error} // '';

    # Error path: user denied, show error page
    if ($error) {
        return _renderResult($paramRef, $callback, $httpClient, $response, 0, $error);
    }

    # Validate state (CSRF)
    my $pkce = $cache->get("spoton_pkce_$state");
    unless ($pkce) {
        return _renderResult($paramRef, $callback, $httpClient, $response, 0, 'invalid_state');
    }
    $cache->remove("spoton_pkce_$state");

    # Exchange code for tokens (async — must not block)
    require Plugins::SpotOn::API::TokenManager;
    Plugins::SpotOn::API::TokenManager->exchangeCode(
        $code,
        $pkce->{code_verifier},
        $pkce->{client_id},
        $pkce->{redirect_uri},
        sub {
            my ($tokens, $err) = @_;
            _renderResult($paramRef, $callback, $httpClient, $response,
                          $tokens ? 1 : 0,
                          $err // '');
        }
    );

    # Return undef — response will be sent from async callback above
    return;
}
```

## D-06 / D-08 Resolution: Redirect URI and Callback Mechanism

**Research conclusion:** Use LMS's own HTTP server as the callback endpoint. No external proxy needed.

**How it works:**
1. `Slim::Web::Pages->addPageFunction('plugins/SpotOn/settings/callback', \&handler)` registers the route.
2. The redirect URI is `http://127.0.0.1:{httpPort}/plugins/SpotOn/settings/callback`.
3. When Spotify redirects the browser to this URL, LMS routes it to the handler.
4. The handler receives `$paramRef->{code}` and `$paramRef->{state}` (LMS parses all GET parameters).
5. No polling loop, no secondary HTTP listener, no external service.

**What users must do:** Register `http://127.0.0.1:{port}/plugins/SpotOn/settings/callback` in their Spotify Developer App dashboard. The Setup Wizard must display this exact string (with port auto-filled from LMS config).

**Comparison with Spotty's approach:** Spotty uses `https://api.lms-community.org/auth/callback` as the redirect URI (external proxy) to avoid requiring users to register their own App. SpotOn requires user App registration (D-04), so the external proxy is unnecessary and undesirable (adds external dependency).

**Comparison with Settings.pm handler:** The Settings handler (`Slim::Web::Settings`) is registered via `protectURI`, receives POST forms, and cannot receive Spotify's GET redirect. A separate `addPageFunction` route is the correct pattern. [VERIFIED: LMS source]

## D-12 Resolution: Token Storage

**Research conclusion:** Split storage by token type.

| Token | Storage | Reason |
|-------|---------|--------|
| `access_token` | `Slim::Utils::Cache` with TTL = `expires_in - 300` | Short-lived (1h); Cache's TTL eviction is a feature, not a bug |
| `refresh_token` | `Slim::Utils::Prefs` inside `accounts` hash | Must survive LMS restarts; no TTL eviction; already existing pattern |
| `clientId` (per plugin, not per account) | `Slim::Utils::Prefs` as `$prefs->get('clientId')` | User-configured once; must persist |

**Prefs structure:**
```perl
$prefs->init({
    clientId      => '',        # User's Spotify Developer App Client ID
    accounts      => {},        # accountId => { displayName, refreshToken }
    activeAccount => '',
    bitrate       => 320,
    binary        => '',
});
```

The `username` field disappears. The `refreshToken` field is added per-account. The `accountId` generation changes from MD5(username) to MD5(spotify_user_id from /me response) — more stable across username changes.

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | Yes | OAuth 2.0 PKCE — no password stored, no client secret |
| V3 Session Management | Yes | state parameter (nonce in Cache, single-use); access token in Cache with TTL |
| V4 Access Control | Partial | LMS settings page protected by `protectURI`; callback route relies on state validation |
| V5 Input Validation | Yes | Validate `code` is non-empty string; validate `state` format; reject `error` from Spotify gracefully |
| V6 Cryptography | Yes | `Crypt::OpenSSL::Random` for verifier + state; `Digest::SHA` for challenge — no hand-rolled crypto |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| CSRF on OAuth callback | Spoofing | `state` nonce round-trip (cache → verify → consume) |
| Authorization code injection | Tampering | `state` validation rejects codes not associated with a local flow |
| Refresh token leakage via log | Information Disclosure | Never log token values (existing `# T-02-10` convention in Client.pm) |
| Stale code_verifier in cache | Information Disclosure | TTL=600s; consume-on-use (`$cache->remove`) |
| client_id exposure | Information Disclosure | Stored in Prefs (local disk); not a secret in PKCE (by design) |
| Redirect URI hijacking | Tampering | LMS port is registered in Spotify App by user; no open redirect possible |
