# Phase 02.1: Code Review Report

**Reviewed:** 2026-05-27T22:15:00Z
**Depth:** standard
**Files Reviewed:** 11
**Status:** issues_found

## Summary

Phase 02.1 delivers the OAuth 2.0 PKCE browser-auth flow: PKCE generation, token exchange, storage and refresh in `TokenManager.pm`, a settings UI in `Settings.pm` + `basic.html`, and an OAuth callback handler in `Settings/Callback.pm`.

The cryptographic and flow-logic core is correct: PKCE verifier/challenge generation follows RFC 7636, state is consumed on first use (anti-replay), refresh-token rotation is handled, and token credentials are stored in Prefs (not Cache). Security-sensitive values (tokens, verifier) are not logged.

Two blockers require fixing before this ships: an unescaped URL inserted into a JavaScript `window.location` assignment (XSS vector when `webroot` is abnormal), and a clientId timing bug that silently uses a stale preference value when a user enters a new Client ID and clicks "Connect" in the same form submission.

Five warnings cover: hardcoded German text inside the EN-language template (`basic.html`), i18n bypass in `Callback.pm` (inline German strings despite string keys existing), an orphaned dead template (`callback.html`), an inaccurate skip count in one test file, and an unescaped localisation string embedded in a JavaScript `confirm()` call.

## Critical Issues

### CR-01: Unescaped `$settingsUrl` Injected into JavaScript and href

**File:** `Plugins/SpotOn/Settings/Callback.pm:141` and `:159`
**Issue:** `$settingsUrl` is built from `$paramRef->{webroot}` (line 121) without any sanitisation and then interpolated directly into a `<script>` block (`window.location = "$settingsUrl";`) and an `<a href="$settingsUrl">` attribute. In the LMS runtime `webroot` is always `'/'` or `'/squeezeboxserver/'`, but it arrives via `$paramRef` whose contents originate from the HTTP request environment. If `webroot` were ever abnormal (e.g., containing a double-quote or semicolon due to a proxy, misconfiguration, or future LMS change), an attacker who can influence that value could inject arbitrary JavaScript into the callback page.

The `href` variant is additionally dangerous because a `javascript:` URL injected there executes script when the user clicks the "Back to Settings" link.

**Fix:** Escape `$settingsUrl` before embedding it in HTML attributes and JavaScript string literals:

```perl
# At the top of _renderResult, after building $settingsUrl:
my $settingsUrlHtml = _html_escape($settingsUrl);
# For the href context (already HTML-safe after this):
# <a href="$settingsUrlHtml">

# For the JavaScript context, additionally escape for JS string literals:
sub _js_escape {
    my ($str) = @_;
    $str =~ s/\\/\\\\/g;
    $str =~ s/"/\\"/g;
    $str =~ s/'/\\'/g;
    $str =~ s/\n/\\n/g;
    $str =~ s/\r/\\r/g;
    return $str;
}
my $settingsUrlJs = _js_escape($settingsUrl);
# window.location = "$settingsUrlJs";
```

### CR-02: `clientId` Read from Stale Prefs When First-Time User Submits Client ID and Clicks "Connect" Simultaneously

**File:** `Plugins/SpotOn/Settings.pm:55`
**Issue:** `Settings->handler()` reads `$prefs->get('clientId')` at line 55 to decide whether to launch the OAuth flow. `$prefs->get()` returns the *previously persisted* value because `SUPER::handler()` — which is what saves `pref_clientId` from the form into prefs — is called at the *end* of the subclass handler (line 123), after the `startOAuth` check has already executed.

Consequence: A first-time user who types their Client ID into the field and immediately clicks the "Connect with Spotify" button in the same form submission will silently fall into the empty-clientId error branch (line 57–58: `$paramRef->{oauthError} = string('PLUGIN_SPOTON_CLIENT_ID_REQUIRED')`), even though they entered a valid value. The JavaScript `disabled` guard in `basic.html` partially mitigates this (the button is disabled until the field has content *and* the stored pref is populated), but only when JS is enabled — and more importantly, the guard relies on a round-trip save happening *before* the OAuth click, which is not enforced in the template or handler logic.

**Fix:** Read the submitted form value directly from `$paramRef` before falling back to prefs, and save it to prefs immediately:

```perl
if ($paramRef->{startOAuth}) {
    # Read from the submitted form field first; fall back to stored pref.
    my $clientId = $paramRef->{'pref_clientId'} // $prefs->get('clientId') // '';
    $clientId =~ s/^\s+|\s+$//g;
    # Persist the trimmed value so SUPER does not overwrite with the untrimmed one.
    $prefs->set('clientId', $clientId) if $clientId;
    if (!$clientId) {
        $paramRef->{oauthError} = string('PLUGIN_SPOTON_CLIENT_ID_REQUIRED');
    } else {
        my ($authUrl, $state) = Plugins::SpotOn::API::TokenManager->startOAuthFlow($clientId);
        ...
    }
}
```

### WR-02: `Callback.pm` Bypasses i18n — Hardcoded German Strings Despite String Keys Existing

**File:** `Plugins/SpotOn/Settings/Callback.pm:127`, `:137`, `:138`, `:148`, `:157`, `:159`
**Issue:** `_renderResult()` builds the entire callback page with hardcoded German strings interpolated directly into a Perl heredoc. The `strings.txt` file already defines corresponding i18n keys for every string used here:

| Hardcoded string | Existing key |
|---|---|
| `"Erfolgreich verbunden!"` | `PLUGIN_SPOTON_AUTH_SUCCESS` |
| `"Du wirst gleich zu den Einstellungen weitergeleitet..."` | `PLUGIN_SPOTON_AUTH_REDIRECT` |
| `"Verbunden als: %s"` | `PLUGIN_SPOTON_CONNECTED_AS` |
| `"Verbindung fehlgeschlagen"` | `PLUGIN_SPOTON_AUTH_FAILED` |
| `"Zurueck zu den Einstellungen"` | `PLUGIN_SPOTON_BACK_TO_SETTINGS` |

English-speaking users will see German text on the OAuth callback page.

**Fix:** Import `Slim::Utils::Strings` and use `string()`:

```perl
use Slim::Utils::Strings qw(string);

# In _renderResult success path:
my $titleHtml   = _html_escape(string('PLUGIN_SPOTON_AUTH_SUCCESS'));
my $redirectMsg = _html_escape(string('PLUGIN_SPOTON_AUTH_REDIRECT'));
my $connectedAs = defined $displayName && $displayName ne ''
    ? sprintf(_html_escape(string('PLUGIN_SPOTON_CONNECTED_AS')), _html_escape($displayName))
    : '';
```

### WR-03: `callback.html` Template Is Dead Code (Unreachable)

**File:** `Plugins/SpotOn/HTML/EN/plugins/SpotOn/settings/callback.html`
**Issue:** `Settings/Callback.pm` registers a plain page-function handler (`addPageFunction`) that builds and delivers an inline HTML response via `_renderResult()`. The `callback.html` TT2 template file is never referenced, loaded, or rendered anywhere in the plugin code. It is dead code that will never be served to a user.

This creates a maintenance hazard: the template contains its own independently-maintained strings and redirect logic that silently diverge from the actual callback implementation in `Callback.pm`. Future developers may be confused about which file controls the callback page.

**Fix:** Delete `callback.html` or add a comment to `Callback.pm` explaining why the template is not used. If there is ever a plan to migrate to a template-based approach, the file should be flagged as a work-in-progress, not left as silent dead code.

### WR-05: Localisation String Embedded Raw in JavaScript `confirm()` Without JS-Escaping

**File:** `Plugins/SpotOn/HTML/EN/plugins/SpotOn/settings/basic.html:78`
**Issue:** The confirmation dialog uses a TT2 `| string` filter to embed the `PLUGIN_SPOTON_ACCOUNT_REMOVE_CONFIRM` string directly inside a single-quoted JavaScript string literal in an `onclick` attribute:

```html
onclick="return confirm('[% 'PLUGIN_SPOTON_ACCOUNT_REMOVE_CONFIRM' | string %]') && ..."
```

The `| string` filter performs LMS string lookup — it does **not** escape the result for a JavaScript string literal context. If a translator adds a contraction with an apostrophe (e.g., "Can't be undone"), or if the string contains a backslash, the resulting JavaScript will be syntactically broken or, in a worst case, execute unintended code.

The current English and German values happen to be safe, but this is fragile by construction.

**Fix:** Either use a `| js` filter if available in the LMS TT2 environment, or move the confirm message to a `data-` attribute and read it in JavaScript:

```html
<input type="submit"
    data-confirm="[% 'PLUGIN_SPOTON_ACCOUNT_REMOVE_CONFIRM' | string | html %]"
    onclick="var msg=this.getAttribute('data-confirm');
             return confirm(msg) && (this.previousElementSibling.disabled=false,true)"/>
```

Using `| html` encodes the string as an HTML attribute value; reading it back via `getAttribute` decodes HTML entities before passing to `confirm()`, which is safe.

### IN-02: `_buildRedirectUri` Called as Public Method From `Settings.pm`

**File:** `Plugins/SpotOn/Settings.pm:121`
**Issue:** `Settings.pm` calls `Plugins::SpotOn::API::TokenManager->_buildRedirectUri()` directly (line 121) to populate `$paramRef->{redirectUri}` for the template. Methods named with a leading underscore are by Perl convention private. Calling a private method from outside its module couples `Settings.pm` to an implementation detail of `TokenManager.pm`, making future refactoring of `_buildRedirectUri` a cross-module change.

**Fix:** Add a public wrapper or promote `_buildRedirectUri` to a public method named `buildRedirectUri`:

```perl
# In TokenManager.pm:
sub buildRedirectUri { _buildRedirectUri(@_) }
# Or simply rename _buildRedirectUri -> buildRedirectUri throughout.
```
