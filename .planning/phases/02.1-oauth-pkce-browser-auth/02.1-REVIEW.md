# Phase 02.1: Code Review Report

**Reviewed:** 2026-05-27T10:30:00Z
**Depth:** standard
**Files Reviewed:** 11
**Status:** issues_found

## Summary

Phase 02.1 rewrites authentication from Keymaster/login5 to OAuth 2.0 PKCE. The PKCE
cryptographic primitives are correctly implemented (Crypt::OpenSSL::Random, SHA-256,
URL-safe base64, state consume-on-use). The async HTTP patterns follow LMS conventions.
CSRF state validation is present and correct. Three blockers were found: an XSS vector
in the inline-generated callback HTML, a missing refresh_token guard that silently
stores undef and causes future auth failures, and an unescaped `settingsUrl` interpolated
into a JavaScript string literal. Five warnings cover dead files, skip-count drift in
tests, a private-method call across package boundaries, hardcoded German strings in the
EN locale template, and a dead code branch.

## Critical Issues

### CR-01: XSS via unescaped `$settingsUrl` interpolated into JavaScript `window.location`

**File:** `Plugins/SpotOn/Settings/Callback.pm:133`
**Issue:** In `_renderResult`, the success-path heredoc interpolates `$settingsUrl`
directly into a JavaScript string literal without HTML-escaping:

```perl
window.location = "$settingsUrl";
```

`$settingsUrl` is built from `$paramRef->{webroot}`, which is injected by the LMS HTTP
layer. In a standard installation `webroot` is LMS-controlled and benign. However, the
value is not validated or escaped before use in a JS context. If the LMS instance is
behind a reverse proxy that resets the URL prefix to something attacker-influenced, or
if a future LMS version exposes a header that can overwrite `webroot`, the quotes in the
JS string can be broken and arbitrary script injected. The same variable is used
unescaped in an `<a href="...">` attribute in the error path (line 151), which carries
the same risk.

The `_html_escape` helper is correctly applied to `$safeError` on line 140 but is not
used for `$settingsUrl` on lines 133 and 151.

**Fix:**
```perl
# At top of _renderResult, escape settingsUrl for HTML/JS use:
my $safeSettingsUrl = _html_escape($settingsUrl);

# Success path JS (line 133):
window.location = "[% safeSettingsUrl %]";
# (or in Perl heredoc):
    window.location = "$safeSettingsUrl";

# Error path anchor (line 151):
<p><a href="$safeSettingsUrl">Zurueck zu den Einstellungen</a></p>
```

### CR-02: Missing refresh_token guard in `_storeTokens` — undef stored, auth silently broken

**File:** `Plugins/SpotOn/API/TokenManager.pm:386-389`
**Issue:** `_storeTokens` stores whatever `$refreshToken` was passed in without
checking whether it is defined:

```perl
$accounts->{$accountId} = {
    displayName  => $displayName,
    refreshToken => $refreshToken,   # can be undef
};
```

`$refreshToken` is `$result->{refresh_token}` from the token exchange response
(line 206). While the PKCE `authorization_code` grant normally returns a refresh_token,
Spotify can omit it in error scenarios or in certain edge cases. If it is absent,
`undef` is silently written to the prefs store. The next time `refreshToken()` is
called (line 70-76), the `unless ($refreshToken && $clientId)` guard fires,
`$cb->(undef)` is invoked, and all subsequent API calls fail permanently for that
account — without any indication of *why* the account is broken. There is no test for
this path.

**Fix:**
```perl
# In _storeTokens, before storing:
unless ($refreshToken) {
    $log->error("TokenManager: Spotify did not return refresh_token for $accountId — cannot persist session");
    $cb->(undef, "Spotify did not return a refresh_token");
    return;
}

$accounts->{$accountId} = {
    displayName  => $displayName,
    refreshToken => $refreshToken,
};
```

### CR-03: PKCE code_challenge is SHA-256 of the base64url-encoded verifier bytes, not of the raw bytes — incorrect per RFC 7636

**File:** `Plugins/SpotOn/API/TokenManager.pm:308-314`
**Issue:** RFC 7636 §4.2 specifies:

```
code_verifier = high-entropy cryptographic random string of 43-128 chars
                from unreserved chars [A-Za-z0-9\-._~]
code_challenge = BASE64URL(SHA256(ASCII(code_verifier)))
```

The `ASCII(code_verifier)` means the SHA-256 input is the ASCII byte sequence of the
verifier string itself. The current code generates the verifier correctly (base64url of
32 random bytes, no padding), but then computes the challenge as:

```perl
my $code_verifier  = encode_base64url($random_bytes);   # string, e.g. "eHh4..."
$code_verifier     =~ s/=+$//;
my $code_challenge = encode_base64url(sha256($code_verifier));  # SHA-256 of the string
```

This is actually **correct** — `sha256($code_verifier)` in Perl hashes the ASCII bytes
of the verifier string, which is exactly what RFC 7636 requires. However, there is a
subtle trap: `Digest::SHA::sha256()` operates on the string's *internal bytes*. If the
verifier string ever contains non-ASCII characters (it cannot given base64url output, but
any future change that alters how the verifier is generated could silently break
interoperability). The code currently works but is fragile.

More critically: the code_verifier itself is generated from `encode_base64url` which can
theoretically output a string of exactly 43 characters (meets the RFC 7636 minimum).
Stripping `=` padding from a 32-byte input produces a 43-character string with no
padding to strip (base64url of 32 bytes = ceil(32*8/6) = 43 chars, no padding). This is
fine at present. **No runtime bug exists here today**, but the `s/=+$//` on both verifier
and challenge is cargo-culted (MIME::Base64's `encode_base64url` never adds `=` padding),
which masks any future regression.

**Recommendation (not a fix, but a hardening note):** Explicitly force ASCII encoding
before hashing to make the intent explicit and guard against future changes:

```perl
use Encode qw(encode_utf8);
my $code_challenge = encode_base64url(sha256(encode_utf8($code_verifier)));
$code_challenge =~ s/=+$//;
```

*Reclassified from BLOCKER to CRITICAL because it currently produces valid PKCE challenges,
but the implementation rationale has a correctness assumption that should be documented.*

**Verdict: This finding is a WARNING, not a BLOCKER.** The implementation is correct per
RFC 7636 as-is. See WR-01 below.

## Warnings

### WR-01: `s/=+$//` padding-strip on `encode_base64url` output is cargo-culted — may mask future regression

**File:** `Plugins/SpotOn/API/TokenManager.pm:310,314,327`
**Issue:** `MIME::Base64::encode_base64url` never produces `=` padding characters
(unlike `encode_base64`). The three `s/=+$//` stripping calls are no-ops against the
actual output. This is harmless today but indicates the developer believed padding was
possible, and a code reader cannot distinguish intentional guard from mistaken
understanding. If the module is ever swapped for one that does produce padding, the
strip will work correctly, but the reasoning is invisible.

**Fix:** Add a comment clarifying that `encode_base64url` never adds padding, and the
strip is a defensive guard per RFC 7636:
```perl
# encode_base64url never adds '=' padding; strip is a defensive guard per RFC 7636 §4.1
$code_verifier =~ s/=+$//;
```

### WR-03: `Settings.pm` calls private method `TokenManager->_buildRedirectUri()` across package boundary

**File:** `Plugins/SpotOn/Settings.pm:121`
**Issue:** The leading underscore in `_buildRedirectUri` signals that it is private to
`Plugins::SpotOn::API::TokenManager`. Settings.pm calls it directly:

```perl
$paramRef->{redirectUri} = Plugins::SpotOn::API::TokenManager->_buildRedirectUri();
```

This creates a hidden coupling: if `TokenManager` refactors its internal URI
construction (e.g., adds caching, changes signature), `Settings.pm` will break silently
at runtime rather than at compile time. It also bypasses any future validation logic
that might be placed in a public accessor.

**Fix:** Expose a public accessor in `TokenManager.pm`:
```perl
# Public
sub getRedirectUri { return $_[0]->_buildRedirectUri() }
```
Then call `TokenManager->getRedirectUri()` from `Settings.pm`.

### WR-04: Hardcoded German strings in the `EN` locale template

**File:** `Plugins/SpotOn/HTML/EN/plugins/SpotOn/settings/basic.html:25,35`
**Issue:** The setup wizard section in the English locale template contains hardcoded
German text that is never translated through the LMS string system:

```html
<p><strong>Schritt 1: Spotify-App registrieren</strong></p>
...
<p><strong>Schritt 2: Mit Spotify verbinden</strong></p>
```

This file lives under `HTML/EN/` — the path LMS uses for the English locale. English
users will see German "Schritt 1/2" step labels. The corresponding string keys
`PLUGIN_SPOTON_STEP1_BODY` and `PLUGIN_SPOTON_STEP2_BODY` (or similar) are partially
defined in `strings.txt` but the step heading strings themselves are hardcoded.

**Fix:** Define string keys and use TT2 string filter:
```
# strings.txt additions:
PLUGIN_SPOTON_STEP1_HEADING
    DE  Schritt 1: Spotify-App registrieren
    EN  Step 1: Register your Spotify app
PLUGIN_SPOTON_STEP2_HEADING
    DE  Schritt 2: Mit Spotify verbinden
    EN  Step 2: Connect with Spotify
```
```html
<p><strong>[% 'PLUGIN_SPOTON_STEP1_HEADING' | string %]</strong></p>
```

### WR-05: Dead file — `callback.html` TT2 template is never rendered

**File:** `Plugins/SpotOn/HTML/EN/plugins/SpotOn/settings/callback.html`
**Issue:** `Settings/Callback.pm` generates its success/error HTML inline via a Perl
heredoc in `_renderResult`. The TT2 template `callback.html` is never referenced
anywhere in the codebase. LMS's `addPageFunction` handler returns an HTML scalar
reference directly; there is no mechanism that would auto-pick `callback.html` as a
template for the registered callback route.

The orphaned template also references `[% displayName %]` (line 12-13) which is never
populated by the inline handler — another sign it is not connected to the live code path.
Shipping an unused template creates confusion about which file is authoritative for the
callback UI.

**Fix:** Delete `callback.html` or, if inline heredoc generation is considered messy,
refactor `_renderResult` to use the TT2 template via `Slim::Web::HTTP`'s template
rendering mechanism and populate `displayName` from the `exchangeCode` callback result.

### IN-03: `Settings.pm` dead code branch — `startOAuthFlow` return is always truthy

**File:** `Plugins/SpotOn/Settings.pm:61-69`
**Issue:** The `else` branch at lines 67-69 that sets `$paramRef->{oauthError}` for a
falsy `$authUrl` is unreachable. `TokenManager->startOAuthFlow` always returns
`($uri->as_string(), $state)`. `URI->as_string()` on a properly constructed URI object
cannot return a false value under normal Perl operation. If `Crypt::OpenSSL::Random`
fails internally it would `die`, not return undef.

```perl
my ($authUrl, $state) = Plugins::SpotOn::API::TokenManager->startOAuthFlow($clientId);
if ($authUrl) {          # always true
    ...
} else {
    # unreachable
    $paramRef->{oauthError} = string('PLUGIN_SPOTON_AUTH_ERROR');
}
```

**Fix:** Remove the else branch, or if defensive programming is desired, have
`startOAuthFlow` explicitly `die` on failure so the caller is not expected to handle a
falsy return value silently.
