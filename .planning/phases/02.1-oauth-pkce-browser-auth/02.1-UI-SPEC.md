# Phase 02.1 — UI Design Contract

> Visual and interaction contract for the OAuth-PKCE browser auth settings page.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Context: LMS Settings Page (Not a SPA)

This phase adds UI exclusively inside the LMS plugin settings page rendered by
`Slim::Web::Settings`. The environment is:

- Server-rendered HTML via Perl Template Toolkit (TT2)
- Template path: `Plugins/SpotOn/HTML/EN/plugins/SpotOn/settings/basic.html`
- New callback result template: `Plugins/SpotOn/HTML/EN/plugins/SpotOn/settings/callback.html`
- No React, Vue, Tailwind, or shadcn — none applicable
- LMS delivers its own stylesheet (`slimserver.css`, `skin.css`) — the plugin does not ship CSS
- LMS browser context loads Prototype.js + Scriptaculous (no jQuery, no ES modules)
- All form structure uses `[% WRAPPER setting title="STRING_KEY" desc="" %]` blocks

**Consequence for this contract:** Spacing, typography, and color values below describe
the *LMS-system values* as observed in `slimserver.css`. They are reference values for
writing copy and structuring markup — not tokens to configure in a design tool.

Source of pre-populated values: `CONTEXT.md` (D-04 through D-09), `RESEARCH.md`
(architecture diagram, callback HTML pattern, UI observations), `basic.html` (existing
markup patterns), `slimserver.css` (observed system values).

## Component Inventory (LMS HTML Patterns)

These are the exact markup patterns to use. No custom components.

### Pattern A: Setting wrapper (all setting blocks)
```
[% WRAPPER setting title="STRING_KEY" desc="" %]
  ... content ...
[% END %]
```

### Pattern B: Standard form control
```html
<input type="text" class="stdedit" name="clientId" id="clientId" value="..." />
<select class="stdedit" name="pref_bitrate" ...>
```

### Pattern C: Submit button (primary action)
```html
<input type="submit" name="action_name" value="[% 'STRING_KEY' | string %]" class="submit" />
```

### Pattern D: Error display (inline, above affected field)
```html
<div style="color:red">[% errorMessage | html %]</div>
```

### Pattern E: Success/status display
```html
<div style="color:green">[% 'STRING_KEY' | string %]: [% displayName | html %]</div>
```

### Pattern F: Informational copy (below a field)
```html
<p class="hint">[% 'STRING_KEY' | string %]</p>
```

### Pattern G: Monospace redirect URI display
```html
<code>[% redirectUri | html %]</code>
```
Use `<code>` for the exact redirect URI string. LMS does not ship a custom monospace style;
the browser default monospace is sufficient and correct.

## Screen Inventory

### Screen 1: Settings Page — Unconfigured State (Setup Wizard)

Shown when `clientId` pref is empty OR no accounts exist.

**Sections (top to bottom):**

1. **Binary Status** — existing `WRAPPER setting title="PLUGIN_SPOTON_BINARY_STATUS"` block, unchanged.

2. **Setup Wizard** — new `WRAPPER setting title="PLUGIN_SPOTON_SETUP_WIZARD"` block.
   - Schritt 1 label: `<strong>Schritt 1: Spotify-App registrieren</strong>`
   - Body copy: see Copywriting Contract
   - Redirect-URI display: `<code>[% redirectUri | html %]</code>` (auto-filled by Settings.pm)
   - Link: `<a href="https://developer.spotify.com/dashboard" target="_blank">Spotify Developer Dashboard</a>`
   - Client-ID input: `<input type="text" class="stdedit" name="pref_clientId" id="pref_clientId" value="[% prefs.pref_clientId | html %]" />`
   - Schritt 2 label: `<strong>Schritt 2: Mit Spotify verbinden</strong>`
   - Connect button: `<input type="submit" name="startOAuth" value="[% 'PLUGIN_SPOTON_CONNECT_BTN' | string %]" class="submit" id="startOAuthBtn" />`
   - The connect button MUST be disabled (HTML `disabled` attribute) when `pref_clientId` is empty.
     Use inline JS (Prototype.js available): `$('pref_clientId').observe('input', function(){ $('startOAuthBtn').disabled = ($('pref_clientId').value.trim() == ''); });`

3. **Bitrate** — existing `WRAPPER setting title="PLUGIN_SPOTON_BITRATE"` block, unchanged.

**What is removed:** The `PLUGIN_SPOTON_ACCOUNT_ADD` block with username/password inputs.
Both `<label>` elements (username, password) and their inputs are deleted entirely.
Source: CONTEXT.md D-02, RESEARCH.md "State of the Art" table.

---

### Screen 2: Settings Page — Configured State (Connected)

Shown when `clientId` pref is set AND at least one account exists in `accounts` pref.

**Sections (top to bottom):**

1. **Binary Status** — unchanged.

2. **Konto-Einstellungen** — modified `WRAPPER setting title="PLUGIN_SPOTON_ACCOUNT_SETTINGS"` block.
   - Account table: retain existing `<table>` pattern with displayName + active/switch/remove controls.
   - **Remove** `accounts.$id.username` column — username is not available post-PKCE (user is
     identified by displayName from `/me` response). Replace with displayName only.
   - "Verbunden als" status: `<div style="color:green">` above the table when `activeAccount` is set.
   - "Weiteres Konto hinzufuegen" link/button replaces the old username/password form:
     `<input type="submit" name="addAnotherAccount" value="[% 'PLUGIN_SPOTON_ADD_ANOTHER' | string %]" class="submit" />`
     (This re-triggers the Setup Wizard / startOAuth flow for an additional account.)

3. **Bitrate** — unchanged.

---

### Screen 3: OAuth Callback Result Page (callback.html)

Served at `http://127.0.0.1:{port}/plugins/SpotOn/settings/callback`.
This is a minimal standalone HTML page (NOT wrapped in the LMS settings frame).
It is never bookmarked; it auto-redirects.

**Success path:**
```html
<!DOCTYPE html>
<html>
<head>
  <meta charset="utf-8">
  <title>SpotOn – Verbunden</title>
  <script>
    // Redirect to settings page after 2 seconds
    setTimeout(function() {
      window.location = '[% webroot | html %]settings/plugin/spoton.html';
    }, 2000);
  </script>
</head>
<body style="font-family:sans-serif; padding:24px;">
  <p style="color:green"><strong>[% 'PLUGIN_SPOTON_AUTH_SUCCESS' | string %]</strong></p>
  <p>[% 'PLUGIN_SPOTON_AUTH_REDIRECT' | string %]</p>
</body>
</html>
```

**Error path:**
```html
<body style="font-family:sans-serif; padding:24px;">
  <p style="color:red"><strong>[% 'PLUGIN_SPOTON_AUTH_FAILED' | string %]</strong></p>
  <p>[% errorMessage | html %]</p>
  <p><a href="[% webroot | html %]settings/plugin/spoton.html">[% 'PLUGIN_SPOTON_BACK_TO_SETTINGS' | string %]</a></p>
</body>
```

The success page auto-redirects in 2 seconds. The error page shows a manual "Zurück
zu den Einstellungen" link instead of auto-redirect (user may need to read the error).

Source: RESEARCH.md Pattern 8 (Callback Response), CONTEXT.md D-09.

### Error State Rules

| Error Condition | Where Displayed | String Key | What User Must Do |
|----------------|----------------|-----------|-------------------|
| Client-ID is empty when form submitted | Inline below Client-ID field, `color:red` | `PLUGIN_SPOTON_CLIENT_ID_REQUIRED` | Trage deine Client-ID ein. / Enter your Client ID. |

## Interaction Contracts

### IC-01: Client-ID Input Gating

- When: Settings page renders with `clientId` empty (first visit or after removal)
- The "Mit Spotify verbinden" submit button renders with `disabled` attribute
- A `<script>` block (Prototype.js `observe('input', ...)`) enables the button when the
  Client-ID field is non-empty
- Fallback (JS disabled): server-side validation in `Settings.pm::handler()` rejects
  `startOAuth` with `clientId` empty and re-renders with `PLUGIN_SPOTON_CLIENT_ID_REQUIRED` error

### IC-02: OAuth Flow Initiation

- When: User clicks "Mit Spotify verbinden" with a non-empty Client-ID
- Server-side: `Settings.pm::handler()` detects `$paramRef->{startOAuth}`, calls `startOAuth()`,
  builds the Spotify auth URL
- Response: HTTP 302 redirect to `https://accounts.spotify.com/authorize?...`
  (server-side redirect, not JS `window.location`) — most reliable across all browsers
- The user's browser navigates away from LMS settings to `accounts.spotify.com`

### IC-03: OAuth Callback

- When: Spotify redirects browser back to `http://127.0.0.1:{port}/plugins/SpotOn/settings/callback`
- `Settings/Callback.pm::handler()` processes the callback asynchronously
- While the async token exchange is pending: no page is shown (connection appears loading)
- On success: callback.html success path is rendered; auto-redirect after 2s
- On error: callback.html error path is rendered with error message + back link

### IC-04: Post-Auth Settings Page State

- After successful auth + redirect back to settings:
  Settings page renders in Configured State (Screen 2)
- "Verbunden als: [displayName]" is shown in green above the account table
- The Setup Wizard block (`PLUGIN_SPOTON_SETUP_WIZARD`) is hidden when at least one account
  is configured: `[% IF NOT accounts.keys.size %] ... wizard ... [% END %]`
- `clientId` pref field remains visible and editable at all times (user may need to update it)

### IC-05: Account Removal

- Confirmation via `confirm()` JS dialog (see Destructive Action above)
- If confirmed: existing `removeAccount` hidden input pattern (from `basic.html`) is retained
- After removal: if no accounts remain, page reverts to Unconfigured State (Screen 1)
- If the removed account was the active account and others remain: Settings.pm auto-selects
  the next account (existing logic, unchanged)
