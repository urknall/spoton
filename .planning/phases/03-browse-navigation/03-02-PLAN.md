    Extend Plugin.pm with the following additions. All new code goes BELOW the existing _switchAccount sub and ABOVE the trailing "1;" line.

    **A. Shared helper functions** (used by this plan and Plan 03):

    1. _getAccountId($client) — returns $prefs->client($client)->get('activeAccount') || $prefs->get('activeAccount') || ''. Extract from existing inline pattern in _accountSwitcherFeed.

    2. _largestImage($images_arrayref) — returns URL string of largest image by width from Spotify images array. Returns '' if empty/undef. Sort by width descending, take first. Per PATTERNS.md Artwork pattern.

    3. _isMadeForYou($playlist_hashref) — returns true if ($playlist->{owner}{id} // '') eq 'spotify'. Per D-04, RESEARCH Pattern 5.

    4. _trackItem($client, $track) — builds OPML audio item hashref. Fields per RESEARCH Pattern 3 and PATTERNS.md Track-Item pattern:
       - name: "$title \x{2014} $artist" (em-dash, fallback for old clients)
       - line1: $title (track name)
       - line2: "$artist \x{2022} $album" (bullet separator; omit album part if empty)
       - url: 'spotify://' . $track->{uri} (Play-Intent per D-06)
       - play: 'spotify://' . $track->{uri}
       - on_select: 'play'
       - image: _largestImage($track->{album}{images})
       - duration: ($track->{duration_ms} || 0) / 1000
       - type: 'audio'
       Context navigation items (D-07 "Artist anzeigen", "Album anzeigen") will be added in Plan 03 when detail feed handlers exist.

    5. _albumItem($client, $album) — builds OPML link item for album navigation:
       - name: $album->{name}
       - url: \&_albumFeed (reference to sub defined in Plan 03 — use string ref 'Plugins::SpotOn::Plugin::_albumFeed' or forward-declare; simplest: just use \&_albumFeed since Perl resolves at runtime)
       - passthrough: [{ albumId => $album->{id} }]
       - image: _largestImage($album->{images})
       - line2: first artist name . ' (' . ($album->{release_date} // '') . ')'
       - type: 'link'

    6. _artistItem($client, $artist) — builds OPML link item for artist navigation:
       - name: $artist->{name}
       - url: \&_artistFeed
       - passthrough: [{ artistId => $artist->{id} }]
       - image: _largestImage($artist->{images})
       - type: 'link'

    7. _playlistItem($client, $playlist) — builds OPML link item for playlist navigation:
       - name: $playlist->{name}
       - url: \&_playlistFeed
       - passthrough: [{ playlistId => $playlist->{id} }]
       - image: _largestImage($playlist->{images})
       - line2: $playlist->{owner}{display_name} // ''
       - type: 'link'

    - Library item: name => cstring($client, 'PLUGIN_SPOTON_LIBRARY'), url => \&_libraryFeed, type => 'link'

    **C. _homeFeed($client, $callback, $args)** — per D-02, returns three navigation items:
    - Recently Played: name => cstring RECENTLY_PLAYED, url => \&_recentlyPlayedFeed, type => 'link'
    - Made For You: name => cstring MADE_FOR_YOU, url => \&_madeForYouFeed, type => 'link'
    - Top Tracks: name => cstring TOP_TRACKS, url => \&_topTracksFeed, type => 'link'
    Immediately call $callback->({ items => \@items }).

    **D. _recentlyPlayedFeed($client, $callback, $args)** — cursor-based, no offset pagination (Pitfall 4):
    Call Client->getRecentlyPlayed with limit => 50. Map $data->{items} through _trackItem (access $_->{track}). Call $callback->({ items => \@items }). No total field (single-page).

    **E. _madeForYouFeed($client, $callback, $args)** — fetches all user playlists, filters Made-For-You:
    Call Client->getUserPlaylists with offset => 0, limit => 50. Filter items where _isMadeForYou is true. Map through _playlistItem. Call $callback->({ items => \@items }). Note: if user has >50 playlists, some Made-For-You may be missed — acceptable for Phase 3 (Spotify typically puts them in first 50).

    **F. _topTracksFeed($client, $callback, $args)** — per D-05, time_range=medium_term:
    Call Client->getTopTracks with time_range => 'medium_term', limit => 50. Map $data->{items} through _trackItem. Call $callback->({ items => \@items }).

    **G. _libraryFeed($client, $callback, $args)** — per D-03, returns four navigation items:
    - Liked Songs: name => cstring LIKED_SONGS, url => \&_savedTracksFeed, type => 'link'
    - Albums: name => cstring ALBUMS, url => \&_savedAlbumsFeed, type => 'link'
    - Artists: name => cstring ARTISTS, url => \&_followedArtistsFeed, type => 'link'
    - Playlists: name => cstring PLAYLISTS, url => \&_userPlaylistsFeed, type => 'link'
    Immediately call $callback->({ items => \@items }).

    **H. _savedTracksFeed($client, $callback, $args)** — offset-paginated per RESEARCH Pattern 1:
    Map LMS $args->{index} to Spotify offset, $args->{quantity} to limit (cap at 50). Call Client->getSavedTracks. Map $data->{items} through _trackItem (access $_->{track}). Return with total => $data->{total} for LMS pagination. Per NAV-08: unconditional access (no gating). Per NAV-09: API default sort is added_at desc.

    **I. _savedAlbumsFeed($client, $callback, $args)** — offset-paginated:
    Same pagination mapping. Call Client->getSavedAlbums. Map $data->{items} through _albumItem (access $_->{album}). Return with total.

    **J. _followedArtistsFeed($client, $callback, $args)** — cursor-based (Pitfall 4/7):
    Call Client->getFollowedArtists with limit => 50. Response structure: $data->{artists}{items}. Map through _artistItem. No total, no offset. Single-page.

    **K. _userPlaylistsFeed($client, $callback, $args)** — offset-paginated, EXCLUDES Made-For-You:
    Map pagination. Call Client->getUserPlaylists. Filter items where NOT _isMadeForYou (per D-03). Map through _playlistItem. Return with total => $data->{total}. Note: total from API includes Made-For-You playlists, so the displayed count may be slightly off — acceptable.

    **Error handling pattern:** For all feed callbacks, check if first arg ($data) is undef. If so, return $callback->({ items => [{ name => cstring($client, 'PLUGIN_SPOTON_NO_RESULTS'), type => 'textarea' }] }).
