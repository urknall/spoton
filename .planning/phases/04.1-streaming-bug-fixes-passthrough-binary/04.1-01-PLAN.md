  Lines 225-227 currently contain:
    PLUGIN_SPOTON_NORMALIZATION_DESC
    	DE
    	EN

  The empty values after the language code cause LMS Slim::Utils::Strings::parseStrings to fail because its regex requires `.+` (at least one character) after the second tab.

  Replace lines 225-227 with:
    PLUGIN_SPOTON_NORMALIZATION_DESC
    	DE	Aktiviert Lautstaerkenormalisierung (ReplayGain) fuer gleichmaessige Lautstaerke
    	EN	Enables volume normalization (ReplayGain) for consistent playback volume

  Note: Use ASCII-only characters in the replacement text (ae instead of ä, ue instead of ü) to avoid encoding issues. The strings.txt file uses plain ASCII for German special characters in other entries (check the existing pattern — actually, the file DOES use UTF-8 umlauts in existing strings like "Lautstärkenormalisierung" on line 222). Match the existing file encoding. Read the file first to determine the correct encoding, then use the same encoding for the new text. The existing DE strings use proper UTF-8 umlauts, so use: "Aktiviert Lautstärkenormalisierung (ReplayGain) für gleichmäßige Lautstärke"

  **basic.html fix (Bug 6):**

  Line 54 currently reads:
    class="submit" id="startOAuthBtn"[% IF NOT clientId %] disabled[% END %]/>

  The template variable `clientId` is set by Settings.pm handler (line 123: `$paramRef->{clientId} = $prefs->get('clientId') || ''`). However, the LMS Settings framework warns when non-pref_ prefixed variable names are used in settings templates.

  Change line 54 from:
    [% IF NOT clientId %] disabled[% END %]
  to:
    [% IF NOT prefs.pref_clientId %] disabled[% END %]

  This uses the standard LMS prefs object which is automatically populated by the prefs() method in Settings.pm (which already declares 'clientId' in its return list on line 29).
