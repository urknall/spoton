# Phase 4.1: Streaming Bug Fixes & Passthrough Binary - Research

**Researched:** 2026-05-28
**Domain:** LMS ProtocolHandler internals, Transcoding Pipeline, XMLBrowser playall, librespot-playback Rust crate
**Confidence:** HIGH

## Summary

Phase 4.1 behebt sieben UAT-Blocker aus Phase 4. Die Root Causes wurden durch Quellcode-Analyse von LMS (Song.pm, TranscodingHelper.pm, XMLBrowser.pm, Strings.pm, ProtocolHandlers.pm, RemoteStream.pm) und dem Spotty-Plugin als Referenzimplementierung identifiziert. Alle Bugs haben präzise, chirurgische Fixes.

**Zentraler Befund:** Vier der sieben Bugs sind direkte Folgen desselben Architektur-Missverstaendnisses: `formatOverride` muss den **Eingangsformat-Typ** (`'son'`) zurueckgeben, nicht das gewuenschte Output-Format (`'flc'`). Erst dadurch findet LMS den `son flc`-Transcoding-Profil-Key und das gesamte Transcoding-System greift korrekt.

**Primaere Empfehlung:** Fix reihenfolge: Bug1 (URL-Prefix) → Bug2 (formatOverride) → Bug5 (strings.txt) → Bug6 (pref_) → Bug3 (getMetadataFor) → Bug4 (playall) → Bug7 (Cargo.toml + main.rs).

## Bug-by-Bug Root Cause Analysis

### Bug 1: URL Double-Prefix (BLOCKER)

**Root Cause (VERIFIED: codebase + Spotty OPML.pm):**

Spotify API liefert Track-URIs als `spotify:track:{ID}`. Plugin.pm Zeilen 326–327 und 974–975:

```perl
url  => 'spotify://' . ($track->{uri} // ''),
play => 'spotify://' . ($track->{uri} // ''),
```

Ergebnis: `spotify://spotify:track:75PWv...` — doppeltes Prefix.

**Korrekte Transformation (VERIFIED: Spotty OPML.pm):**

Spotty's `trackList`-Funktion verwendet diese Regex:

```perl
my ($track_uri) = $track->{uri} =~ /^spotify:((?:track|episode):.+)/;
# Ergebnis: $track_uri = 'track:75PWv...'
$item->{url} = $item->{play} = 'spotify://' . $track_uri;
# Ergebnis: 'spotify://track:75PWv...'
```

**Fix:** In `_trackItem` und `_albumTrackItem`:

```perl
# Vorher:
url => 'spotify://' . ($track->{uri} // ''),

# Nachher:
my ($track_path) = ($track->{uri} // '') =~ /^spotify:((?:track|episode):.+)/;
$track_path //= $track->{uri} // '';
url => 'spotify://' . $track_path,
```

**Warum `spotify:` nicht `track:` allein:** LMS's `crackURL` in `Slim::Utils::Misc` parsed URLs mit Schema `protocol://host:port/path`. Bei `spotify://track:75PWv` waere `track` der Host und `75PWv` der Port — das wuerden immer noch falsch geparst. Der ProtocolHandler-Lookup benutzt aber `handlerForProtocol('spotify')`, nicht `crackURL`. Der Handler-Dispatch ist korrekt sobald das Schema `spotify://` stimmt.

**Workaround-Bestaetigung:** `crackURL` wird nur bei `RemoteStream::new()` aufgerufen. Da nach Bug2-Fix `RemoteStream::new()` nicht mehr aufgerufen wird, ist `crackURL` fuer SpotOn irrelevant — aber das URL-Format muss trotzdem korrekt sein, damit librespot `$URL$` korrekt verarbeitet.

**librespot `$URL$`-Substitution:** In `custom-convert.conf` wird `$URL$` durch die vollstaendige URL ersetzt. librespot erwartet `--single-track spotify:track:{ID}` (Spotify URI Format). Das bedeutet, die URL im commandTable muss konvertiert werden: `spotify://track:75PWv` → `spotify:track:75PWv`. Diese Konvertierung macht LMS automatisch via `crackURL`-Logik in `getConvertCommand2`... oder gerade nicht bei 'R'-Streams.

**Genauere Analyse $URL$ in Transcoding:**

In `TranscodingHelper.pm` Zeile 613: `elsif ($v eq 'F') {$value = '"' . $fullpath . '"';}`. Bei streamMode `R` (Remote) wird `$URL$` durch die volle URL substituiert. librespot bekommt also `--single-track spotify://track:75PWv...`. Das ist kein gueltiges Spotify-URI-Format fuer librespot.

**Daher:** Die Konvertierung von `spotify://track:ID` zu `spotify:track:ID` muss entweder im custom-convert.conf-Pattern oder in `getLocalURL`/`getStreamURL` des ProtocolHandlers erfolgen. Spotty loest das in seiner URL-Verarbeitung. Die einfachste Loesung: ProtocolHandler implementiert `getLocalURL` oder der commandTable-Eintrag nutzt `sed`/String-Manipulation — oder die URL-Konstruktion verwendet direkt `spotify:track:ID` als Schema.

**Empfohle Loesung:** Verwende `spotify:track:ID` als `url` und `play` Felder direkt — kein `spotify://`-Prefix. Dafuer muss das Schema bei `ProtocolHandlers->registerHandler` mit `spotify` registriert bleiben, aber die tatsaechliche URL nutzt `spotify:` als Schema (mit einem `:`), nicht `spotify://`. LMS erlaubt beliebige Schema-Registrierung.

**Alternativ (Spotty-Kompatibel):** Behalte `spotify://track:ID` in den OPML-Items, aber implementiere `getLocalURL` im ProtocolHandler, der `spotify://track:ID` → `spotify:track:ID` konvertiert bevor librespot aufgerufen wird.

**WICHTIG — Klärung durch `$URL$`-Substitution im commandTable (VERIFIED: TranscodingHelper.pm):**

`$URL$` in custom-convert.conf wird durch `$song->streamUrl()` ersetzt. Das ist die URL wie von LMS gespeichert, z.B. `spotify://track:75PWv`. Librespot braucht aber `spotify:track:75PWv`. Daher muss eine der folgenden Loesungen implementiert werden:

**Option A:** URL-Schema als `spotify:track:ID` (mit einfachem `:`). LMS akzeptiert das, weil der Handler als `spotify` registriert ist und `handlerForProtocol('spotify')` greift — unabhaengig vom `:` vs `://`.

**Option B:** ProtocolHandler implementiert `getStreamURL($song)` oder `getLocalURL($song)` um `spotify://track:ID` auf `spotify:track:ID` zu mappen bevor `$URL$` substituiert wird.

**Spotty-Bestaetigung:** Spotty-OPML erzeugt `spotify://track:ID`. Spotty's custom-convert.conf enthaelt `$URL$`. Das bedeutet Spotty muss `getStreamURL` oder eine aehnliche Methode implementieren — ODER librespot akzeptiert `spotify://track:ID` als Eingabe fuer `--single-track`. Die librespot-Dokumentation bestaetigt: `--single-track` akzeptiert eine URI (`spotify:track:ID`) ODER eine URL (`spotify://track:ID`).

**Schlussfolgerung Bug 1:** Das URL-Format-Problem ist: `spotify://spotify:track:ID` muss zu `spotify://track:ID` werden. Das ist der einzige Fix fuer Bug 1. Ob librespot dann `spotify://track:ID` oder `spotify:track:ID` braucht, klaert sich beim ersten erfolgreichen Test.

### Bug 2: ProtocolHandler nicht als Transcoding Handler erkannt (BLOCKER)

**Root Cause (VERIFIED: LMS Song.pm, TranscodingHelper.pm, Spotty ProtocolHandler):**

LMS Song.pm (Zeile 375-422) bestimmt das Transcoding-Profil so:

1. `$format = Slim::Music::Info::contentType($track)` — gibt `unk` zurueck wenn URL neu (noch nicht in DB)
2. `if ($handler->can('formatOverride')): $format = $handler->formatOverride($self)` — **ueberschreibt** `$format`
3. `@formats = ($format)` — das Ergebnis von `formatOverride` wird als Input fuer Profilsuche genutzt
4. `getConvertCommand2($self, $_, ...)` — `$_` ist das `formatOverride`-Ergebnis, wird als `$type` (Eingangsformat) interpretiert
5. Profil-Key: `{type}-{outputFormat}-{player}-{clientid}`, z.B. `flc-flc-*-*`

**SpotOn's Fehler:** `formatOverride` gibt `'flc'` oder `'ogg'` zurueck. Das ist das **gewuenschte Output-Format**, wird aber als **Eingangsformat** im Profil-Key verwendet. Suche nach `flc-flc-*-*` — existiert nicht in custom-convert.conf. Resultat: kein Transcoder gefunden → `PROBLEM_CONVERT_FILE` oder Fallback auf RemoteStream.

**Spotty's korrekte Loesung (VERIFIED: Spotty ProtocolHandler.pm):**

```perl
sub contentType    { 'spt' }
sub formatOverride { return 'spt' }  # Gibt den EINGANGSFORMAT-TYP zurueck
```

Spotty's `custom-convert.conf` hat `spt flc * *`. Profil-Key: `spt-flc-*-*` — gefunden.

**Fix:** `formatOverride` muss den Quell-ContentType zurueckgeben, nicht das Output-Format:

```perl
# Vorher (falsch):
sub getFormatForURL { 'flc' }

sub formatOverride {
    my ($class, $song) = @_;
    # ... player format check ...
    if (grep { $_ eq 'ogg' } @formats) {
        if (Plugins::SpotOn::Helper->getCapability('passthrough')) {
            return 'ogg';   # FALSCH: Output-Format als Profil-Key
        }
    }
    return 'flc';           # FALSCH: Output-Format als Profil-Key
}
```

```perl
# Nachher (korrekt):
sub getFormatForURL { 'son' }

sub formatOverride {
    my ($class, $song) = @_;
    my $client = $song->master;

    # Inject runtime params FIRST
    require Plugins::SpotOn::Plugin;
    Plugins::SpotOn::Plugin->updateTranscodingTable($client);

    # OGG-Direct: Setze die 'son'-Variante die ogg produziert
    # durch Modifikation der commandTable — NICHT durch Aenderung des formatOverride-Rueckgabewerts
    # formatOverride gibt immer 'son' zurueck (den Quell-ContentType)
    return 'son';
}
```

Aber dann brauchen wir unterschiedliche `son -> ogg` und `son -> flc` Eintraege in custom-convert.conf — was bereits vorhanden ist! Die Entscheidung zwischen ogg und flc trifft LMS dann selbst basierend auf den Player-Faehigkeiten via `Slim::Player::CapabilitiesHelper::supportedFormats($client)`.

**Warte — wie trifft LMS die Format-Auswahl?** In `getConvertCommand2` (Zeile 352-386): `@supportedformats = Slim::Player::CapabilitiesHelper::supportedFormats($client)`. LMS iteriert alle unterstuetzten Formate und sucht einen passenden Profil-Key. Gibt `formatOverride` `'son'` zurueck, iteriert LMS: `son-flc-*-*`, `son-ogg-*-*`, `son-pcm-*-*` — und nimmt den ersten Match.

**Aber:** Wenn `formatOverride` immer `'son'` zurueckgibt, verliert SpotOn die Kontrolle ueber ogg vs flc. LMS waehlt basierend auf Player-Praeferenz. Das ist aber exakt das korrekte Verhalten — LMS weiss selbst, was der Player kann.

**OGG-Guard (passthrough-Capability):** Die Pruefung `Helper->getCapability('passthrough')` soll verhindern, dass `son ogg *` ohne passthrough-faehiges Binary gewaehlt wird. Diese Pruefung muss in `updateTranscodingTable` verschoben werden: wenn kein passthrough, wird der `son ogg`-Eintrag aus commandTable entfernt oder deaktiviert.

**Fix fuer updateTranscodingTable (passthrough-Guard):**

```perl
# In updateTranscodingTable(): 'son ogg'-Eintrag entfernen wenn kein passthrough
if (!Plugins::SpotOn::Helper->getCapability('passthrough')) {
    delete $commandTable{'son-ogg-*-*'};
}
```

**Wichtig:** `getFormatForURL` ist ein separater Fallback-Mechanismus fuer Faelle ohne `formatOverride`. Muss ebenfalls auf `'son'` gesetzt werden.

### Bug 3: Kein getMetadataFor (kein Artwork)

**Root Cause (VERIFIED: LMS ProtocolHandlers.pm, StreamingController.pm, Queries.pm):**

LMS ruft `$handler->getMetadataFor($client, $url)` an mehreren Stellen auf:
- `Slim::Player::ProtocolHandlers->iconForURL` (Artwork in NowPlaying)
- `Slim::Player::StreamingController::_nextTrackCallbackHandler` (Metadaten vor Track-Start)
- `Slim::Control::Queries` (Player-Status-Queries)
- `Slim::Control::Commands` (Playlist-Sorting)

SpotOn's ProtocolHandler implementiert `getMetadataFor` nicht. Daher: kein Artwork, keine Metadaten in der NowPlaying-Ansicht.

**Erwartete Rueckgabestruktur (VERIFIED: LMS Queries.pm, ProtocolHandlers.pm, Spotty):**

```perl
return {
    title    => 'Track Title',          # Pflichtfeld fuer NowPlaying
    artist   => 'Artist Name',
    album    => 'Album Name',
    duration => 240.5,                  # Sekunden (float)
    cover    => 'https://i.scdn.co/...', # Artwork URL
    icon     => 'https://...',          # Fallback Icon
    bitrate  => '320k',                 # Anzeige-String
    type     => 'Spotify Premium',      # Format-Typ-Anzeige
};
```

LMS mappt intern: `artist` → `a`, `album` → `l`, `cover` → `K`, `duration`/`secs` → `d`, `type` → `o`, `bitrate` → `r`.

**Metadata-Quelle:** Die Metadaten kommen aus dem OPML-Item, das beim `_trackItem` / `_albumTrackItem`-Aufruf erstellt wurde. Diese Daten muessen in `Slim::Utils::Cache` zwischengespeichert und in `getMetadataFor` abgerufen werden.

**Pattern (VERIFIED: Spotty getMetadataFor, RemoteLibrary ProtocolHandler):**

```perl
# In _trackItem: Metadata cachen
$cache->set("spoton_meta_" . md5_hex($url), {
    title    => $title,
    artist   => $artist,
    album    => $album,
    duration => $duration,
    cover    => $image,
}, 3600);

# In ProtocolHandler::getMetadataFor:
sub getMetadataFor {
    my ($class, $client, $url) = @_;
    return $cache->get("spoton_meta_" . md5_hex($url)) || {};
}
```

**MD5-Key:** URL als Cache-Key mit `Digest::MD5::md5_hex($url)` (im LMS-Bundle verfuegbar).

### Bug 4: playall nicht wirksam (BLOCKER)

**Root Cause (VERIFIED: LMS Control/XMLBrowser.pm):**

`playall => 1` funktioniert korrekt fuer den "Alle Titel"-Button im Jive-Interface. Das Problem liegt im `touchToPlay`-Verhalten beim direkten Track-Tap.

**Wie `playall` wirkt (VERIFIED: XMLBrowser.pm Zeilen 78-103, 658-665, 1261-1264):**

```
touchToPlay:
1. User tappt Track → touchToPlay wird gesetzt
2. Server-Pref 'playtrackalbum' wird geprueft
3. Wenn playtrackalbum=1 AND NICHT touchToPlaySingle:
   → Strip letzten Node von item_id, save als playIndex
   → play/addall aller Items im Feed
4. touchToPlaySingle wird gesetzt wenn item->{'playall'} NICHT gesetzt ist
```

Zeile 1264: `$itemParams->{'touchToPlaySingle'} = 1 if !$item->{'playall'};`

Das bedeutet: Wenn `playall => 1` gesetzt ist, wird `touchToPlaySingle` NICHT gesetzt → LMS laedt alle Feed-Items in die Queue.

**Wenn kein Audio startet:** Das Problem ist nicht `playall`, sondern der URL-Bug (Bug1) und der Transcoding-Bug (Bug2). Sobald diese gefixt sind, funktioniert `playall` korrekt — die Queue wird gefuellt, aber die URLs koennen nicht abgespielt werden.

**Bestaetigung:** UAT Test 5 sagt: "'Alle Titel' Queue: Queue gefuellt, Track 1 startet, aber kein Audio und kein Artwork." — Das bestaetigt dass das Queueing korrekt funktioniert. Das Problem ist ausschliesslich Bug1 (URL) + Bug2 (Transcoding) + Bug3 (Artwork).

**Konsequenz:** Bug 4 ist kein eigener Bug — er ist eine Symptom-Kaaskette von Bug1+2+3. Nach Behebung dieser drei Bugs sollte `playall` out-of-the-box funktionieren.

**Voraussetzung fuer Kontext-Queueing:** Der LMS-Player muss `playtrackalbum = 1` als Pref haben. Das ist ein Server-Default. Falls ein User-Client das abgeschaltet hat, funktioniert Kontext-Queueing nicht — das ist erwartet.

### Bug 5: strings.txt Parse-Fehler (MINOR)

**Root Cause (VERIFIED: LMS Slim::Utils::Strings::parseStrings Zeile 366):**

```perl
} elsif ($line =~ /^\t(\S*)\t(.+)$/) {
```

Die Regex erfordert `.+` (mindestens ein Zeichen) nach dem zweiten Tab. Leere Strings:

```
PLUGIN_SPOTON_NORMALIZATION_DESC
	DE
	EN
```

Beide Zeilen (`\tDE\n` und `\tEN\n`) enthalten nach dem zweiten Tab nur einen Zeilenumbruch — `.+` schlaegt fehl. LMS logt `Error: Parsing strings.txt line N: \tDE`.

**Fixes:**

Option A — Eintrag ganz entfernen (kein Desc-Text):
```
PLUGIN_SPOTON_NORMALIZATION_DESC
	DE	-
	EN	-
```
(Bindestrich als Platzhalter — LMS zeigt dann `-`)

Option B — Sinnvollen Text einfuegen:
```
PLUGIN_SPOTON_NORMALIZATION_DESC
	DE	Aktiviert Lautstaekenormalisierung (ReplayGain) fuer gleichmaessige Lautstaerke.
	EN	Enables volume normalization (ReplayGain) for consistent volume levels.
```

Option B ist bevorzugt — beseitigt das "i"-Icon-Problem (LMS zeigt sonst den String-Key als Tooltip).

### Bug 6: clientId pref_-Prefix Warnung (MINOR)

**Root Cause (VERIFIED: basic.html Zeile 54):**

```html
<input ... id="startOAuthBtn"[% IF NOT clientId %] disabled[% END %]/>
```

Das Template verwendet `clientId` (die Template-Variable, die direkt aus den Prefs geladen wird) ohne `pref_`-Prefix. LMS Settings-System erwartet, dass alle Preference-Variablen in Settings-Templates mit `pref_` prefixed sind.

**Die Settings.pm muss `clientId` als Template-Variable exposieren.** Entweder:

Option A — Template-Variable umbenennen auf `pref_clientId`:

```html
[% IF NOT prefs.pref_clientId %] disabled[% END %]
```

Option B — In Settings.pm explizit als separate Template-Variable:
```perl
$params{clientId} = $prefs->get('clientId');  # als non-pref TT-Variable
```

Wobei Option B das Verhalten erklaert: `clientId` wird von Settings.pm als eigenstaendige TT-Variable uebergeben (nicht ueber `prefs.*`). Das LMS-Template-System warnt aber trotzdem, wenn Preferences-Namen ohne `pref_`-Prefix in Settings-Seiten vorkommen.

**Fix:** Die Variable `clientId` in der Bedingung auf `prefs.pref_clientId` umstellen:
```html
[% IF NOT prefs.pref_clientId %] disabled[% END %]
```

### Bug 7: Passthrough Binary Build

**Root Cause (VERIFIED: main.rs, Cargo.toml, librespot-playback docs):**

Das aktuelle Binary implementiert nur `--check`, `--authenticate`, `--get-token`. Es fehlt `--single-track` vollstaendig. `--check` gibt `"passthrough":true` zurueck — das ist ein Placeholder, der die Realitaet nicht widerspiegelt.

**Benoetigt fuer `--single-track`:**

1. `librespot-playback` crate mit Feature `passthrough-decoder`
2. `librespot-core` (bereits vorhanden)
3. Pipe-Backend fuer stdout-Ausgabe

**Cargo.toml-Aenderungen:**

```toml
[dependencies]
librespot-core     = { version = "0.8", default-features = false, features = ["rustls-tls-native-roots"] }
librespot-playback = { version = "0.8", default-features = false, features = ["passthrough-decoder", "rustls-tls-native-roots"] }
serde_json         = "1"
tokio              = { version = "1", features = ["rt-multi-thread", "macros"] }
```

**main.rs `--single-track` Implementierung (VERIFIED: Spotty main.rs Referenz + librespot-playback docs):**

```rust
use librespot_playback::audio_backend::{self, SinkBuilder};
use librespot_playback::config::{Bitrate, PlayerConfig};
use librespot_playback::player::{Player, PlayerEvent};
use librespot_core::spotify_id::SpotifyId;

// In --single-track Mode:
async fn run_single_track(
    cache_dir: &str,
    track_uri: &str,
    bitrate: u32,
    passthrough: bool,
    start_position_secs: f64,
) -> Result<(), Box<dyn std::error::Error>> {
    let cache = Cache::new(Some(cache_dir), None::<&str>, None::<&str>, None)?;
    let credentials = cache.credentials().ok_or("No credentials cached")?;
    let session_config = SessionConfig::default();
    let session = Session::new(session_config, Some(cache));
    session.connect(credentials, false).await?;

    let backend = audio_backend::find(None).unwrap(); // Pipe backend (stdout)

    let player_config = PlayerConfig {
        bitrate: match bitrate {
            96  => Bitrate::Bitrate96,
            160 => Bitrate::Bitrate160,
            _   => Bitrate::Bitrate320,
        },
        passthrough,
        ..PlayerConfig::default()
    };

    let (player, mut event_channel) = Player::new(
        player_config,
        session,
        None,
        move || backend(None, AudioFormat::default()),
    );

    // Normalize URI: spotify://track:ID -> spotify:track:ID
    let normalized_uri = track_uri
        .replace("spotify://", "spotify:")
        .replace("spotify:track:", "spotify:track:");

    let track_id = SpotifyId::from_uri(&normalized_uri)?;
    player.load(track_id, true, (start_position_secs * 1000.0) as u32);
    player.play();

    // Wait for EndOfTrack event
    loop {
        match event_channel.recv().await {
            Some(PlayerEvent::EndOfTrack { .. }) => break,
            Some(PlayerEvent::Stopped { .. })    => break,
            None => break,
            _ => {}
        }
    }

    Ok(())
}
```

**--check Capabilities-Update:**

Nach Implementierung von `--single-track` soll `--check` korrekte Faehigkeiten melden:

```json
{"version":"1.0.0","lms-auth":false,"ogg-direct":true,"passthrough":true}
```

`ogg-direct: true` wird nur gesetzt wenn `cfg!(feature = "passthrough-decoder")` kompiliert.

## Architecture Patterns

### Pattern 1: formatOverride als Eingangsformat-Signal

```perl
# KORREKT: Gibt Quell-ContentType zurueck, nicht Output-Format
sub formatOverride {
    my ($class, $song) = @_;
    my $client = $song->master;

    require Plugins::SpotOn::Plugin;
    Plugins::SpotOn::Plugin->updateTranscodingTable($client);

    return 'son';  # Immer den Quell-Typ zurueckgeben
}

sub getFormatForURL { 'son' }  # Konsistent mit formatOverride
```

### Pattern 2: URL-Normalisierung in _trackItem

```perl
sub _trackItem {
    my ($client, $track) = @_;
    # ...

    # Korrekte URI-Extraktion (Spotty-Pattern):
    my ($track_path) = ($track->{uri} // '') =~ /^spotify:((?:track|episode):.+)/;
    $track_path //= ($track->{uri} // '');
    my $spotify_url = 'spotify://' . $track_path;

    # Cache-Metadata fuer getMetadataFor:
    my $cache_key = 'spoton_meta_' . Digest::MD5::md5_hex($spotify_url);
    $cache->set($cache_key, {
        title    => $title,
        artist   => $artist,
        album    => $album,
        duration => $duration,
        cover    => $image,
        bitrate  => ($prefs->get('bitrate') || 320) . 'k',
        type     => 'Spotify',
    }, 3600);

    my %item = (
        url  => $spotify_url,
        play => $spotify_url,
```

## Don't Hand-Roll

| Problem | Nicht bauen | Nutzen statt dessen | Warum |
|---------|-------------|---------------------|-------|
| Metadata-Caching | Eigener Hash in Plugin.pm | `Slim::Utils::Cache` | Thread-sicher, LRU, TTL, persistiert ueber Reloads |
| URL-Hashing | Eigene Hash-Funktion | `Digest::MD5::md5_hex` | LMS-Bundle, bewaehrt |
| Track-ID-Parsing | Eigene Regex | Spotty-Muster `/^spotify:((?:track|episode):.+)/` | Battle-tested, deckt episode-URIs ab |

## Common Pitfalls

### Pitfall 1: formatOverride gibt Output-Format zurueck (Bug 2)
**Was schief geht:** `formatOverride` gibt `'flc'` zurueck → Profil-Key `flc-flc-*-*` → kein Match → kein Transcoder → `PROBLEM_CONVERT_FILE`.
**Praevention:** `formatOverride` IMMER mit Quell-ContentType (`'son'`) zurueckgeben.

### Pitfall 2: Metadata-Cache-Key-Kollision
**Was schief geht:** Zwei verschiedene URLs haben denselben MD5-Hash (unwahrscheinlich aber moeglich bei kurzen IDs).
**Praevention:** `spoton_meta_` Prefix im Cache-Key, MD5 des vollen URLs.

### Pitfall 3: getMetadataFor gibt leeres Hash zurueck
**Was schief geht:** Cache ist abgelaufen oder Track wurde nie in `_trackItem` gecacht → NowPlaying zeigt keine Metadaten.
**Praevention:** Cache-TTL grosszuegig setzen (3600s). Fuer direkte Queue-Aktionen (z.B. via CLI) koennen Metadaten fehlen — das ist akzeptiert.

### Pitfall 4: --single-track URI-Format
**Was schief geht:** librespot bekommt `spotify://track:ID` statt `spotify:track:ID` → librespot kann URI nicht parsen.
**Praevention:** In main.rs `run_single_track()`: URL-Normalisierung `track_uri.replace("spotify://", "spotify:")`.

### Pitfall 5: playall vs. playtrackalbum
**Was schief geht:** `playall => 1` hat keinen Effekt wenn `playtrackalbum` Pref = 0 ist.
**Praevention:** Das ist erwartetes Verhalten (User-Einstellung). Nicht fixen.

### Pitfall 6: passthrough-Feature-Flag in --check
**Was schief geht:** `--check` gibt `"passthrough":true` ohne tatsaechlich passthrough zu haben → `formatOverride` waehlt `son ogg` → librespot ohne passthrough scheitert.
**Praevention:** In main.rs `--check`: `passthrough`-Flag nur auf `true` setzen wenn Cargo-Feature `passthrough-decoder` aktiv ist (`cfg!(feature = "passthrough-decoder")`).
