# Phase 04.1: Code Review Report

**Reviewed:** 2026-05-28T22:00:00Z
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

Phase 04.1 fixes five Perl-side UAT blockers and implements --single-track mode in the Rust binary. The core bug fixes (URL double-prefix, formatOverride, getMetadataFor, strings.txt, clientId prefix) are correctly implemented and the Rust code compiles cleanly against the librespot 0.8 API.

One critical issue was found: the `_killOrphanedProcesses` function checks whether *any* player is playing (any protocol, not just Spotify) before killing spoton processes. If a non-Spotify player is active, orphaned spoton processes survive indefinitely. Conversely, it could kill actively-transcoding spoton processes if no LMS player reports `isPlaying()` during the brief window between track transitions.

Four warnings were found covering missing album metadata in the NowPlaying cache, a potential regex mismatch for the `--bitrate` substitution pattern, unreachable code after process::exit(), and a process-killing race condition.

## Critical Issues

### CR-01: _killOrphanedProcesses kann aktive Transkodierungsprozesse toeten

**File:** `Plugins/SpotOn/Plugin.pm:118-139`
**Issue:** Die Funktion prueft `$client->isPlaying()` fuer ALLE Player (nicht nur Spotify-Player). Das hat zwei Fehlermodi:

1. **False negative (zu aggressiv):** Waehrend eines kurzen Track-Wechsels (z.B. gapless playback, Queue-Uebergang) meldet kein Player `isPlaying()`, obwohl ein spoton-Transkodierungsprozess gerade gestartet wird. In diesem Fenster wuerde `pkill -f` den gerade gestarteten spoton-Prozess toeten.

2. **False positive (zu passiv):** Wenn ein anderer Player (nicht Spotify) gerade spielt, werden Orphans nie bereinigt, weil `$isBusy` immer gesetzt ist.

Der fundamentale Fehler: `pkill -f '$safeHelper'` toetet ALLE Prozesse mit dem Binary-Namen, inklusive solcher, die LMS gerade aktiv als Transkodierungspipeline nutzt. Ein korrekter Ansatz muss zwischen aktiven Pipeline-Prozessen und echten Orphans unterscheiden.

**Fix:** Die Funktion sollte nur Spotify-spielende Clients pruefen und PIDs der aktiven Transkodierungsprozesse von LMS (via TranscodingHelper oder Song-Objekt) von der Kill-Liste ausnehmen:
```perl
unless ($isBusy) {
    my ($helper) = Plugins::SpotOn::Helper->get();
    if ($helper) {
        # Collect PIDs of active transcoding processes to exclude them
        my %activePids;
        for my $client (Slim::Player::Client::clients()) {
            my $song = $client->playingSong() || next;
            my $pid = $song->currentTrackHandler && $song->transcodeProcess;
            $activePids{$pid} = 1 if $pid;
        }
        # Only kill processes not in the active set
        # (Implementation depends on available LMS APIs for PID tracking)
    }
}
```
Anmerkung: Das genaue LMS-API fuer PID-Tracking haengt von der LMS-Version ab. Im Minimum sollte die Pruefung auf Spotify-URLs eingeschraenkt werden (pruefen ob `$client->playingSong()->currentTrack()` eine spotify:// URL ist).

## Warnings

### WR-01: Album-Name fehlt im Metadata-Cache fuer Album-Track-Kontext

**File:** `Plugins/SpotOn/Plugin.pm:997`
**Issue:** In `_albumTrackItem` wird `$albumName` immer als leerer String gesetzt und in den Metadata-Cache geschrieben. Der Kommentar behauptet "Album name not available in simplified track objects from album context" — aber der Album-Name IST im aufrufenden Code verfuegbar: `_albumFeed` hat `$album->{name}` (Zeile 918-930). Er wird nur nie an `_albumTrackItem` durchgereicht.

Das bedeutet: Wenn ein Track aus einem Album-Kontext abgespielt wird, zeigt NowPlaying keinen Album-Namen an. Wenn derselbe Track spaeter aus der Suche oder Bibliothek abgespielt wird, ueberschreibt `_trackItem` den Cache-Eintrag mit dem korrekten Album-Namen. Aber die umgekehrte Reihenfolge ist schlechter: Ein guter Cache-Eintrag aus `_trackItem` wird durch den leeren Album-Namen aus `_albumTrackItem` ueberschrieben.

**Fix:** `$albumName` (oder den Album-Namen) als 5. Parameter an `_albumTrackItem` uebergeben:
```perl
# In _albumFeed (Zeile 930):
my @items = map { _albumTrackItem($client, $_, $images, $artist0, $album->{name}) } @{$tracks};

# In _albumTrackItem Signatur (Zeile 967):
my ($client, $track, $albumImages, $albumArtist, $albumName) = @_;
# Dann Zeile 997 entfernen: my $albumName = '';
# Falls $albumName undef ist (2. Seite via getAlbumTracks), Fallback:
$albumName //= '';
```

### WR-02: Bitrate-Regex matcht nicht bei 4-stelligen oder unbekannten Werten

**File:** `Plugins/SpotOn/Plugin.pm:1110`
**Issue:** Die Regex `--bitrate \d{2,3}` matcht nur 2- oder 3-stellige Zahlen. Wenn der commandTable-Eintrag aus irgendeinem Grund einen anderen Wert enthaelt (z.B. nach einem fehlerhaften manuellen Edit oder wenn ein zukuenftiges Spotify-Tier 1411 kbps unterstuetzt), wuerde die Substitution stillschweigend fehlschlagen und der alte Wert bestehen bleiben.

Wichtiger: Die Regex nutzt kein `/g`-Flag, also wird nur das erste Vorkommen ersetzt. Das ist aktuell OK (nur ein `--bitrate` pro Eintrag), aber fragil.

**Fix:** Robustere Regex verwenden:
```perl
$commandTable->{$key} =~ s/--bitrate \d+/--bitrate $bitrate/;
```

### WR-03: Unreachable Code nach process::exit() in main()

**File:** `librespot-spoton/src/main.rs:271`
**Issue:** Am Ende von `main()` steht `let _ = name_provided;` (Zeile 271). Dieses Statement ist nach dem `match mode` Block unerreichbar, weil JEDER Arm entweder `process::exit()` aufruft oder die Funktion normal beendet (Mode::Check druckt und faellt durch).

Konkret: Nur `Mode::Check` erreicht Zeile 271, da es keinen `process::exit()` hat. Fuer alle anderen Modi ist diese Zeile toter Code. Das `name_provided` Warning wird tatsaechlich nur fuer `Mode::Check` unterdrueckt.

Das eigentliche Problem: `Mode::Check` beendet `main()` ohne `process::exit(0)`, was den Tokio-Runtime-Cleanup abwarten laesst. Das ist funktional korrekt (Exit-Code 0 durch normalen Return), aber inkonsistent mit allen anderen Modi.

**Fix:** `process::exit(0)` am Ende des `Mode::Check`-Arms hinzufuegen und `let _ = name_provided;` in den Variablen-Deklarationsbereich verschieben:
```rust
Mode::Check => {
    println!("ok spoton v{}", VERSION);
    let has_passthrough = cfg!(feature = "passthrough-decoder");
    let json = serde_json::json!({ ... });
    println!("{}", json);
    process::exit(0);  // Konsistent mit anderen Modi
}
```

### WR-04: pkill Race-Condition bei schnellem Track-Wechsel

**File:** `Plugins/SpotOn/Plugin.pm:138`
**Issue:** `pkill -f '$safeHelper'` toetet alle Prozesse, deren Kommandozeile den Helper-Pfad enthaelt. Waehrend eines schnellen Track-Wechsels (z.B. Skip) kann folgendes passieren:

1. Alter spoton-Prozess laeuft noch (gerade beendet sich)
2. LMS startet neuen spoton-Prozess fuer naechsten Track
3. `_killOrphanedProcesses` Timer feuert (unwahrscheinlich im 1h-Intervall, aber moeglich)
4. `pkill` toetet BEIDE Prozesse

Das ist ein Edge-Case, der durch den 1-Stunden-Intervall sehr unwahrscheinlich ist. Aber das fundamentale Design (alle Prozesse mit dem Namen toeten) ist fragil. In Kombination mit CR-01 (fehlende Unterscheidung aktiv vs. orphan) wird dies zum realen Risiko.

**Fix:** Siehe CR-01-Fix. Alternativ: PID-Tracking in Plugin.pm einfuehren, sodass nur bekannte Orphan-PIDs getoetet werden.

## Info

### IN-01: Unnoetige doppelte require-Anweisung fuer Helper

**File:** `Plugins/SpotOn/Plugin.pm:1135`
**Issue:** `require Plugins::SpotOn::Helper;` steht in `updateTranscodingTable` (Zeile 1135), aber Helper.pm wird bereits in `initPlugin` (Zeile 50) geladen. `require` in Perl ist idempotent (prueft `%INC`), also ist dies kein Bug, aber unnoetig und suggeriert faelschlicherweise, dass Helper erst hier geladen wird.

**Fix:** Entfernen oder durch Kommentar ersetzen:
```perl
# Helper already loaded in initPlugin
unless (Plugins::SpotOn::Helper->getCapability('passthrough')) {
```

### IN-02: Fehlende Dokumentation fuer $CACHE$ Makro in custom-convert.conf

**File:** `Plugins/SpotOn/custom-convert.conf:3`
**Issue:** Der Wert `"$CACHE$"` in custom-convert.conf ist kein Standard-LMS-Makro (LMS kennt `$FILE$`, `$URL$`, `$START$` etc., aber nicht `$CACHE$`). Es ist ein Platzhalter, der von `updateTranscodingTable` per Regex ersetzt wird. Das ist ein cleverer Ansatz, aber nirgends dokumentiert. Ein Entwickler, der custom-convert.conf liest, wuerde `$CACHE$` als LMS-Variable interpretieren.

**Fix:** Kommentar in custom-convert.conf hinzufuegen:
```
# Note: "$CACHE$" is NOT an LMS macro -- it is a placeholder replaced at runtime
# by Plugin.pm updateTranscodingTable() with the actual spoton cache directory.
```
