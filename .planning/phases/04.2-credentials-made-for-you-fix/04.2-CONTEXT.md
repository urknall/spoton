# Phase 04.2: Credentials + Made For You Fix - Context

**Gathered:** 2026-05-28
**Status:** Ready for planning

<domain>
## Phase Boundary

SpotOn provisioniert eigene librespot-Credentials aus dem bestehenden OAuth-Token (kein manueller Copy von Spotty-Cache) und der "Für dich gemacht"-Feed nutzt den Spotify Category-Endpoint statt des broken `me/playlists` owner-Filters.

</domain>

<decisions>
## Implementation Decisions

### Credential-Provisionierung
- **D-01:** Token-Bridge als primäre Strategie — Nach erfolgreichem OAuth PKCE Callback nutzt das Plugin den Access Token, um über einen neuen Binary-Modus (`--token-login --token <access_token> --cache <dir>`) librespot-kompatible Credentials zu erzeugen. librespot erstellt eine Session mit dem Token, Spotify's AP antwortet mit `reusable_auth_credentials`, die als `credentials.json` gecacht werden.
- **D-02:** ZeroConf-Discovery als Fallback — Falls librespot 0.8 keine Token-basierte Credential-Erstellung unterstützt, startet das Plugin das Binary kurz im Connect-Modus (`--enable-discovery`), User verbindet einmal über die Spotify-App, Credentials werden automatisch gecacht. Research muss klären ob `Credentials::with_access_token` oder ein äquivalenter Mechanismus in librespot 0.8 existiert.
- **D-03:** Credential-Provisioning wird automatisch nach dem OAuth-Callback getriggert — kein separater User-Schritt nötig. Plugin ruft den Binary-Modus direkt auf, prüft ob `credentials.json` existiert, und meldet Erfolg/Fehler im Settings-UI.
- **D-04:** Spotty-Credentials-Hack entfernen — Die manuelle Kopie von `/var/lib/squeezeboxserver/cache/spotty/.../credentials.json` nach `/var/lib/squeezeboxserver/cache/spoton/` wird durch den eigenen Provisioning-Mechanismus ersetzt. SpotOn funktioniert danach komplett unabhängig von Spotty.

### Made For You Feed
- **D-05:** Category-Endpoint als primäre Quelle — `GET /browse/categories/0JQ5DAt0tbjZptfcdMSKl3/playlists` (Spotty-bewährte hardcoded Category-ID für Personal Mixes). Neue API-Methode `getPersonalMixes` in Client.pm.
- **D-06:** Fallback auf Name-Matching — Falls der Category-Endpoint im Dev Mode nicht verfügbar ist (HTTP 404/403): Fallback auf `me/playlists` mit Regex-basiertem Name-Matching (Daily Mix, Discover Weekly, Release Radar etc.) statt `owner.id == 'spotify'` Filter.
- **D-07:** Spotify-Reihenfolge beibehalten — Keine eigene Sortierung der Playlists. Anzeige in der Reihenfolge wie sie von der API kommen.
- **D-08:** `_madeForYouFeed` in Plugin.pm umbauen — von `getUserPlaylists` + owner-Filter auf `getPersonalMixes` + Name-Matching-Fallback.

### Claude's Discretion
- Exakter `--token-login` Mechanismus in librespot 0.8 (welcher auth_type, Credentials-Konstruktor)
- Name-Matching Regex-Patterns für den Fallback (Research prüft aktuelle Spotify-Playlist-Namen)
- Error-Handling wenn Credential-Provisioning fehlschlägt (Retry-Strategie, UI-Feedback)
- Ob `credentials.json` per Account-ID subdirectory gespeichert wird (wie Spotty: `cache/spoton/{account_hash}/credentials.json`) oder flach

</decisions>

## Existing Code Insights

### Reusable Assets
- `Plugins/SpotOn/Helper.pm` — Binary-Finder (`_findBin`), Capability-Check (`getCapability`), kann für `--token-login`-Aufruf wiederverwendet werden
- `Plugins/SpotOn/API/TokenManager.pm` — Hat bereits `refreshToken()` und `_cacheToken()` — Credential-Provisioning kann dort nach erfolgreichem OAuth-Callback eingehängt werden
- `librespot-spoton/src/main.rs` — `run_authenticate()` als Vorlage für `run_token_login()`; `Cache::new()` und `Session::connect()` Pattern bereits etabliert

### Established Patterns
- Binary-Modi-Pattern: enum Mode + CLI-Parsing + async run_* Funktion (Check, Authenticate, GetToken, SingleTrack — TokenLogin folgt demselben Schema)
- Cache-Dir-Pattern: `catdir($serverPrefs->get('cachedir'), 'spoton')` in Plugin.pm Zeile 1089
- API-Methoden-Pattern: `$class->_request('get', $path, $params, $cb)` in Client.pm

### Integration Points
- `TokenManager.pm` OAuth Callback → Credential-Provisioning (nach successful token exchange)
- `Plugin.pm updateTranscodingTable` → liest Cache-Dir, das jetzt eigene Credentials haben muss
- `Plugin.pm _madeForYouFeed` → neue getPersonalMixes() statt getUserPlaylists()
