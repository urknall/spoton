## Summary

Phase 04.2 loest zwei unabhaengige, aber dringliche Probleme: (1) SpotOn muss librespot-kompatible Credentials selbst provisionieren, ohne auf Spotty's Cache angewiesen zu sein. (2) Der "Fuer dich gemacht"-Feed nutzt den falschen Filter (`owner.id == 'spotify'`), der in Dev Mode nicht mehr funktioniert, und muss auf den bewaehrten Category-Endpoint umgestellt werden.

**Credential-Provisionierung:** Die Research bestaetigt, dass `Credentials::with_access_token()` in librespot-core 0.8.0 **vorhanden und funktional ist** (verifiziert im lokalen Cargo-Cache). Der Mechanismus ist sogar in session.rs dokumentiert: Bei `AUTHENTICATION_SPOTIFY_TOKEN` verbindet librespot zweimal — einmal mit dem Token, dann sofort nochmal mit den zurueckgegebenen `reusable_credentials`. Mit `store_credentials=true` wird `credentials.json` in `cache_dir` gespeichert. D-01 ist damit vollstaendig umsetzbar ohne Fallback (D-02 ist damit ueberfluessig aber bleibt als Sicherheit).

**Made For You Feed:** Die Category-ID `0JQ5DAt0tbjZptfcdMSKl3` ist in Spotty bewaehrt (verifiziert in `/home/sti/spotty-ng/Spotty-Plugin/API.pm:18`). Das Regex-Pattern fuer den Fallback ist ebenfalls aus Spotty uebernommen (OPML.pm:394-396). Der Client.pm-Request-Pattern ist etabliert und einfach zu erweitern.

**Primary recommendation:** `--token-login` Mode in `main.rs` implementieren, der `Credentials::with_access_token()` nutzt und `session.connect(creds, true)` aufruft — kein Fallback auf ZeroConf benoetigt.

### Pattern 1: Token-Login Mode in main.rs

**Was:** Neuer `Mode::TokenLogin` analog zu `Mode::Authenticate`.
**Wann:** Nach OAuth PKCE Callback, getriggert von TokenManager.pm.

```rust
// Source: librespot-core-0.8.0/src/authentication.rs (VERIFIED: local cargo cache)
// Credentials::with_access_token() existiert in librespot-core 0.8.0

// Neuer Mode in enum Mode:
// TokenLogin,

// run_token_login() Vorlage (analog run_authenticate()):
async fn run_token_login(
    access_token: &str,
    cache_dir: &str,
) -> Result<(), Box<dyn std::error::Error>> {
    // AUTHENTICATION_SPOTIFY_TOKEN auth_type
    let credentials = Credentials::with_access_token(access_token);

    // Cache mit credentials_path = cache_dir
    let cache = Cache::new(Some(cache_dir), None::<&str>, None::<&str>, None)?;

    let session_config = SessionConfig::default();
    let session = Session::new(session_config, Some(cache));

    // store_credentials=true: schreibt reusable_credentials nach cache_dir/credentials.json
    // WICHTIG: librespot verbindet bei AUTHENTICATION_SPOTIFY_TOKEN zweimal intern
    // (siehe session.rs:184-201 — automatischer Reconnect mit reusable_credentials)
    session.connect(credentials, true).await?;

    // Nach connect(): credentials.json ist in cache_dir geschrieben
    println!("credentials_saved");
    Ok(())
}
```

**CLI-Contract:**
```
spoton -n 'SpotOn' --token-login --token <access_token> --cache <dir>
stdout: "credentials_saved" on success
exit 0 on success, non-zero on failure
```

**Capability-Flag:** `"token-login": true` im `--check` JSON-Manifest hinzufuegen.

### Pattern 2: TokenManager Provisioning-Trigger

**Was:** `_provisionCredentials` nach erfolgreichem `_storeTokens` aufrufen.
**Wann:** Einmalig nach OAuth Callback und nach jedem erfolgreichen `refreshToken` (optional).

```perl
# In TokenManager.pm::_storeTokens, nach $cb->($accountId):
# Credential-Provisioning asynchron triggern (kein Blockieren des Callbacks)
$class->_provisionCredentials($accountId, $accessToken);

# Neue private Methode:
sub _provisionCredentials {
    my ($class, $accountId, $accessToken) = @_;

    require Plugins::SpotOn::Helper;
    my ($helper) = Plugins::SpotOn::Helper->get();
    unless ($helper) {
        $log->warn("TokenManager: cannot provision credentials — binary not found");
        return;
    }

    # Cache-Dir analog zu Plugin.pm::updateTranscodingTable (Zeile 1109)
    my $serverPrefs = preferences('server');
    my $cacheDir    = File::Spec::Functions::catdir(
        $serverPrefs->get('cachedir'), 'spoton'
    );

    # Async: non-blocking fork via Slim::Utils::Network::async_resolver
    # ODER: blocking system() call, da kurz (< 2s) und einmalig nach OAuth.
    # Empfehlung: blocking system() mit Timeout-Wrapper, analog zu helperCheck() in Helper.pm.
    (my $safeToken = $accessToken) =~ s/'/'\\''/g;
    (my $safeDir   = $cacheDir)    =~ s/'/'\\''/g;
    (my $safeHelper = $helper)     =~ s/'/'\\''/g;

    my $cmd = sprintf("'%s' -n 'SpotOn' --token-login --token '%s' --cache '%s' 2>&1",
        $safeHelper, $safeToken, $safeDir);

    my $output = `$cmd`;
    my $exit   = $? >> 8;

    if ($exit == 0 && ($output =~ /credentials_saved/)) {
        main::INFOLOG && $log->info(
            "TokenManager: credentials provisioned for account $accountId");
    } else {
        $log->error(
            "TokenManager: credential provisioning failed for $accountId (exit $exit)");
        # Kein hard failure — Streaming funktioniert weiterhin wenn credentials.json
        # bereits vorhanden (z.B. aus frueherem --authenticate).
    }
}
```

**Wichtig:** Token darf NICHT geloggt werden (T-02.1-03). Der `sprintf`-Call muss den Token mit `[REDACTED]` in Log-Ausgaben ersetzen.

### Anti-Patterns to Avoid

- **Token in Log-Ausgabe:** `$accessToken` darf NIEMALS in `$log->` erscheinen (T-02.1-03). Beim Aufbau des Shell-Kommandos in `_provisionCredentials` ist der Token in der Log-Ausgabe durch `[REDACTED]` zu ersetzen.
- **Blocking HTTP im Provisioning-Pfad:** `_provisionCredentials` wird in `_storeTokens` aufgerufen, das selbst ein SimpleAsyncHTTP-Callback ist. Ein synchrones `system()` ist dort zwar technisch moeglich (kurze Laufzeit), aber muss dokumentiert werden. Alternativ: `Slim::Utils::Timers::setTimer($class, Time::HiRes::time() + 0.1, sub { ... })` fuer deferred Aufruf.
- **credentials.json im Spotty-Verzeichnis:** SpotOn nutzt ausschliesslich `catdir($serverPrefs->get('cachedir'), 'spoton')`, nie `spotty`.
- **owner.id-Filter beibehalten:** Der alte Filter (`owner.id eq 'spotify'`) ist in Dev Mode kaputt. Beide Stellen (Feed + Library-Filter) muessen aktualisiert werden.
- **Category-Endpoint als einzige Quelle ohne Fallback:** In Dev Mode gibt `browse/categories/...` HTTP 403 oder 404 zurueck. Immer Fallback implementieren.

## Common Pitfalls

### Pitfall 1: Token-Auth erfordert doppelten Connect-Handshake
**Was schiefgeht:** `session.connect()` mit `AUTHENTICATION_SPOTIFY_TOKEN` schlaegt fuer Keymaster-basierte Anfragen (z.B. `get-token`) fehl, weil token-authed sessions Keymaster nicht nutzen koennen.
**Warum:** librespot-core session.rs:184-201 — bei `AUTHENTICATION_SPOTIFY_TOKEN` reconnectet librespot intern automatisch mit den zurueckgegebenen `reusable_credentials`. Dieser zweite Connect ist transparent fuer den Aufrufer. Nach `connect()` sind die reusable_credentials gespeichert und der Store-Typ ist `AUTHENTICATION_STORED_SPOTIFY_CREDENTIALS` — kompatibel mit Keymaster.
**Wie vermeiden:** `run_token_login()` muss nur `session.connect(creds, true)` aufrufen und danach `process::exit(0)`. Kein weiterer manueller Reconnect-Code noetig.
**Warnsignale:** Binary haengt oder gibt Timeout-Fehler nach dem ersten Connect-Versuch.

### Pitfall 2: Category-Endpoint Antwortstruktur
**Was schiefgeht:** `$data->{items}` ist undef, weil die Category-Playlist-Antwort `{playlists: {items: [...]}}` zurueckgibt, nicht `{items: [...]}`.
**Warum:** `/browse/categories/{id}/playlists` gibt ein `PlaylistSimplifiedPaged`-Objekt unter dem Key `playlists` zurueck, nicht direkt ein Array unter `items`. Spotty's `categoryPlaylists` verarbeitet das in der `browse()`-Methode automatisch durch Extraktion des `playlists`-Schluessel-Werts.
**Wie vermeiden:** In `_madeForYouFeed` immer `$data->{playlists}{items}` verwenden, nicht `$data->{items}`.
**Warnsignale:** Leeres Menu, kein Fehler-Logging.

### Pitfall 3: Token-Wert im Shell-Kommando
**Was schiefgeht:** Access Token erscheint im System-Prozess-Listing (`ps aux`) oder in LMS-Logs.
**Warum:** `system()` / Backtick-Aufruf mit Token als CLI-Argument ist in Prozesslisten sichtbar.
**Wie vermeiden:** Token per Environment-Variable uebergeben statt als CLI-Argument, ODER akzeptieren (kurzlebig, nur auf dem lokalen System) und sicherstellen, dass der Token nie geloggt wird.
**Empfehlung:** Fuer Phase 04.2 als CLI-Arg akzeptabel (kurze Lebenszeit, nur lokal). Dokumentieren fuer spaetere Haertung.
**Warnsignale:** Token erscheint in `$log->warn/error` Ausgaben.

### Pitfall 4: _isMadeForYou an zwei Stellen
**Was schiefgeht:** `_isMadeForYou` wird auch in `_libraryPlaylistsFeed` (Plugin.pm:675) verwendet, um MFY-Playlists aus der Library zu **entfernen**. Wenn nur `_madeForYouFeed` aktualisiert wird, bleibt die alte Owner-Filter-Logik im Library-Feed aktiv.
**Warum:** Beide Feeds nutzen dieselbe Helper-Funktion zur Klassifikation.
**Wie vermeiden:** `_isMadeForYou` einmal auf Regex-Pattern umstellen — beide Call-Sites profitieren automatisch. Keine selektive Aktualisierung.

### Pitfall 5: Provisioning-Timing (Token noch gueltig?)
**Was schiefgeht:** `_provisionCredentials` wird nach dem OAuth Callback getriggert, wenn der Access Token frisch ist. Bei spaeterem Token-Refresh wird Provisioning nicht erneut ausgefuehrt — `credentials.json` bleibt gueltig (librespot cached reusable credentials, nicht den Access Token).
**Warum:** `credentials.json` enthaelt `reusable_auth_credentials` (ein anderes Credential-Format als der Access Token). Diese sind laenger gueltig und werden von librespot intern aufgefrischt.
**Wie vermeiden:** Provisioning nur einmalig beim OAuth Callback ausfuehren. Kein Re-Provisioning bei Token-Refresh noetig, ausser `credentials.json` existiert nicht (z.B. nach manuellem Loeschen).
**Warnsignale:** `run_single_track` gibt "No cached credentials found" aus, obwohl Token-Refresh erfolgreich war.

### Pitfall 6: Category-Endpoint in Dev Mode
**Was schiefgeht:** `GET /browse/categories/0JQ5DAt0tbjZptfcdMSKl3/playlists` gibt HTTP 403 oder 404 in Dev Mode zurueck (Februar 2026: Browse Categories entfernt in Dev Mode laut CLAUDE.md).
**Warum:** Laut CLAUDE.md Endpoint-Tabelle: `Browse Categories` ist seit Feb 2026 im Dev Mode entfernt.
**Wie vermeiden:** Fallback zwingend implementieren (D-06). Client.pm-`_onError` gibt `{code: 403}` oder `{code: 404}` zurueck — in `_madeForYouFeed` explizit abfangen und auf `getUserPlaylists` + Regex-Fallback umschalten.

async fn run_authenticate(

## Open Questions

1. **Subdirectory-Struktur in credentials.json**
   - Was wir wissen: `Cache::new(Some(cache_dir), ...)` + `save_credentials()` schreibt in `cache_dir`
   - Was unklar: Ob librespot intern noch ein Subdirectory basierend auf dem Username/AccountHash anlegt
   - Empfehlung: Im ersten `run_token_login()`-Testlauf mit `ls -la <cache_dir>` pruefen. Falls Subdirectory, muss `updateTranscodingTable` den Pfad entsprechend anpassen.

2. **Token-Argument Security**
   - Was wir wissen: `--token <value>` erscheint in `ps aux` auf dem lokalen System
   - Was unklar: Ob der LMS-Benutzer das als Sicherheitsproblem betrachtet
   - Empfehlung: Fuer Phase 04.2 als CLI-Arg akzeptieren. In den Kommentaren dokumentieren. Phase 5 kann Environment-Variable evaluieren.

3. **Re-Provisioning nach Token-Rotation**
   - Was wir wissen: `credentials.json` enthaelt `reusable_auth_credentials`, nicht den Access Token — laenger gueltig
   - Was unklar: Wie lange `reusable_credentials` gueltig sind und ob librespot sie automatisch auffrischt
   - Empfehlung: Provisioning nur beim OAuth Callback (einmalig). Wenn `--single-track` "No cached credentials" meldet, Fehler ans UI melden mit Hinweis "neu authentifizieren".
