# Phase 04.2: Code Review Report

**Reviewed:** 2026-05-29
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Phase 04.2 fuegt den `--token-login`-Subcommand zum Rust-Binary hinzu und verdrahtet ihn mit `TokenManager::_provisionCredentials`. Die Kernlogik der Credential-Provisionierung (Token-Login-Flow im Rust-Binary, `_storeTokens`, PKCE-Generierung) ist solide implementiert. Zwei sicherheitsrelevante Blocker erfordern jedoch Korrekturen vor dem Release: Cache-Contamination bei Mehrfachkonten-Setup und eine Command-Injection-Luecke auf Windows. Vier Warnungen betreffen Fallback-Verhalten, Blocking-Call im Event-Loop, Fehlerbehandlung und Timer-Kopplung.

## Critical Issues

### CR-01: Cache-Contamination bei mehreren konfigurierten Konten

**File:** `Plugins/SpotOn/API/Client.pm:299-301`

**Issue:** Der Response-Cache-Key besteht ausschliesslich aus Pfad und Query-Parametern. Die `_accountId` wird beim Aufbau des `queryStr` explizit herausgefiltert (`next if $key =~ /^_/`). Sind zwei Spotify-Konten konfiguriert (z.B. Benutzer A und Benutzer B), liefert `getSavedTracks` fuer beide denselben Cache-Key `spoton_resp_me/tracks?limit=50&offset=0`. Konto B bekommt die Library von Konto A zurueck, solange der 60-Sekunden-TTL laeuft. Betroffen sind alle cachenden Endpunkte: `me/tracks`, `me/albums`, `me/top/tracks`, `me/following`, `me/playlists`, `artists/*`, `albums/*`, `playlists/*`, `search`, `browse/*`.

**Fix:** `_accountId` in den Cache-Key einbeziehen. Da `_accountId` mit `^_` gefiltert wird, muss er separat hinzugefuegt werden:

```perl
my $cacheKey = $queryStr
    ? "spoton_resp_${accountId}_${path}?${queryStr}"
    : "spoton_resp_${accountId}_${path}";
```

Dabei muss `$accountId` aus `$params->{_accountId}` stammen (bereits im Scope bei Zeile 270 als `my $accountId = $params->{_accountId}`). Der bestehende Token-Invalidierungs-Cache-Key in `_onError` (Zeile 382: `spoton_token_$accountId`) ist davon nicht betroffen.

### CR-02: Command Injection auf Windows via Binary-Name in `taskkill`

**File:** `Plugins/SpotOn/Plugin.pm:143-145`

**Issue:** Der Helper-Pfad stammt aus `$prefs->get('binary')` (benutzerkonfigurierter Wert) oder aus dem Dateisystem. `basename($helper)` liefert nur den Dateinamen, aber der Sanitisierungsschritt entfernt ausschliesslich doppelte Anfuehrungszeichen (`s/"//g`). Shell-Metazeichen wie `&`, `|`, `;` bleiben erhalten. Der resultierende String wird direkt in einen `system()`-Aufruf interpoliert, der auf Windows ueber `cmd.exe` ausgefuehrt wird:

```perl
system(qq{taskkill /IM "$name" /F 1>nul 2>&1});
```

Ein Angreifer mit Schreibzugriff auf die LMS-Preferences (oder ein praeparierter Binary-Pfad) koennte `spoton & del C:\Windows\System32\` als Namen setzen. `cmd.exe` wuerde `&` als Shell-Operator interpretieren.

**Fix:** Den Prozessnamen nur auf alphanumerische Zeichen, Bindestrich, Unterstrich und Punkt beschraenken. Da es sich ausschliesslich um einen Binary-Dateinamen handelt, ist das eine strikte Whitelist:

```perl
my $name = basename($helper);
$name =~ s/[^A-Za-z0-9._-]//g;    # Whitelist statt Blacklist
if ($name) {
    system(qq{taskkill /IM "$name" /F 1>nul 2>&1});
}
```

Alternativ: `Win32::Process::KillProcess` verwenden, das keine Shell-Interpolation durchfuehrt.

## Warnings

### WR-01: `_provisionCredentials` blockiert den LMS-Event-Loop

**File:** `Plugins/SpotOn/API/TokenManager.pm:470`

**Issue:** `_provisionCredentials` wird synchron (Backtick-Aufruf: `` my $output = `$cmd` ``) in `_storeTokens` aufgerufen, das seinerseits aus einem `SimpleAsyncHTTP`-Callback kommt. LMS ist single-threaded. Der Backtick-Aufruf blockiert den Event-Loop fuer die Dauer des Binary-Aufrufs. Der Kommentar sagt "< 2s", aber librespot-core baut eine TCP-Verbindung zum Spotify-AP auf und benoetigt deutlich mehr Zeit bei schlechter Verbindung oder DNS-Problemen (potenziell 30+ Sekunden bis zum Timeout). Das friert das gesamte LMS fuer alle Clients ein.

**Fix:** `Proc::Background` oder `Slim::Utils::Timers` mit `fork`/`open3` verwenden, oder alternativ den Credential-Aufruf in einen separaten Timer-Callback verlagern:

```perl
# Statt synchronem Backtick-Aufruf:
Slim::Utils::Timers::setTimer(undef, Time::HiRes::time() + 0.1, sub {
    $class->_provisionCredentials($accountId, $accessToken);
});
```

Hinweis: Der Timer-basierte Ansatz ist ein einfacher erster Schritt; ein echter async-Ansatz via `open()` im non-blocking Mode waere robuster.

### WR-02: `_madeForYouFeed`-Fallback wird bei anderen HTTP-Fehlern nicht ausgeloest

**File:** `Plugins/SpotOn/Plugin.pm:495`

**Issue:** Die Fallback-Bedingung prueft nur `$err->{code} == 404 || $err->{code} == 403`:

```perl
if (!$data || ($err && ($err->{code} == 404 || $err->{code} == 403))) {
```

Wenn `$data` undef ist aber `$err` undefiniert oder leer (z.B. bei Netzwerkfehler, Timeout — `_onError` liefert `code => 0` bei fehlendem HTTP-Response), wird der Fallback nicht getriggert: `!$data` ist wahr, `$err` ist definiert mit `{ error => ..., code => 0 }`, also ist die Gesamtbedingung `!$data || ($err && (0 == 404 || 0 == 403))` = `undef || (1 && false)` = false. Die Funktion faellt durch ohne Callback-Aufruf, was im LMS-Browser zu einem haengenden "Laden..."-Spinner fuehrt.

**Fix:** Fallback bei jedem Fehler ausfuehren, wenn `$data` fehlt:

```perl
if (!$data) {
    # Fallback: me/playlists + Name-Matching
    ...
    return;
}
```

Der Code danach prueft bereits `$data->{playlists}{items}`, ein undef-Ergebnis waere also sicher. Die 404/403-Unterscheidung kann fuer Logging beibehalten werden.

### WR-03: Kein Guard gegen leere `$spotifyUserId` in `_storeTokens`

**File:** `Plugins/SpotOn/API/TokenManager.pm:388-389`

**Issue:** Wenn die `/me`-Antwort kein `id`-Feld enthaelt (Feb-2026-Einschraenkungen: `User`-Objekte verlieren `email`, `country` etc. — `id` ist noch vorhanden, aber der Wert koennte leer sein), dann ergibt `md5_hex("")` = `d41d8cd9`. Alle betroffenen Konten bekommen dieselbe `accountId`, d.h. das zweite Konto ueberschreibt das erste in `$accounts` und in der Token-Cache.

Der Wert `$userProfile->{id} || ''` setzt dies bereits auf leeren String — das OR-Fallback auf `''` ist kein Schutz, sondern verstaerkt das Problem.

**Fix:** Abbruch wenn `$spotifyUserId` leer ist:

```perl
my $spotifyUserId = $userProfile->{id} // '';
unless ($spotifyUserId) {
    $log->error("TokenManager: Spotify user ID missing — cannot generate accountId");
    $cb->(undef, "Missing Spotify user ID");
    return;
}
my $accountId = substr(md5_hex($spotifyUserId), 0, 8);
```

### WR-04: Scope-Divergenz zwischen PKCE-OAuth-Flow und `--get-token`-Default

**File:** `Plugins/SpotOn/API/TokenManager.pm:28-41` vs. `librespot-spoton/src/main.rs:396-403`

**Issue:** `REQUIRED_SCOPES` im PKCE-Flow (TokenManager.pm) und die Default-Scopes im `--get-token`-Aufruf (main.rs) sind unterschiedlich:

| Scope | PKCE (TokenManager) | get-token (Rust) |
|-------|-------------------|-----------------|
| `streaming` | ja | nein |
| `user-library-modify` | nein | ja |
| `playlist-modify-public/private` | nein | ja |
| `user-follow-modify` | nein | ja |
| `user-read-email` | nein | ja |

Das Keymaster-Token (`--get-token`) hat `playlist-modify-*` und `user-library-modify`, das PKCE-Token hat `streaming`. Wenn der Code einen Via-PKCE erhaltenen Token fuer Library-Modifikationen nutzen will, schlaegt das mit 403 fehl. Das `streaming`-Scope ist fuer den PKCE-Flow irrelevant (Playback laeuft ueber librespot, nicht ueber die Web-Playback-API), erhoehe aber die Oberflaeche der Berechtigungsanfrage.

**Fix:** Scopes vereinheitlichen. Fuer den PKCE-Flow `streaming` entfernen und stattdessen `user-library-modify`, `playlist-modify-public`, `playlist-modify-private`, `user-follow-modify`, `user-read-email` hinzufuegen:

```perl
use constant REQUIRED_SCOPES => join(' ', qw(
    user-read-playback-state
    user-modify-playback-state
    user-read-currently-playing
    user-read-recently-played
    user-read-private
    user-read-email
    playlist-read-collaborative
    playlist-read-private
    playlist-modify-public
    playlist-modify-private
    user-follow-read
    user-follow-modify
    user-library-read
    user-library-modify
    user-top-read
));
```

## Info

### IN-01: `getPersonalMixes` ruft einen seit Feb 2026 entfernten Endpunkt auf

**File:** `Plugins/SpotOn/API/Client.pm:154-161`

**Issue:** `browse/categories/{id}/playlists` ist laut CLAUDE.md seit Feb 2026 im Development Mode entfernt. Die Implementierung ruft diesen Endpunkt primaer auf und faellt erst bei 404/403 auf `me/playlists` zurueck. Im Dev-Mode laeuft also jeder `_madeForYouFeed`-Aufruf garantiert durch einen fehlschlagenden API-Call hindurch, bevor der Fallback greift. Das ist dokumentiert und bewusst akzeptiert, erzeugt aber unnoetige Latenz und Fehler-Logs in jeder Session.

**Fix:** Fuer Development-Mode-Deployments den Fallback-Pfad zur primaeren Route machen oder den primaeren Call beim ersten 404 dauerhaft deaktivieren (mit Cache-Flag). Kein dringender Fix, aber den "happy path" zu kennen wuerde die Latenz von 2 API-Calls auf 1 reduzieren.

### IN-02: Kein Fehlerhandling fuer fehlgeschlagene `File::Path::make_path` in `updateTranscodingTable`

**File:** `Plugins/SpotOn/Plugin.pm:1135-1138`

**Issue:** Der Cache-Verzeichnis-Erstellungsaufruf wird ohne Error-Handling aufgerufen:

```perl
unless (-d $cacheDir) {
    require File::Path;
    File::Path::make_path($cacheDir);
}
```

`make_path` kann scheitern (Berechtigungen, voller Datentraeger). Ein Scheitern ist still — das Plugin versucht danach trotzdem den `--single-track`-Aufruf, der dann mit "No cached credentials" scheitert, ohne dem Benutzer den eigentlichen Grund zu nennen.

**Fix:**

```perl
unless (-d $cacheDir) {
    require File::Path;
    my $err;
    File::Path::make_path($cacheDir, { error => \$err });
    if ($err && @{$err}) {
        $log->error("updateTranscodingTable: cannot create cache dir '$cacheDir': " .
            join(', ', map { values %{$_} } @{$err}));
    }
}
```
