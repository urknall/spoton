# Phase 04.3: ZeroConf + Keymaster Auth - Context

**Gathered:** 2026-05-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Kompletter Auth-Architektur-Umbau: PKCE Browser-OAuth (Spotify Developer App + Client-ID + Browser-Redirect) wird ersetzt durch ZeroConf Discovery (Spotify App → librespot credentials) + Keymaster `--get-token` (Web API Tokens aus librespot credentials). Ein einziger Auth-Schritt über die Spotify App ersetzt den mehrstufigen PKCE-Flow. Kein Spotify Developer App nötig, keine Client-ID, kein Browser-Redirect.

**Hintergrund:** Phase 04.2 hat `--token-login` implementiert (PKCE Access Token → librespot credentials), aber PKCE-Tokens sind inkompatibel mit Spotify AP. ZeroConf ist der bewährte Weg für Credential-Erstellung.

</domain>

<decisions>
## Implementation Decisions

### ZeroConf Discovery UX
- **D-01:** Auto-Start + manueller Button — Plugin erkennt beim Start fehlende credentials.json und startet ZeroConf Discovery automatisch. Zusätzlich Button in Settings zum manuellen Neustart (Account-Wechsel, erneutes Pairing).
- **D-02:** Bildliche Anleitung — Settings-Seite zeigt Bild/Screenshot eines Handys mit markiertem Spotify Connect Icon als möglichst bildliche Anleitung für den User.
- **D-03:** Device-Name = LMS-Server-Name — ZeroConf Discovery nutzt dynamisch den Servernamen aus LMS Prefs, damit der User sein Gerät in der Spotify App wiedererkennt.
- **D-04:** Auto-Refresh per JavaScript-Timer — Settings-Seite pollt per AJAX-Timer (~3s) einen Endpoint der credentials.json-Status prüft. Status wechselt automatisch von "Warte auf Verbindung" zu "Verbunden" ohne manuelles Seite-Neuladen.
- **D-05:** Account-Liste von Anfang an — Nach erfolgreicher ZeroConf-Verbindung zeigt Settings: "Verbunden als [Spotify Display Name]" + Liste aller verbundenen Accounts mit Switch/Remove-Buttons. Multi-Account-UI wird sofort mitgebaut.

### PKCE-Code-Verbleib
- **D-06:** Komplett löschen nach UAT-Bestätigung — PKCE-Code (startOAuthFlow, exchangeCode, PKCE-Methoden, Client-ID-Feld, OAuth-Button, Callback-Route) wird entfernt, aber erst nachdem UAT bestätigt hat, dass alle Items (Library, Playlists, Search, Browse) über Keymaster-Tokens listbar und erreichbar sind.
- **D-07:** Kein Migrations-Thema — SpotOn ist noch in Entwicklung ohne externe User. Bestehende PKCE-Tokens in Test-Prefs werden manuell aufgeräumt.

### Claude's Discretion
- TokenManager-Umbau: Strategie für --get-token Aufruf (Binary-Spawn + Cache vs. Persistent Daemon), Caching-TTLs, Refresh-Timer
- Error-Handling bei --get-token Fehler (Retry-Strategie, Auto-Re-Auth vs. manuelle Neuverbindung)
- getToken()-Interface zu Client.pm: Beibehalten oder anpassen — Research prüft ob das bestehende Interface ausreicht
- Multi-Account Credential-Speicherung: Account-spezifische Cache-Dirs vs. Credentials-Umbenennung — Research prüft librespot Cache-Architektur
- Multi-Account: Flow zum Hinzufügen weiterer Accounts (gleicher ZeroConf-Flow vs. sequenziell erzwungen)
- Account-Identifikation nach ZeroConf: --get-token + /me vs. credentials.json parsen — Research prüft Datenformate
- Zeitlicher Ablauf der PKCE-Löschung (Zwei-Schritt in Phase 04.3 vs. eigene Mini-Phase)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `main.rs::run_get_token()` — Bereits funktionsfähig: liest credentials.json, verbindet mit Spotify, holt Web API Token via `session.token_provider().get_token()`. Alle benötigten Scopes als Default hinterlegt.
- `main.rs::Mode::GetToken` — CLI-Parsing für `--get-token`, `--cache`, `--scope` bereits implementiert.
- `Helper.pm::get()` — Binary-Discovery, liefert Pfad und Version. Direkt nutzbar für --get-token Spawn.
- `TokenManager.pm::_cacheToken()` — Token-Caching mit TTL bleibt relevant, nur die Token-Quelle ändert sich.
- `TokenManager.pm::getAccountIds()`, `getActiveAccountName()`, `removeAccount()` — Account-Management-Methoden bleiben weitgehend nutzbar.
- `Settings.pm` — Settings-Handler-Infrastruktur (handler(), prefs()), AJAX-Patterns, Account-Switch/Remove-Logik.

### Established Patterns
- Binary-Spawn-Pattern: `_provisionCredentials()` in TokenManager.pm zeigt wie Binary aufgerufen wird (Backtick mit Shell-Escaping, Exit-Code-Prüfung)
- Cache-Dir-Pattern: `catdir($serverPrefs->get('cachedir'), 'spoton')` — zentrale Cache-Dir-Konstruktion
- Account-ID-Pattern: `substr(md5_hex($spotifyUserId), 0, 8)` — 8-Zeichen Hex-Hash der Spotify User-ID
- Timer-Pattern: `Slim::Utils::Timers::setTimer()` für asynchrone Operationen
- Prefs-Pattern: `$prefs->get('accounts')` Hash mit Account-Daten, `$prefs->get('activeAccount')` für aktiven Account

### Integration Points
- `TokenManager.pm::getToken($accountId, $cb)` → Client.pm — Zentrale Token-Schnittstelle die umgebaut wird
- `TokenManager.pm::refreshAllTokens()` → Timer-Callback — Wird von PKCE-HTTP-Refresh auf Keymaster umgestellt
- `Plugin.pm::initPlugin()` → Discovery Auto-Start wenn keine Credentials
- `Settings.pm::handler()` → ZeroConf Discovery Start/Stop, AJAX-Status-Endpoint
- `custom-convert.conf` → `$CACHE$` muss auf Account-spezifisches Dir zeigen
- `Plugin.pm::updateTranscodingTable()` → Cache-Dir-Referenz muss mit Multi-Account kompatibel sein

</code_context>
