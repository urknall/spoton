# Dual-Token-Architektur: Research-Ergebnisse aus Phase 04.3 UAT

## Kernvorgabe

**Keine zweite Authentifizierung.** ZeroConf Discovery (einmal Spotify App → Gerät auswählen)
bleibt der einzige Autorisierungsschritt. Die gespeicherten Credentials (`credentials.json`)
reichen für beide Token-Flavors — Keymaster kann mit beliebiger Client-ID Tokens erzeugen:

```
spoton --get-token --cache /path --client-id EIGENE_ID    → "own" Token
spoton --get-token --cache /path                           → "bundled" Token
```

## Problem: Rate-Limit auf Keymaster-Default-Client-ID

Bei der UAT von Phase 04.3 entdeckt: Spotify rate-limitiert die Web API aggressiv
für die interne librespot-Client-ID (`65b708073fc0480ea92a077233ca87bd`).

### Empirische Befunde (2026-05-29)

| Token-Quelle | `GET /me` | `GET /me/tracks` | `browse/categories` | `search` |
|---|---|---|---|---|
| Keymaster + librespot-Default-ID | 429 | 429 | 429 (nicht 404!) | 429 |
| PKCE + eigene Client-ID | 200 | 200 | 404 (Dev-Mode) | 200 |
| **Keymaster + eigene Client-ID** | **200** | **200** | **404** (Dev-Mode) | **200** |

### Schlüsselerkenntnis

- `browse/categories` gibt **429** mit der librespot-Default-ID → Endpoint existiert noch für interne IDs
- `browse/categories` gibt **404** mit eigener Dev-App-ID → Dev-Mode-Block
- Keymaster `get_token_with_client_id(scopes, custom_id)` existiert in librespot-core 0.8.0 (token.rs:70)
- Binary hat jetzt `--client-id` Flag → durchgereicht an `get_token_with_client_id`

## Spotty-NG Referenz-Architektur

Spotty-NG löst das mit einem **Dual-Flavor-Token-System** (API.pm, Phase 2/2.5/2.6):

### Routing-Logik

```
Request → URL-Pattern matchen
  ├─ me/* Familie        → IMMER "own" Token (eigene Client-ID)
  ├─ search, albums/*    → "own" Token
  ├─ browse/categories/* → "bundled" Token (interne ID)
  ├─ browse/featured-*   → "bundled" Token
  ├─ playlists/37i9*     → "bundled" Token (kuratierte Spotify-Playlists)
  └─ Unbekannter Pfad    → "own" erst, bei 403/410 → "bundled" Fallback
```

### Deprecated-Families (aus Spotty-NG API.pm:66-91)

```perl
@KNOWN_DEPRECATED_FAMILIES = (
    qr{^browse/featured-playlists\b},
    qr{^browse/categories/[^/?]+/playlists\b},
    qr{^browse/categories\b},
    qr{^browse/new-releases\b},
    qr{^recommendations\b},
    qr{^users/[^/?]+/playlists\b},
    qr{^artists/[^/?]+/top-tracks\b},
    qr{^artists/[^/?]+/related-artists\b},
    qr{^playlists/37i9[A-Za-z0-9]+\b},  # Kuratierte Playlists (37i9-Prefix)
);
```

### Smart Caching

- Erster Call: "own" versuchen → 403/410 → "bundled" retry → Erfolg → **Hint cachen (24h TTL)**
- Folge-Calls: Hint-Cache hit → direkt "bundled" (kein 2x-Kosten)
- `me/*` Guard: User-Endpunkte NIEMALS auf bundled fallen lassen (Pitfall #1)

### Warum Spotty-NG kein 429 bekommt

1. Rate-Limit-Druck auf **zwei** Client-IDs verteilt
2. `browse/categories` nur über bundled (wenige Calls)
3. Korrekte `Retry-After`-Behandlung von Anfang an (AsyncRequest.pm:104-112)
4. `$maybeResponse` (dritter Error-Callback-Parameter) wird korrekt erfasst

## Was SpotOn für die Implementierung braucht

### Bereits vorhanden (aus Phase 04.3)

- [x] `--client-id` Flag im Binary → `get_token_with_client_id()`
- [x] Keymaster-Credentials via ZeroConf Discovery (einmalige Autorisierung)
- [x] TokenManager._fetchKeymasterToken mit `--client-id` aus Prefs
- [x] Client.pm `_onError` mit korrektem `$response` und `Retry-After`-Parsing
- [x] Selbe credentials.json → zwei Token-Flavors ohne zweiten Auth-Schritt

### Noch zu bauen

- [ ] **Dual-Token-Cache:** Zwei Token-Slots pro Account (own + bundled)
- [ ] **Request-Router:** URL-Pattern → Token-Flavor Mapping in Client.pm
- [ ] **Fallback-Logik:** 403/410 auf "own" → retry mit "bundled" → Hint cachen
- [ ] **`me/*` Guard:** Harter Block gegen bundled für User-Endpunkte
- [ ] **Bundled-Token-Refresh:** Separater Refresh-Zyklus für bundled-Token
- [ ] **Client-ID-Quelle klären:** Bundled mitliefern? Settings-UI? Beides?

### Architektur-Entscheidung

**Option A: Zwei `_fetchKeymasterToken`-Aufrufe mit verschiedenen `--client-id`**
- Pro: Einfach, nutzt bestehende Infrastruktur
- Contra: Zwei Spotify-AP-Verbindungen pro Refresh-Zyklus

**Option B: Ein Token pro Flavor, Lazy-Fetch bei Bedarf**
- Pro: Nur bundled-Token holen wenn browse/categories gebraucht wird
- Contra: Erster Browse-Call hat Latenz

**Empfehlung:** Option B (Spotty-NG-Muster) — bundled-Token nur on-demand.
