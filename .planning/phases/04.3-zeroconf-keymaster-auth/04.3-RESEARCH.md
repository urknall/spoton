## Summary

Phase 04.3 ersetzt die PKCE Browser-OAuth-Architektur vollständig durch ZeroConf Discovery + Keymaster Token-Provider. Die technische Basis ist vollständig vorhanden und verifiziert: `librespot-discovery 0.8.0` existiert auf crates.io [VERIFIED: cargo search], `session.connect(credentials, true)` schreibt `credentials.json` automatisch [VERIFIED: librespot-core-0.8.0/src/cache.rs], und `run_get_token()` in `main.rs` funktioniert bereits vollständig für Web API Token-Abruf [VERIFIED: Codebase].

**Hauptarchitektur:** Das binary erhält einen neuen `--discover-once` Mode: startet `librespot-discovery`, wartet auf die erste Verbindung via Spotify App, schreibt `credentials.json` via `session.connect(creds, true)`, gibt `credentials_saved\n{username}` auf stdout aus und beendet sich. Der Perl-Layer spawnt dieses Binary im Hintergrund via `Proc::Background`, und ein AJAX-Timer pollt alle 3s einen `/plugins/SpotOn/settings/discoveryStatus` Endpoint, der auf `credentials.json` prüft.

**Token-Quelle:** Keymaster (`session.token_provider().get_token(scopes)`) nutzt das Mercury-Protokoll (intern `hm://keymaster/token/authenticated?...`) — kein HTTPS, kein Developer App Client-ID erforderlich, Token TTL ~3600s. Scopes sind komma-separiert und alle bereits als Default in `run_get_token()` hinterlegt.

**Primary recommendation:** Binary erhält `--discover-once` Mode mit `librespot-discovery = "0.8.0"`; Perl spawnt als Hintergrundprozess; AJAX-polling zeigt Status live; `credentials.json` wird gelesen für username; `--get-token + /me` liefert display_name.

### Pattern 1: librespot-discovery Binary-Mode (--discover-once)

**Was:** Rust async function in `main.rs` die Discovery-Crate nutzt um einmalig Credentials via Spotify App zu empfangen und dann `credentials.json` zu schreiben.

**Wann:** Beim ersten Plugin-Start (keine credentials.json) oder manueller Re-Auth über Button in Settings.

**Beispiel (main.rs):**
```rust
// Source: librespot-discovery 0.8.0 docs.rs + librespot-core 0.8.0/src/session.rs

use librespot_discovery::{Discovery, DeviceType};
use futures_util::StreamExt;

async fn run_discover_once(
    device_name: &str,
    device_id: &str,
    cache_dir: &str,
) -> Result<(), Box<dyn std::error::Error>> {
    // KEYMASTER_CLIENT_ID ist der standard librespot Client-ID (aus librespot-core config.rs)
    const KEYMASTER_CLIENT_ID: &str = "65b708073fc0480ea92a077233ca87bd";

    let mut discovery = Discovery::builder(device_id, KEYMASTER_CLIENT_ID)
        .name(device_name)
        .device_type(DeviceType::Speaker)
        .launch()?;

    // Warte auf die erste Verbindung (blockiert bis Spotify App verbindet)
    let credentials = discovery.next().await
        .ok_or("Discovery stream ended without credentials")?;

    // Session mit store_credentials=true → schreibt credentials.json
    let cache = Cache::new(Some(cache_dir), None::<&str>, None::<&str>, None)?;
    let session_config = SessionConfig::default();
    let session = Session::new(session_config, Some(cache));
    session.connect(credentials, true).await?;

    // Username aus Session auslesen
    let username = session.username();
    println!("credentials_saved");
    println!("{}", username);

    Ok(())
}
```

**Contract für Perl:**
```
Command: spoton -n 'LMS-Server-Name' -c '{cachedir}/spoton/__DISCOVER__' --discover-once
stdout:  "credentials_saved\n{spotify_username}"
exit 0:  Erfolg — credentials.json wurde in cache_dir geschrieben
exit 1:  Fehler (Timeout, Netzwerk, etc.)
```

**Wichtig:** Das Binary läuft im Hintergrund (Proc::Background), NICHT als synchroner Backtick-Call. Der Timeout liegt bei 15 Minuten (wie in Spotty).

### Pattern 2: Keymaster --get-token in TokenManager.pm

**Was:** Synchroner Binary-Spawn (backtick) um einen Web API Access Token via Keymaster zu holen. Wird durch `_cacheToken()` gecacht (TTL = expiresIn - 300s).

**Wann:** Bei jedem `getToken($accountId, $cb)` Cache-Miss.

**Beispiel (Perl):**
```perl
# Source: Verifiziert gegen librespot-core-0.8.0/src/token.rs

sub _fetchKeymasterToken {
    my ($class, $accountId, $cb) = @_;

    require Plugins::SpotOn::Helper;
    my ($helper) = Plugins::SpotOn::Helper->get();
    unless ($helper) {
        $log->error("TokenManager: binary nicht gefunden für Account $accountId");
        $cb->(undef);
        return;
    }

    my $serverPrefs = preferences('server');
    my $cacheDir    = catdir($serverPrefs->get('cachedir'), 'spoton', $accountId);

    (my $safeHelper = $helper)   =~ s/'/'\\''/g;
    (my $safeDir    = $cacheDir) =~ s/'/'\\''/g;

    my $cmd = sprintf("'%s' --get-token --cache '%s' 2>/dev/null", $safeHelper, $safeDir);
    main::INFOLOG && $log->info("TokenManager: --get-token für Account $accountId");

    # Deferred via Timer — kein direkter Backtick im Event-Loop (WR-01 Pattern aus Phase 04.2)
    Slim::Utils::Timers::setTimer(undef, Time::HiRes::time() + 0.1, sub {
        my $output = `$cmd`;
        my $exit   = $? >> 8;

        if ($exit != 0 || !$output) {
            $log->error("TokenManager: --get-token fehlgeschlagen für $accountId (exit $exit)");
            $cb->(undef);
            return;
        }

        my $result = eval { from_json($output) };
        if ($@ || !$result->{accessToken}) {
            $log->error("TokenManager: JSON-Parse-Fehler bei --get-token: $@");
            $cb->(undef);
            return;
        }

        $class->_cacheToken($accountId, $result->{accessToken}, $result->{expiresIn});
        $cb->($result->{accessToken});
    });
}
```

**--get-token stdout Format (bereits implementiert in main.rs):**
```json
{"accessToken":"BQA...","expiresIn":3600}
```

**Token-Lifetime:** ~3600 Sekunden [VERIFIED: librespot-core-0.8.0/src/token.rs — TokenData.expires_in ist u64]. Effektive Cache-TTL = expiresIn - TOKEN_EXPIRY_BUFFER (300s) = ~55 Minuten.

### Pattern 3: AJAX Discovery Status Endpoint

**Was:** Raw-Function-Endpoint in Settings.pm (via `addRawFunction`) der den aktuellen Discovery-Status als JSON zurückgibt.

**Wann:** Alle 3s vom JavaScript-Timer auf der Settings-Seite gepollt (D-04).

**Beispiel (Perl — Settings.pm):**
```perl
# Source: Pattern aus Spotty Settings/Auth.pm::checkCredentials

sub _registerDiscoveryStatusEndpoint {
    Slim::Web::Pages->addRawFunction(
        'plugins/SpotOn/settings/discoveryStatus',
        \&_discoveryStatusHandler
    );
}

sub _discoveryStatusHandler {
    my ($httpClient, $response) = @_;

    my $serverPrefs = preferences('server');
    my $discoverDir = catdir($serverPrefs->get('cachedir'), 'spoton', '__DISCOVER__');
    my $credsFile   = catfile($discoverDir, 'credentials.json');

    my $result;
    if (-f $credsFile) {
        $result = { status => 'connected' };
    } elsif ($discoveryProc && $discoveryProc->alive()) {
        $result = { status => 'waiting' };
    } else {
        $result = { status => 'idle' };
    }

    my $content = to_json($result);
    $response->header('Content-Length' => length($content));
    $response->code(200);
    $response->header('Connection' => 'close');
    $response->content_type('application/json');
    Slim::Web::HTTP::addHTTPResponse($httpClient, $response, \$content);
}
```

**JavaScript-Timer (basic.html):**

### Pattern 4: credentials.json lesen und Account anlegen

**Was:** Nach erfolgreicher Discovery — Perl liest `credentials.json`, extrahiert `username`, baut accountId, verschiebt `__DISCOVER__` → `{accountId}`, ruft `--get-token + /me` für display_name auf.

```perl
# credentials.json Format (VERIFIED gegen echte Datei):
# {"username":"3fiqdghdkdx37cnnuv1vemc1n","auth_type":1,"auth_data":"base64..."}
# username = Spotify canonical user ID (nicht display_name)

sub _setupAccountFromCredentials {
    my ($class, $cb) = @_;

    my $serverPrefs = preferences('server');
    my $baseDir     = catdir($serverPrefs->get('cachedir'), 'spoton');
    my $discoverDir = catdir($baseDir, '__DISCOVER__');
    my $credsFile   = catfile($discoverDir, 'credentials.json');

    open(my $fh, '<', $credsFile) or do {
        $log->error("TokenManager: credentials.json nicht lesbar: $!");
        $cb->(undef);
        return;
    };
    local $/;
    my $json = <$fh>;
    close $fh;

    my $creds = eval { from_json($json) };
    if ($@ || !$creds->{username}) {
        $log->error("TokenManager: credentials.json lesen fehlgeschlagen: $@");
        $cb->(undef);
        return;
    }

    my $spotifyUserId = $creds->{username};
    my $accountId     = substr(md5_hex($spotifyUserId), 0, 8);

    # Dir-Rename: __DISCOVER__ → {accountId}
    my $finalDir = catdir($baseDir, $accountId);
    if (-d $finalDir) {
        # Account bereits vorhanden — überschreiben
        require File::Path;
        File::Path::remove_tree($finalDir);
    }
    require File::Copy;
    File::Copy::move($discoverDir, $finalDir) or do {
        $log->error("TokenManager: Dir-Rename fehlgeschlagen: $!");
        $cb->(undef);
        return;
    };

    # --get-token + /me für display_name
    $class->_fetchDisplayName($accountId, $spotifyUserId, $cb);
}
```

### Pattern 5: LMS Server-Name für Discovery Device-Name

```perl
# Source: LMS Slim::Utils::Misc::getLibraryName() Zeile 606:
# my $hostname = $prefs->get('libraryname') || '';
# preferences('server')->get('libraryname') liefert den Servernamen
# Fallback: Hostname (via Slim::Utils::Network::hostname() oder POSIX::uname)

sub _getLmsServerName {
    my $serverPrefs = preferences('server');
    my $name = $serverPrefs->get('libraryname') || '';
    unless ($name) {
        require Sys::Hostname;
        $name = eval { Sys::Hostname::hostname() } || 'Lyrion Music Server';
    }
    # Max 60 Zeichen (Spotify-Limit analog Spotty Daemon.pm Zeile 70)
    return substr($name, 0, 60);
}
```

### Anti-Patterns to Avoid

- **Backtick für Discovery-Prozess:** Discovery kann Minuten dauern (bis User verbindet). NIEMALS als synchronen Backtick-Call. Immer `Proc::Background` + AJAX-Poll.
- **credentials.json im Flat-Root:**  `spoton/credentials.json` (single-account) reicht nicht für Multi-Account. Immer `spoton/{accountId}/credentials.json` (Spotty-Pattern).
- **PKCE-Methoden behalten während ZeroConf-Umbau:** Verwirrt Tests und lässt alten Code aktiv. Stufenweise: erst ZeroConf implementieren + testen, dann PKCE entfernen (D-06).
- **--get-token ohne Deferred-Timer:** Keymaster macht eine Mercury-Verbindung zu Spotify AP (~100-500ms). Backtick direkt im Event-Loop-Callback würde LMS für diese Zeit blockieren. Immer über `Slim::Utils::Timers::setTimer(undef, 0.1, sub {...})` deferren.
- **username aus credentials.json als display_name verwenden:** `username` ist die interne Spotify User-ID (z.B. `3fiqdghdkdx37cnnuv1vemc1n`). Display Name muss via `/me` API geholt werden.
- **Device-ID zufällig bei jedem Start generieren:** librespot-discovery braucht eine stabile `device_id` damit die Spotify App das Gerät wiedererkennt. Ableitung: MD5 des LMS-Servernamens oder `$serverPrefs->get('libraryname') || hostname()`.

## Common Pitfalls

### Pitfall 1: librespot-discovery Builder benötigt stable device_id

**Was schiefgeht:** Bei jedem Plugin-Start eine neue random UUID als device_id → Gerät erscheint als neues Gerät in der Spotify App bei jedem Neustart.

**Warum:** librespot-discovery sendet die device_id via ZeroConf; Spotify App listet es als neues Gerät wenn sich die ID ändert.

**Wie vermeiden:** Stabile device_id ableiten — z.B. `md5_hex($serverPrefs->get('libraryname') . $serverPrefs->get('httpport'))` oder als gespeicherte Pref beim ersten Start generieren.

**Warnsignal:** Gerät erscheint bei jedem LMS-Neustart doppelt in der Spotify App.

---

### Pitfall 2: __DISCOVER__ Verzeichnis bleibt nach Abbruch zurück

**Was schiefgeht:** User bricht Discovery ab (schließt Browser-Tab). `__DISCOVER__/` existiert aber ohne `credentials.json`. Beim nächsten Start wird das Dir fälschlicherweise als "pending" erkannt.

**Warum:** Proc::Background killt den Prozess, aber das Verzeichnis bleibt.

**Wie vermeiden:** Beim Start von `startDiscovery()` immer `__DISCOVER__` bereinigen/neu anlegen. Beim Discovery-Stop immer `__DISCOVER__` löschen falls leer.

---

### Pitfall 3: Token TTL vs. Keymaster Token-Lifetime

**Was schiefgeht:** `expiresIn` aus `--get-token` = ~3600s. Bei `expiresIn - 300` = 3300s Cache-TTL. Bei vielen simultanen API-Calls werden alle gleichzeitig abgelaufen → Mass-Refresh alle ~55 Minuten.

**Warum:** Alle Accounts werden gleichzeitig durch `refreshAllTokens()` aktualisiert.

**Wie vermeiden:** Refresh-Jitter einbauen (schon vorhandenes Timer-Pattern verwenden, aber mit leichtem Versatz per Account).

**Warnsignal:** Logs zeigen alle Accounts gleichzeitig Token-Refresh alle 55 Minuten.

---

### Pitfall 4: --get-token schlägt fehl wenn credentials.json beschädigt/leer

**Was schiefgeht:** `--get-token` gibt exit 1 zurück. `getToken()` liefert undef. Client.pm gibt `no_token` Fehler zurück. Alle API-Calls schlagen still fehl.

**Warum:** credentials.json kann durch Systemabsturz oder Disk-Full beschädigt werden.

**Wie vermeiden:** Bei exit != 0 in `_fetchKeymasterToken()`: Account als "ungültig" markieren, ZeroConf Re-Auth anbieten (analog zu 401-Handling in Client.pm). User informieren über Settings-Statusanzeige.

---

### Pitfall 5: Spotify Display Name von User mit Zahlen-Username

**Was schiefgeht:** username in credentials.json = `3fiqdghdkdx37cnnuv1vemc1n` (interne ID). Ohne `/me`-Aufruf ist der Display-Name unbekannt.

**Warum:** librespot-core speichert nur die technische Spotify User-ID.

**Wie vermeiden:** Nach Discovery immer `--get-token + /me` aufrufen um `display_name` zu holen. Fallback: username direkt anzeigen wenn `/me` fehlschlägt.

---

### Pitfall 6: Multi-Account und updateTranscodingTable

**Was schiefgeht:** `updateTranscodingTable()` injiziert einen festen `$cacheDir` = `spoton/` (ohne Account-Subdir). Mit Multi-Account braucht `--single-track` aber `spoton/{accountId}/credentials.json`.

**Warum:** Die aktuelle Implementierung nutzt den globalen Cache-Dir-Pfad ohne Account-Spezifizierung.

**Wie vermeiden:** `updateTranscodingTable()` muss den aktiven Account-ID einbeziehen und `spoton/{activeAccountId}` injizieren. Wird per-track aufgerufen (LMS-11), also immer aktuell.

**Warnsignal:** `--single-track` gibt "No cached credentials found" obwohl ZeroConf erfolgreich war.

---

### Pitfall 7: Discovery läuft endlos ohne Timeout

**Was schiefgeht:** User startet Discovery, macht nichts. Binary läuft ewig, hält Socket offen.

**Wie vermeiden:** 15-Minuten-Timeout im Binary (`--discover-once` mit `tokio::time::timeout(Duration::from_secs(900), discovery.next())`). Perl-Seite: `HELPER_TIMEOUT => 60*15` Timer (Spotty-Pattern) der `Proc::Background->die()` aufruft.

### Spotty ZeroConf Auth Flow als Referenz

Spotty implementiert ZeroConf-Auth durch:
1. Connect-Daemon wird in `__AUTHENTICATE__/` Ordner gestartet (Settings/Auth.pm::_cacheFolder)
2. Discovery ist ENABLED (kein `--disable-discovery`)
3. `checkCredentials` Endpoint (`addRawFunction`) prüft `hasCredentials(AUTHENTICATE)`
4. JavaScript-Timer pollt alle 2s (Spotty) / 3s (SpotOn, D-04)
5. Nach Verbindung: `cleanup()` renamed `__AUTHENTICATE__` → MD5(username) [AccountHelper.pm::renameCacheFolder]

SpotOn folgt dem gleichen Pattern, aber mit `__DISCOVER__` statt `__AUTHENTICATE__` und mit `--discover-once` Mode statt dem vollen Connect-Daemon.

## Open Questions (RESOLVED)

1. **session.username() Verfügbarkeit nach session.connect()** (RESOLVED)
   - Was wir wissen: `session.set_username(username)` wird in `session.rs:247` aufgerufen nach connect
   - Was unklar: Gibt es eine public `session.username()` Methode in librespot-core 0.8.0?
   - Empfehlung: Im Binary nach `session.connect()` die `credentials.json` lesen (sie wurde eben geschrieben) und `username` daraus extrahieren — das ist sicherer als Session-API
   - **Resolution:** Plan 01 Task 1 implementiert credentials.json-Lesen nach session.connect() statt session.username(). Sicherer Fallback gewaehlt.

2. **Discovery mit --with-libmdns Feature Flag** (RESOLVED)
   - Was wir wissen: librespot-discovery hat feature flags: `with-libmdns` (default), `with-avahi`, `with-dns-sd`
   - Was unklar: Sind rustls-tls-native-roots Features kompatibel mit librespot-discovery 0.8.0?
   - Empfehlung: Starte mit `default-features = false, features = ["with-libmdns", "rustls-tls-native-roots"]` — wenn Kompilierung fehlschlägt, auf `default-features = true` zurückfallen
   - **Resolution:** Plan 01 Task 1 verwendet `default-features = false, features = ["with-libmdns", "rustls-tls-native-roots"]`. Kompilierung wird in cargo build --release verifiziert.

3. **Discovery-Timeout in Binary vs. Perl-Seite** (RESOLVED)
   - Was wir wissen: Spotty verwendet 15-Minuten-Timeout
   - Was unklar: Ob `tokio::time::timeout()` im Binary oder Proc::Background->die() in Perl das Timeout steuern soll
   - Empfehlung: Beide — Binary hat internen Timeout (15min via tokio), Perl hat externen Watchdog-Timer als Backup
   - **Resolution:** Dual-Timeout implementiert: Binary hat tokio::time::timeout(900s) (Plan 01), Perl hat DISCOVERY_TIMEOUT Watchdog-Timer (Plan 02). Belt-and-suspenders Ansatz.
