<objective>
Implement dual-flavor token routing in TokenManager.pm and Client.pm: own-token (eigene Client-ID via --client-id) for me/* and search endpoints, bundled-token (librespot-Default-ID without --client-id) for deprecated browse endpoints and curated playlists. Includes 403/410 fallback with hint cache, per-flavor rate limiting, and me/* hard guard.

Purpose: Eliminates 429 rate-limit errors by distributing API pressure across two Spotify Client-IDs (Spotty-NG reference architecture). Ensures browse/categories works via bundled token while me/* endpoints use the own token.
Output: Modified TokenManager.pm with flavor-aware getToken/cacheToken/removeAccount; Modified Client.pm with dual-token routing pipeline, fallback logic, hint cache, per-flavor rate-limit keys.
</objective>

    Rewrite Client.pm _request() pipeline to implement dual-flavor token routing per D-01 through D-06.

    **New imports:** Add `use Digest::MD5 qw(md5_hex);` for hint cache key generation.

    **New constants** (after existing constants block, line ~23):
    - `BUNDLED_HINT_TTL => 86400` (24 hours)
    - `BUNDLED_HINT_KEY_PREFIX => 'spoton_bundled_hint_'`
    - `SPOTON_DEFAULT_CLIENT_ID => ''` (placeholder, same as TokenManager)

    **New package-level compiled regexes** (after constants, before methods):
    - `$_meFamilyRegex = qr{^me(?:$|/|\?)}` — me/* family guard per D-05
    - `@KNOWN_DEPRECATED_FAMILIES` — array of qr{} patterns per D-04, sourced from PATTERNS.md: browse/featured-playlists, browse/categories/*/playlists, browse/categories, browse/new-releases, recommendations, users/*/playlists, artists/*/top-tracks, artists/*/related-artists, playlists/37i9*

    **New private helper subs:**
    - `_resolveStartFlavor($cleanPath, $isMeFamily)`: Returns 'own' if me/* (D-05). Checks hint cache via `_lookupBundledHint` (D-06). Checks degraded mode via `$prefs->get('clientId') || SPOTON_DEFAULT_CLIENT_ID` (D-03). Returns 'own' or 'bundled'.
    - `_lookupBundledHint($cleanPath)`: Iterates @KNOWN_DEPRECATED_FAMILIES, returns 'bundled' if `$cache->get(BUNDLED_HINT_KEY_PREFIX . md5_hex("$rx"))` hits.
    - `_rememberBundledHint($cleanPath)`: Iterates @KNOWN_DEPRECATED_FAMILIES, on match sets `$cache->set($key, 1, BUNDLED_HINT_TTL)`. Logs hint creation.
    - `_is404Deprecated($cleanPath)`: Returns 1 if path matches any @KNOWN_DEPRECATED_FAMILIES regex. Used to distinguish real 404 from deprecated-endpoint 404 (Pitfall 4).

    **Rewritten _request():** The pipeline changes are:
    1. Strip leading slash: `$cleanPath =~ s{^/}{}`.
    2. Compute `$isMeFamily = ($cleanPath =~ $_meFamilyRegex)`.
    3. Per-flavor rate-limit check (D-01): if me/*, check `spoton_rate_limit_own`; if not me/*, check `spoton_rate_limit_bundled`. Also check legacy `RATE_LIMIT_CACHE_KEY` for both (backward compat).
    4. Response cache check (unchanged, API-03).
    5. Concurrency cap (unchanged, shared across both flavors per Pitfall 6 — single `$inflightCount`).
    6. Increment `$inflightCount`.
    7. Double-callback guard: wrap `$cb` in `$userCb` closure with `$userCbCalled` flag; `$userCb` decrements `$inflightCount` exactly once.
    8. Resolve `$startFlavor` via `_resolveStartFlavor($cleanPath, $isMeFamily)`.
    9. Call `_doFlavouredRequest($method, $cleanPath, $params, $userCb, $startFlavor, 0, $isMeFamily)`.

    **New _doFlavouredRequest():** Extracted inner method for single-flavor HTTP execution with fallback capability. Parameters: `($class, $method, $cleanPath, $params, $userCb, $flavor, $isRetry, $isMeFamily)`.
    1. Calls `TokenManager->getToken($accountId, $flavor, sub { ... })`.
    2. On no token: `$userCb->(undef, { error => 'no_token', flavor => $flavor })`.
    3. Sets `$params->{_flavor} = $flavor` for `_onError`.
    4. Builds URL from API_BASE + $cleanPath + query params (same URL-building logic currently in _request lines 285-294).
    5. Computes and checks cache key (same cache logic currently in _request lines 300-311). Cache key does NOT include flavor — cached responses are valid regardless of which token fetched them (per API-03 and RESEARCH).
    6. Fires `SimpleAsyncHTTP` with intercept callback.
    7. Intercept callback on error: if `!$isRetry && $flavor eq 'own' && !$isMeFamily && ($code == 403 || $code == 410 || ($code == 404 && _is404Deprecated($cleanPath)))`: recursively call `_doFlavouredRequest` with `flavor='bundled', isRetry=1`. Return without calling `$userCb`.
    8. Intercept callback on success after retry: if `!$err && $isRetry && $flavor eq 'bundled'`, call `_rememberBundledHint($cleanPath)`.
    9. Otherwise: call `$userCb->($result, $err)`.

    **Modified _onSuccess():** Remove `$inflightCount--` (now handled by `$userCb` guard). JSON parse and cache logic unchanged.

    **Modified _onError():** Remove `$inflightCount--` (now handled by `$userCb` guard). Add per-flavor rate-limit key logic: read `$flavor = $params->{_flavor} // 'own'`. On 429, set both `$rlKey = spoton_rate_limit_$flavor` and legacy `RATE_LIMIT_CACHE_KEY`. On 401, invalidate token cache with flavor suffix: `$cache->remove("spoton_token_${accountId}_${flavor}")`.

    **Important architectural constraints:**
    - Concurrency cap (`MAX_CONCURRENT_REQUESTS => 3`) remains shared across both flavors (Pitfall 6).
    - `$inflightCount` is managed exclusively by the `$userCb` guard in `_request()` — NOT in `_onSuccess`/`_onError` (prevents double-decrement on fallback retry).
    - Response cache keys do NOT include flavor (browse/categories response is the same regardless of which token fetched it).
    - me/* paths NEVER trigger bundled fallback, even on 403/410 (D-05 hard guard).
    - 404 triggers bundled retry ONLY if `_is404Deprecated` returns true (Pitfall 4).
    - Hint cache only written on successful (2xx) bundled retry, never speculatively (Anti-Pattern from RESEARCH).
    - Log lines include `[flavor=$flavor]` annotation but NEVER log token values (T-02-10, T-04.3-06).
