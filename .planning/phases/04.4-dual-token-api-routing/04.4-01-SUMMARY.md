# Phase 04.4 Plan 01: Dual-Token API Routing Summary

**One-liner:** Dual-flavor token routing (own Client-ID for me/*, bundled librespot-Default-ID for browse/* and curated playlists) with 403/410 fallback, 24h hint cache, per-flavor rate-limit keys, and me/* hard guard.

## What Was Built

### TokenManager.pm — Flavor-aware token lifecycle

- `getToken($accountId, $flavor, $cb)` — backward-compatible via `ref $flavorOrCb eq 'CODE'` dispatch; flavor='own' as default for old callers
- `_fetchKeymasterToken($accountId, $flavor, $cb)` — dispatches `--client-id` only for flavor='own' with truthy own Client-ID; bundled flavor spawns without `--client-id`
- `_cacheToken($accountId, $flavor, $accessToken, $expiresIn)` — writes to `spoton_token_${accountId}_${flavor}` key
- `removeAccount()` — deletes both `_own`, `_bundled`, and legacy `spoton_token_$accountId` keys for clean migration
- `refreshAllTokens()` — calls `_fetchKeymasterToken($id, 'own', ...)` (bundled is lazy-only)
- `_fetchDisplayName()` — calls `_fetchKeymasterToken($accountId, 'own', ...)`
- `SPOTON_DEFAULT_CLIENT_ID` constant (empty string placeholder)

### Client.pm — Dual-token routing pipeline

- `_request()` rewritten with dual-token pipeline:
  - Step 0: strip leading slash, compute `$isMeFamily` via `$_meFamilyRegex`
  - Step 1: per-flavor rate-limit check (`spoton_rate_limit_own` for me/*, `spoton_rate_limit_bundled` for others)
  - Step 2: response cache check (cache key excludes flavor — API-03)
  - Step 3: concurrency cap shared across both flavors (Pitfall 6)
  - Step 4: double-callback guard (`$userCbCalled`) prevents inflight counter drift (T-04.4-03)
  - Steps 5-6: flavor resolution and `_doFlavouredRequest` dispatch

- New helper methods:
  - `_resolveStartFlavor($cleanPath, $isMeFamily)` — D-05 me/* > D-06 hint cache > D-03 degraded > D-04 own-first
  - `_doFlavouredRequest(...)` — single-flavor HTTP execution, 403/410 bundled retry, hint caching on success
  - `_lookupBundledHint($cleanPath)` — reads 24h hint from cache
  - `_rememberBundledHint($cleanPath)` — writes 24h hint on confirmed bundled-retry success
  - `_is404Deprecated($cleanPath)` — distinguishes deprecated-404 from real-404 (Pitfall 4)

- `_onError()` updated: per-flavor rate-limit keys; flavor-specific token cache invalidation on 401; `$inflightCount` NOT decremented (managed by `$userCb` guard)

## Threat Mitigations Implemented

| Threat ID | Mitigation |
|-----------|-----------|
| T-04.4-01 | No token values in any log line — only path, flavor, status code |
| T-04.4-02 | Cache keys include accountId + flavor suffix — no cross-account contamination |
| T-04.4-03 | $userCbCalled guard ensures inflightCount decrements exactly once per request |
| T-04.4-04 | $_meFamilyRegex checked FIRST in routing — me/* NEVER falls back to bundled (D-05) |
| T-04.4-05 | Single-quote escaping on --client-id (T-04.3-05, already present) |

## Deviations from Plan

None — plan executed exactly as written. Both architectural patterns (inline `_request()` extension, `_doFlavouredRequest` extraction) implemented per PATTERNS.md and RESEARCH.md specifications.
