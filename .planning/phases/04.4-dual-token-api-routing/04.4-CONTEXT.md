# Phase 04.4: Dual-Token API Routing - Context

**Gathered:** 2026-05-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Dual-Flavor-Token-Routing in Client.pm: Jeder API-Request wird basierend auf dem URL-Pfad automatisch mit dem passenden Token-Flavor bedient — own-Token (eigene Client-ID) für User-Endpoints und Search, bundled-Token (librespot-Default-ID) für deprecated Browse-Endpoints und kuratierte Playlists. Mit 403/410-Fallback, 24h-Hint-Cache und hartem me/*-Guard. Ziel: Keine 429 Rate-Limit-Fehler durch Verteilung auf zwei Client-IDs (Spotty-NG-Referenzarchitektur).

**Hintergrund:** Phase 04.3 UAT hat gezeigt, dass die librespot-Default-Client-ID (`65b708073fc0480ea92a077233ca87bd`) aggressiv rate-limitiert wird (429 auf me/*, search, browse). Mit `--client-id` eigene-ID liefert Keymaster funktionsfähige Tokens, aber `browse/categories` gibt 404 (Dev-Mode-Block). Lösung: Dual-Token wie Spotty-NG.

</domain>

<decisions>
## Implementation Decisions

### Rate-Limiting
- **D-01:** Rate-Limiting pro Token-Flavor getrennt — ein 429 auf bundled-Token blockiert nur browse/categories-Requests, me/* und search laufen über own-Token weiter. Zwei separate Cache-Keys (spoton_rate_limit_own / spoton_rate_limit_bundled). _onError muss den aktuellen Token-Flavor kennen.

### Client-ID-Quelle
- **D-02:** SpotOn liefert eine eigene Client-ID mit (Default). Settings bieten zusätzlich ein optionales Feld für eine benutzerdefinierte Client-ID. Wer will, registriert eine eigene Spotify Developer App und trägt die ID in Settings ein.
- **D-03:** Degraded Mode bei blockierter SpotOn-Client-ID — alle Requests fallen auf bundled-Token zurück. User sieht Warnung in Settings mit aktivem Hinweis und Link zur Spotify Developer App-Registrierung ("Eigene Client-ID eintragen für volle Funktionalität").

### Token-Routing
- **D-04:** Implizites Routing bevorzugt — URL-Präfix (`me/*`, `browse/*`, `search`) bestimmt den Token-Flavor. Regex-Pattern-Array nur als Fallback für Sonderfälle wie kuratierte Playlists (`37i9*`-Prefix).
- **D-05:** Harter me/*-Guard für alle me/*-Pfade — me/tracks, me/playlists, me/player/* (Connect), me/following, me/top — immer own-Token, niemals bundled-Fallback. Gilt auch für zukünftige me/*-Endpoints (Phase 5 Connect).

### Default-Verhalten
- **D-06:** Unbekannte Pfade = own-first + Fallback — Pfade die nicht explizit geroutet sind, versuchen own-Token. Bei 403/410 automatisch Retry mit bundled-Token. Hint wird für 24h gecacht (LMS Cache). Neue Endpoints funktionieren automatisch.

### Claude's Discretion
- Token-Routing-Architektur: Wo die Flavor-Entscheidung lebt (Client.pm _request() vs. TokenRouter.pm vs. TokenManager.getToken() mit Flavor-Parameter) — Research/Planner wählt basierend auf Code-Struktur
- Fallback-Lifecycle: Ob 403/410-Retry transparent im selben Request-Lifecycle passiert oder als separater Request — Research/Planner entscheidet
- 429→Fallback-Verhalten: Ob ein 429 auf own-Token für nicht-me/*-Pfade einen bundled-Fallback versucht oder nur wartet — Research/Planner entscheidet basierend auf Spotify API-Verhalten
- Client-ID Speicherort: Konstante in Plugin.pm, Client.pm oder anderswo — Research/Planner wählt
- Bundled-Token-Refresh: Lazy-only (on-demand bei Bedarf) vs. eigener Timer — gemäß Research Option B (on-demand) bevorzugt

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `Client.pm::_request()` (Zeile 233-326) — Komplette Request-Pipeline mit Rate-Limit-Check, Cache, Concurrency, Token-Injection. Dual-Token-Routing wird hier oder vor dem Token-Step eingefügt.
- `Client.pm::_onError()` (Zeile 360-398) — 429/401-Handling bereits implementiert. Muss um per-Flavor Rate-Limit-Keys und 403/410-Fallback erweitert werden.
- `TokenManager.pm::_fetchKeymasterToken()` (Zeile 307-355) — Spawnt Binary mit/ohne --client-id. Bereits Dual-Token-fähig: ohne --client-id = bundled, mit = own.
- `TokenManager.pm::_cacheToken()` (Zeile 427-437) — Token-Cache mit TTL. Cache-Key muss um Flavor erweitert werden (spoton_token_{accountId}_own / spoton_token_{accountId}_bundled).

### Established Patterns
- Rate-Limit-Cache: `$cache->set(RATE_LIMIT_CACHE_KEY, 1, $retryAfter)` — wird zu per-Flavor Keys erweitert
- Token-Injection: `TokenManager->getToken($accountId, $cb)` — Interface muss Flavor-Parameter bekommen oder intern routen
- Binary-Spawn: `'%s' --get-token --cache '%s' --client-id '%s'` vs. `'%s' --get-token --cache '%s'` — Dual-Flavor über An/Abwesenheit von --client-id
- Concurrency-Cap: `MAX_CONCURRENT_REQUESTS => 3` — gilt für beide Flavors zusammen (ein Spotify-Account)

### Integration Points
- `Client.pm::_request()` → TokenManager: Muss Flavor-Info übergeben (welcher Token für diesen Pfad?)
- `Client.pm::_onError()` → Fallback-Retry: Bei 403/410 auf nicht-me/*-Pfaden mit anderem Flavor wiederholen
- `TokenManager.pm::refreshAllTokens()` → Muss ggf. auch bundled-Token refreshen (oder lazy-only)
- `Settings.pm` → Neues Client-ID-Feld, Degraded-Mode-Warnung mit Link zu developer.spotify.com

</code_context>
