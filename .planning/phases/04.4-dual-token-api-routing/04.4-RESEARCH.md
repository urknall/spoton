## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Rate-Limiting pro Token-Flavor getrennt — ein 429 auf bundled-Token blockiert nur browse/categories-Requests, me/* und search laufen weiter. Zwei separate Cache-Keys: `spoton_rate_limit_own` / `spoton_rate_limit_bundled`. `_onError` muss den aktuellen Token-Flavor kennen.
- **D-02:** SpotOn liefert eine eigene Client-ID mit (Default). Settings bieten zusätzlich ein optionales Feld für eine benutzerdefinierte Client-ID.
- **D-03:** Degraded Mode bei blockierter SpotOn-Client-ID — alle Requests fallen auf bundled-Token zurück. User sieht Warnung in Settings mit aktivem Hinweis und Link zur Spotify Developer App-Registrierung.
- **D-04:** Implizites Routing bevorzugt — URL-Präfix (`me/*`, `browse/*`, `search`) bestimmt den Token-Flavor. Regex-Pattern-Array nur als Fallback für Sonderfälle wie kuratierte Playlists (`37i9*`-Prefix).
- **D-05:** Harter me/*-Guard für alle me/*-Pfade — me/tracks, me/playlists, me/player/*, me/following, me/top — immer own-Token, NIEMALS bundled-Fallback. Gilt auch für zukünftige me/*-Endpoints.
- **D-06:** Unbekannte Pfade = own-first + Fallback — 403/410 auf own-Token → Retry mit bundled-Token → Hint für 24h cachen.

### Claude's Discretion

- Token-Routing-Architektur: Wo die Flavor-Entscheidung lebt (Client.pm `_request()` vs. eigenständige Funktion)
- Fallback-Lifecycle: Ob 403/410-Retry transparent im selben Request-Lifecycle passiert oder als separater Request
- 429→Fallback-Verhalten: Ob ein 429 auf own-Token für nicht-me/*-Pfade einen bundled-Fallback versucht
- Client-ID Speicherort: Konstante in Plugin.pm, Client.pm oder anderswo
- Bundled-Token-Refresh: Lazy-only (on-demand) vs. eigener Timer — Research/Planner entscheidet

## Summary

Phase 04.4 ist eine chirurgische Erweiterung von zwei Dateien (`API/Client.pm`, `API/TokenManager.pm`) plus einer kosmetischen Änderung in `Settings.pm`. Die gesamte Logik ist durch die Spotty-NG-Referenzimplementierung (`API.pm:1504-1703`) vollständig dokumentiert und empirisch validiert.

Das Kernproblem: Die librespot-Default-Client-ID (`65b708073fc0480ea92a077233ca87bd`) wird von Spotify aggressiv rate-limitiert (429 auf alle Endpoints). Mit einer eigenen Developer-App-Client-ID liefern `me/*` und `search` 200, aber `browse/categories` gibt 404 (Dev-Mode-Block). Die Lösung ist Dual-Flavor-Token-Routing: own-Token (eigene Client-ID) für User-Endpoints, bundled-Token (librespot-Default-ID) für deprecated Browse-Endpoints und kuratierte Playlists.

Die Implementierungsentscheidungen aus dem CONTEXT.md sind vollständig durch die Spotty-NG-Referenz und die empirischen Befunde aus der Phase-04.3-UAT abgedeckt. Keine offene Architekturentscheidung verbleibt — alle "Claude's Discretion"-Punkte sind durch Analyse des bestehenden Codes und der Spotty-NG-Referenz entschieden (siehe unten).

**Primary recommendation:** Implementiere das Routing als inline-Erweiterung von `Client.pm::_request()` (analog zu Spotty-NG `_call`) mit einem extrahierten `_requestOneShot()` für den Fallback-Retry — kein separates TokenRouter.pm-Modul.

## Architecture Patterns

### System Architecture Diagram

```
Client.pm::_request($method, $path, $params, $cb)
    │
    ├─ [Rate-Limit-Check: per Flavor]
    │   ├─ $cache->get('spoton_rate_limit_own')    → 429-Error (me/*, search)
    │   └─ $cache->get('spoton_rate_limit_bundled') → 429-Error (browse/*)
    │
    ├─ [Response-Cache-Check] (unverändert)
    │
    ├─ [Concurrency-Cap] (unverändert, shared über beide Flavors)
    │
    ├─ [Flavor-Routing-Entscheidung] ← NEU
    │   ├─ me/* → IMMER own (harter Guard, kein Fallback)
    │   ├─ Hint-Cache-Hit → direkt bundled (24h gespeichert)
    │   ├─ Degraded-Mode (own-Token nicht verfügbar) → direkt bundled
    │   └─ Alle anderen → own (mit 403/410/404-Fallback)
    │
    ├─ TokenManager->getToken($accountId, $flavor, $cb)
    │   ├─ flavor='own'  → $cache->get("spoton_token_${accountId}_own")
    │   │                 → _fetchKeymasterToken($id, '--client-id', $ownClientId, $cb)
    │   └─ flavor='bundled' → $cache->get("spoton_token_${accountId}_bundled")
    │                        → _fetchKeymasterToken($id, $cb) [ohne --client-id]
    │
    ├─ _requestOneShot($token, $flavor, ...) [HTTP-Request + Intercept-Callback]
    │   ├─ Erfolg (2xx) → Cache, $cb->($result)
    │   ├─ 429 → $cache->set("spoton_rate_limit_${flavor}", ...), $cb->(undef, 429-Error)
    │   ├─ 401 → Token-Cache invalidieren (flavor-spezifisch), $cb->(undef, 401-Error)
    │   ├─ 403/410/404-deprecated (nur wenn flavor='own' && !me/*) → Retry mit bundled
    │   │   └─ bundled-Retry erfolgreich → Hint-Cache schreiben (24h)
    │   └─ Sonstiger Fehler → $cb->(undef, Error)
    │
    └─ Settings.pm → Degraded-Mode-Warnung wenn own-Token nicht verfügbar
```

### Pattern 1: Token-Flavor-Routing in `_request()`

**Was:** Die Flavor-Entscheidung lebt direkt in `Client.pm::_request()`, analog zu Spotty-NG `_call()`. Ein separates `_requestOneShot()` führt den einmaligen HTTP-Call aus (verhindert Re-Eintritt in die Routing-Logik beim Fallback-Retry).

**Warum diese Architektur (Claude's Discretion):** `_request()` ist bereits der einzige Egress-Punkt (API-01). Die Flavor-Logik hier einzubauen erfordert keine neue Abstraktionsebene und minimiert die Diff-Größe. Der Spotty-NG-Präzedenzfall (`_call` / `_callOneShot`) bestätigt die Korrektheit dieser Struktur nach 3 Iterationen Härtung.

```perl
# Source: Spotty-NG API.pm:1504-1703 (adaptiert für SpotOn-Architektur)

# Konstanten (ergänzen bestehende)
use constant BUNDLED_HINT_TTL        => 86400;
use constant BUNDLED_HINT_KEY_PREFIX => 'spoton_bundled_hint_';
use constant SPOTON_DEFAULT_CLIENT_ID => 'EIGENE_SPOTON_CLIENT_ID_HIER';
# Hinweis: SpotOn-eigene Client-ID wird als Konstante in Client.pm oder Plugin.pm definiert (D-02)

# me/* Guard-Regex (package-level, einmalig kompiliert)
my $_meFamilyRegex = qr{^me(?:$|/|\?)};

# Deprecated-Families (analog Spotty-NG API.pm:66-91)
my @KNOWN_DEPRECATED_FAMILIES = (
    qr{^browse/featured-playlists\b},
    qr{^browse/categories/[^/?]+/playlists\b},
    qr{^browse/categories\b},
    qr{^browse/new-releases\b},
    qr{^recommendations\b},
    qr{^users/[^/?]+/playlists\b},
    qr{^artists/[^/?]+/top-tracks\b},
    qr{^artists/[^/?]+/related-artists\b},
    qr{^playlists/37i9[A-Za-z0-9]+\b},  # Kuratierte Spotify-Playlists
);

sub _request {
    my ($class, $method, $path, $params, $cb) = @_;

    my $cleanPath = $path;
    $cleanPath =~ s{^/}{};

    # me/* Guard — harter Block, keine Flavor-Alternative
    my $isMeFamily = ($cleanPath =~ $_meFamilyRegex);

    # Per-Flavor Rate-Limit-Check (D-01)
    my $rlKeyOwn     = 'spoton_rate_limit_own';
    my $rlKeyBundled = 'spoton_rate_limit_bundled';

    if ($isMeFamily && $cache->get($rlKeyOwn)) {
        $cb->(undef, { error => 'rate_limited', code => 429 });
        return;
    }
    if (!$isMeFamily && $cache->get($rlKeyBundled) && _isBundledPath($cleanPath)) {
        $cb->(undef, { error => 'rate_limited', code => 429 });
        return;
    }
    # Globaler Fallback für unbekannte Pfade mit globalem Rate-Limit
    if ($cache->get(RATE_LIMIT_CACHE_KEY)) {
        $cb->(undef, { error => 'rate_limited', code => 429 });
        return;
    }

    # Hint-Cache-Lookup (D-06) — nur für nicht-me/* Pfade
    my $hintFlavor  = $isMeFamily ? undef : _lookupBundledHint($cleanPath);
    my $startFlavor = $isMeFamily ? 'own' : ($hintFlavor || 'own');

    # Degraded-Mode-Check (D-03): own-Token nicht verfügbar → direkt bundled
    unless ($isMeFamily) {
        my $ownClientId = $prefs->get('clientId') || SPOTON_DEFAULT_CLIENT_ID;
        unless ($ownClientId) {
            $startFlavor = 'bundled';  # Kein own-Token möglich
        }
    }

    # Cache-Check, Concurrency-Cap etc. (unverändert) ...
    # [bestehender Code]

    $class->_doFlavouredRequest($method, $cleanPath, $params, $cb, $startFlavor, 0, $isMeFamily);
}
```

### Pattern 2: `_requestOneShot()` — Einmaliger HTTP-Call ohne Re-Routing

**Was:** Extrahierter innerer Call, analog zu Spotty-NG `_callOneShot`. Führt den Token-Fetch + HTTP-Call mit einem fixen Flavor aus. Wird vom Fallback-Retry direkt aufgerufen, ohne die Routing-Logik erneut zu durchlaufen.

**Warum (Claude's Discretion — Fallback-Lifecycle):** Der Retry passiert **transparent im selben Request-Lifecycle** (synchron in der Callback-Chain). Der Caller erhält exakt einen `$cb->()` Aufruf. Das ist das Spotty-NG-Muster und verhindert den "double-callback"-Fehler. [CITED: Spotty-NG API.pm:1380-1383]

```perl
# Source: Spotty-NG API.pm:1389-1502 (adaptiert)
sub _requestOneShot {
    my ($class, $flavor, $method, $path, $params, $cb) = @_;

    my $accountId = $params->{_accountId};

    require Plugins::SpotOn::API::TokenManager;
    Plugins::SpotOn::API::TokenManager->getToken($accountId, $flavor, sub {
        my $token = shift;
        unless ($token) {
            $cb->(undef, { error => 'no_token', flavor => $flavor });
            return;
        }
        # [HTTP-Call wie bisher, mit flavor-annotiertem _onSuccess/_onError]
        # $params->{_flavor} = $flavor wird mitgegeben für _onError
        $class->_executeHttp($token, $method, $path, $params, $cb);
    });
}
```

### Pattern 3: TokenManager — Flavor-Parameter in `getToken()`

**Was:** `getToken($accountId, $flavor, $cb)` bekommt einen optionalen `$flavor`-Parameter (`'own'` | `'bundled'`). Ohne Flavor-Parameter verhält sich die Methode wie bisher (backward-compatible, nutzt own-Flavor als Default).

**Cache-Keys:**
```perl
# Alter Key (wird ersetzt):
# spoton_token_{accountId}

# Neue Keys:
# spoton_token_{accountId}_own      ← --client-id eigene-ID
# spoton_token_{accountId}_bundled  ← ohne --client-id
```

**Backward-Compatibility:** Bestehende Aufrufe von `getToken($id, $cb)` ohne Flavor-Parameter werden auf `flavor='own'` gemappt. [ASSUMED — eigene Analyse]

```perl
sub getToken {
    my ($class, $accountId, $flavorOrCb, $cb) = @_;

    # Backward-compat: alter Aufruf ohne Flavor
    my $flavor;
    if (ref $flavorOrCb eq 'CODE') {
        $cb     = $flavorOrCb;
        $flavor = 'own';
    } else {
        $flavor = $flavorOrCb // 'own';
    }

    my $cacheKey = "spoton_token_${accountId}_${flavor}";
    if (my $cached = $cache->get($cacheKey)) {
        $cb->($cached);
        return;
    }

    $class->_fetchKeymasterToken($accountId, $flavor, $cb);
}
```

### Pattern 5: Hint-Cache — Schreiben und Lesen

**Was:** Der Hint-Cache speichert, welche URL-Patterns zuverlässig bundled-Token brauchen. Wird nur geschrieben wenn bundled-Retry erfolgreich (2xx). TTL: 24h. [CITED: Spotty-NG API.pm:93-98]

```perl
# Source: Spotty-NG API.pm:93-98, adaptiert

use constant BUNDLED_HINT_TTL        => 86400;  # 24h
use constant BUNDLED_HINT_KEY_PREFIX => 'spoton_bundled_hint_';

sub _lookupBundledHint {
    my ($cleanPath) = @_;
    for my $rx (@KNOWN_DEPRECATED_FAMILIES) {
        if ($cleanPath =~ $rx) {
            my $key = BUNDLED_HINT_KEY_PREFIX . md5_hex("$rx");
            return 'bundled' if $cache->get($key);
        }
    }
    return undef;
}

sub _rememberBundledHint {
    my ($cleanPath, $matchedRx) = @_;
    $matchedRx //= do {
        my $found;
        for my $rx (@KNOWN_DEPRECATED_FAMILIES) {
            if ($cleanPath =~ $rx) { $found = $rx; last; }
        }
        $found;
    };
    return unless $matchedRx;
    my $key = BUNDLED_HINT_KEY_PREFIX . md5_hex("$matchedRx");
    $cache->set($key, 1, BUNDLED_HINT_TTL);
    main::INFOLOG && $log->info("Client: bundled hint cached for pattern ($cleanPath)");
}
```

### Pattern 6: Per-Flavor Rate-Limit-Keys in `_onError()`

**Was:** `_onError()` muss den Flavor des fehlgeschlagenen Requests kennen, um den richtigen Cache-Key zu setzen. Flavor wird via `$params->{_flavor}` durchgereicht. [CITED: CONTEXT.md D-01]

```perl
sub _onError {
    my ($class, $http, $error, $response, $path, $params, $cb) = @_;

    $inflightCount--;

    my $code = # ... (unverändert)
    my $flavor = $params->{_flavor} // 'own';

    if ($code == 429) {
        my $retryAfter = # ... (unverändert, Cap 300s)
        # Per-Flavor Rate-Limit-Key (D-01)
        my $rlKey = ($flavor eq 'bundled')
            ? 'spoton_rate_limit_bundled'
            : 'spoton_rate_limit_own';
        $cache->set($rlKey, 1, $retryAfter);
        # Globalen Key ebenfalls setzen (Rückwärtskompatibilität)
        $cache->set(RATE_LIMIT_CACHE_KEY, 1, $retryAfter);
        $log->warn("Client: 429 rate limited ($flavor) for ${retryAfter}s");
        $cb->(undef, { error => 'rate_limited', code => 429 });
        return;
    }

    if ($code == 401) {
        # Token für spezifischen Flavor invalidieren
        my $accountId = $params->{_accountId} // '';
        $cache->remove("spoton_token_${accountId}_${flavor}") if $accountId;
        # ... (Rest unverändert)
    }
    # ...
}
```

### Anti-Patterns to Avoid

- **Double-Callback:** `_request()` darf `$cb` niemals zweimal aufrufen. Der `$userCbCalled`-Guard aus Spotty-NG ist essential. [CITED: Spotty-NG API.pm:1588-1593]
- **Recursion in Fallback:** Bei 403/410-Retry NICHT `_request()` erneut aufrufen (würde Hint-Cache und Routing erneut durchlaufen). Stattdessen `_requestOneShot()` direkt aufrufen. [CITED: Spotty-NG API.pm:1380-1383]
- **Hint-Cache vor Erfolg schreiben:** Hint nur cachen wenn bundled-Retry 2xx liefert, nie vorab. [CITED: Spotty-NG API.pm:1656-1662]
- **me/* auf bundled fallen lassen:** Nie. Auch ein 403 auf me/tracks ist kein Trigger für bundled-Fallback. [CITED: Spotty-NG API.pm:107-112]
- **Token-Cache-Key ohne Flavor:** Alter Key `spoton_token_{accountId}` wird ungültig. Code der diesen Key direkt liest (z.B. `removeAccount`) muss beide Flavor-Keys löschen.
- **429 auf own → bundled-Fallback versuchen:** Laut D-01 blockiert 429 auf own nur me/*/search — kein Fallback auf bundled bei 429. 429 bedeutet "warte", nicht "versuche anderen Token". [CITED: CONTEXT.md D-01]

## Don't Hand-Roll

| Problem | Nicht selbst bauen | Nutze stattdessen | Warum |
|---------|-------------------|-------------------|-------|
| Token-Cache mit TTL | Eigenes Hash-Expiry | `Slim::Utils::Cache->set($key, $val, $ttl)` | Eingebaut, thread-safe, korrekte Expiry-Semantik |
| Regex-Compilation | Inline `/$rx/` in Schleife | `my @compiled = (qr{...}, ...)` als package-var | Perl kompiliert qr{} einmalig; Schleife kompiliert nicht neu |
| 403/410-Retry-State | Globale Variable | Closure-Variable `$isRetry` | Closure-Scope verhindert Race-Conditions bei concurrent Requests |
| Client-ID-Persistenz | Plaintext-Datei | `Slim::Utils::Prefs->set('clientId', ...)` | LMS-Standard, automic flush, UI-Integration |

## Common Pitfalls

### Pitfall 1: me/*-Guard nach dem Hint-Cache-Lookup platzieren

**Was schiefläuft:** Wenn der Hint-Cache-Lookup vor dem me/*-Guard steht, könnte ein (korrupt) gecachter Hint für `me/tracks` einen bundled-Lookup auslösen.

**Warum:** Hint-Cache sollte nie me/*-Hints enthalten (da me/* niemals 403/410→bundled triggert). Aber defense-in-depth: me/*-Guard ZUERST prüfen.

**Wie vermeiden:** me/*-Guard ist immer die allererste Routing-Entscheidung. [CITED: Spotty-NG API.pm:1557-1559]

### Pitfall 2: `removeAccount()` löscht nur einen Token-Cache-Key

**Was schiefläuft:** `TokenManager::removeAccount()` löscht `spoton_token_$accountId` (alter Key). Nach der Migration gibt es aber `spoton_token_${accountId}_own` und `spoton_token_${accountId}_bundled`.

**Wie vermeiden:** `removeAccount()` muss beide Flavor-Keys löschen. [ASSUMED — eigene Analyse]

### Pitfall 3: `_cacheToken()` schreibt unter altem Key-Schema

**Was schiefläuft:** `_cacheToken($accountId, $token, $ttl)` schreibt aktuell unter `spoton_token_$accountId`. Muss auf `spoton_token_${accountId}_${flavor}` umgestellt werden.

**Wie vermeiden:** `$flavor` als Parameter zu `_cacheToken()` hinzufügen. [ASSUMED — Codebase-Analyse]

### Pitfall 4: 404 als bundled-Trigger für nicht-deprecated Pfade

**Was schiefläuft:** Ein echter "Track nicht gefunden"-404 löst fälschlicherweise einen bundled-Retry aus, weil der Code 404 global als deprecated-Signal behandelt.

**Warum:** Spotify liefert 404 für dev-mode-deprecated Endpoints, aber auch für "echte" 404s.

**Wie vermeiden:** 404 löst bundled-Retry NUR aus wenn der Pfad einem `@KNOWN_DEPRECATED_FAMILIES`-Regex matcht. [CITED: Spotty-NG API.pm:1613-1618]

### Pitfall 5: Hint-Cache-Key nicht stabil

**Was schiefläuft:** Cache-Key basiert auf dem URL-String direkt, nicht auf dem Regex-Pattern. Dann sind `browse/categories` und `browse/categories/0JQ5/playlists` separate Cache-Einträge — der erste Hint für eine URL deckt nicht die zweite ab.

**Wie vermeiden:** Cache-Key aus `md5_hex("$matchedRx")` ableiten — der Regex repraesentiert die ganze "Family". Alle URLs die denselben Regex matchen teilen sich einen Hint-Cache-Eintrag. [CITED: Spotty-NG API.pm:95-98]

### Pitfall 6: Concurrency-Cap gilt pro Flavor statt global

**Was schiefläuft:** Zwei separate Concurrency-Counter (einer pro Flavor) verdoppeln de facto die Spotify-Last.

**Wie vermeiden:** `MAX_CONCURRENT_REQUESTS => 3` gilt shared für beide Flavors. Ein Spotify-Account ist ein Spotify-Account — die AP-Rate-Limit-Druck-Verteilung kommt über zwei Client-IDs, nicht über zwei Concurrency-Caps. [CITED: CONTEXT.md code_context "Concurrency-Cap gilt für beide Flavors zusammen"]

```perl
# Source: Analyse von Client.pm + Spotty-NG API.pm:1504-1703
sub _request {
    my ($class, $method, $path, $params, $cb) = @_;

    my $cleanPath = $path;
    $cleanPath =~ s{^/}{};

    # Step 0: me/* family guard (D-05) — IMMER zuerst
    my $isMeFamily = ($cleanPath =~ $_meFamilyRegex);

    # Step 1: Per-Flavor Rate-Limit-Check (D-01)
    if ($isMeFamily) {
        if ($cache->get('spoton_rate_limit_own')) {
            $cb->(undef, { error => 'rate_limited', code => 429 });
            return;
        }
    } else {
        if ($cache->get('spoton_rate_limit_bundled') || $cache->get(RATE_LIMIT_CACHE_KEY)) {
            $cb->(undef, { error => 'rate_limited', code => 429 });
            return;
        }
    }

    # Step 2: Response-Cache-Check (unverändert, API-03)
    # [bestehender Code]

    # Step 3: Concurrency-Cap (unverändert, API-02)
    # [bestehender Code]

    # Step 4: Flavor-Entscheidung (D-04, D-06)
    my $hintFlavor  = $isMeFamily ? undef : _lookupBundledHint($cleanPath);
    my $startFlavor;
    if ($isMeFamily) {
        $startFlavor = 'own';
    } elsif ($hintFlavor) {
        $startFlavor = 'bundled';
    } else {
        # Degraded-Mode: SpotOn-eigene Client-ID nicht verfügbar?
        my $hasOwnId = ($prefs->get('clientId') || SPOTON_DEFAULT_CLIENT_ID) ? 1 : 0;
        $startFlavor = $hasOwnId ? 'own' : 'bundled';
    }

    # Step 5: Flavor-Request mit optionalem Fallback
    $inflightCount++;
    my $userCbCalled = 0;
    my $userCb = sub {
        return if $userCbCalled++;
        $inflightCount--;
        $cb->(@_);
    };
    $class->_doFlavouredRequest($method, $cleanPath, $params, $userCb, $startFlavor, 0, $isMeFamily);
}
```

### `_doFlavouredRequest()` mit 403/410-Fallback

```perl
# Source: Spotty-NG API.pm:1595-1703, adaptiert
sub _doFlavouredRequest {
    my ($class, $method, $path, $userCb, $flavor, $isRetry, $isMeFamily) = @_;

    require Plugins::SpotOn::API::TokenManager;
    TokenManager->getToken($params->{_accountId}, $flavor, sub {
        my $token = shift;
        unless ($token) {
            $userCb->(undef, { error => 'no_token' });
            return;
        }
        # HTTP-Call mit Intercept-Callback
        my $interceptCb = sub {
            my ($result, $err) = @_;
            my $code = $err ? ($err->{code} // 0) : 0;

            if (!$isRetry && $flavor eq 'own' && !$isMeFamily
                    && ($code == 403 || $code == 410 || _is404Deprecated($path, $code))) {
                # Bundled-Retry
                $class->_doFlavouredRequest($method, $path, $params, $userCb, 'bundled', 1, $isMeFamily);
                # Hint nur bei Erfolg cachen → erfolgt im bundled-Success-Callback
                return;
            }
            # 2xx: Hint cachen wenn dieser Request der bundled-Retry war
            if (!$err && $isRetry && $flavor eq 'bundled') {
                _rememberBundledHint($path);
            }
            $userCb->($result, $err);
        };
        $class->_executeHttp($token, $flavor, $method, $path, $params, $interceptCb);
    });
}
```

## Open Questions

1. **SpotOn-eigene Default-Client-ID**
   - Was wir wissen: D-02 sagt, SpotOn liefert eine eigene Client-ID als Default mit.
   - Was unklar ist: Die tatsächliche ID ist noch nicht festgelegt (muss ein registriertes Spotify Developer App sein).
   - Empfehlung: Planner legt einen Platzhalter-Placeholder fest (`SPOTON_DEFAULT_CLIENT_ID`). Die tatsächliche ID wird bei der Veröffentlichung des Plugins eingetragen. Für die Entwicklungsphase nutzt der User seine eigene registrierte App-ID.

2. **Spotty-NG hat ein Konzept von "Standard-User-Mode" (kein eigener Dev-App) vs. "Power-User-Mode"**
   - Was wir wissen: SpotOn's D-02/D-03 definiert: SpotOn liefert eine eigene Default-ID. Es gibt keinen "kein own-Token"-Zustand wie in Spotty-NG (wo der User möglicherweise keine eigene App konfiguriert hat).
   - Was unklar ist: Wie verhält sich SpotOn wenn die SpotOn-Default-Client-ID irgendwann rate-limitiert oder gesperrt wird (unwahrscheinlich, aber möglich)?
   - Empfehlung: D-03 Degraded-Mode deckt diesen Fall ab — Fallback auf bundled + Warnung in Settings. Kein blocking issue.
