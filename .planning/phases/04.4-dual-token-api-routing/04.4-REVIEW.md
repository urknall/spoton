# Phase 04.4: Code Review Report

**Reviewed:** 2026-05-29
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Phase 04.4 implements dual-flavor token routing (own-token vs. bundled-token), flavor-aware rate limiting, the 403/410 bundled fallback with hint cache, the me/* hard guard, and a Settings UI for the custom Client-ID. The architecture is sound and the me/* hard guard (D-05) is correctly implemented. However, three blocking defects were found: the T-04.4-01 input sanitization is silently bypassed by the LMS base handler, the RATE_LIMIT_CACHE_KEY cross-contaminates flavor-isolated rate limits (violating D-01), and own-token rate limits are not enforced at the _request entry point for non-me paths. Five warnings cover dead code and coupling issues that reduce maintainability.

---

## Critical Issues

### CR-01: SUPER::handler overwrites sanitized clientId with raw form input — T-04.4-01 bypass

**File:** `Plugins/SpotOn/Settings.pm:43-44, 72-77`

**Issue:** `prefs()` lists `'clientId'` alongside `'bitrate'` and `'normalization'`. When `SUPER::handler` runs (line 158 of Settings.pm), `Slim::Web::Settings::handler` iterates the prefs list and calls `$prefs->set($pref, $paramRef->{'pref_' . $pref})` for every entry — including `clientId` — using the **raw, unsanitized** form value (Slim::Web::Settings line 162). The subclass handler sanitizes and saves clientId before calling `SUPER::handler`, but SUPER then overwrites it with the raw value. The sanitization at lines 73–76 is completely ineffective. A user (or network attacker with an LMS session) can store arbitrary strings, including shell metacharacters like backticks or semicolons, in the `clientId` pref. If `SPOTON_DEFAULT_CLIENT_ID` is ever filled in with a real ID, the single-quote escaping in `_fetchKeymasterToken` would be the only remaining barrier — a defence-in-depth gap rather than a robust control.

**Fix:** Remove `'clientId'` from the `prefs()` return value so SUPER::handler never touches it. The manual sanitized save in the subclass handler is sufficient; the template already reads `customClientId` directly rather than `prefs.pref_clientId`.

```perl
# Settings.pm line 43-44 — remove 'clientId' from the auto-managed list
sub prefs {
    return ($prefs, 'bitrate', 'binary', 'normalization');
    # clientId is saved manually with sanitization in handler() — do NOT include here.
    # Including it causes Slim::Web::Settings::handler to overwrite the sanitized value
    # with the raw form input, bypassing T-04.4-01.
}
```

### CR-02: `RATE_LIMIT_CACHE_KEY` cross-contaminates flavor-isolated rate limits — D-01 violated

**File:** `Plugins/SpotOn/API/Client.pm:275-283, 459, 610`

**Issue:** The per-flavor rate-limit design (D-01) requires that a 429 on own-token blocks only own-paths and a 429 on bundled-token blocks only bundled-paths. The implementation breaks this guarantee in two ways:

1. Every 429, regardless of flavor, sets **both** the per-flavor key (`spoton_rate_limit_own` or `spoton_rate_limit_bundled`) AND the legacy global `RATE_LIMIT_CACHE_KEY` (lines 459 and 610).
2. Both branches of the Step 1 rate-limit check in `_request` test the legacy global key as a fallback (lines 275, 280).

Consequence: a 429 on own-token for a `search` call sets `RATE_LIMIT_CACHE_KEY`. The next `browse/categories` call (bundled-token) checks `spoton_rate_limit_bundled` (clear) but also `RATE_LIMIT_CACHE_KEY` (set) — and is rejected, even though the bundled token is not rate-limited. This is the exact cross-flavor blocking D-01 was designed to prevent.

**Fix:** Stop setting `RATE_LIMIT_CACHE_KEY` in `_doFlavouredRequest`. Keep the per-flavor keys only. Remove `RATE_LIMIT_CACHE_KEY` from both rate-limit check branches. The legacy constant can be deprecated.

```perl
# In _doFlavouredRequest error closure — lines 455-461:
if ($code == 429) {
    my $retryAfter = RATE_LIMIT_DEFAULT_BACKOFF;
    if ($response && ref $response && $response->can('header')) {
        my $headerVal = $response->header('Retry-After');
        $retryAfter = $headerVal if defined $headerVal && $headerVal =~ /^\d+$/;
    }
    $retryAfter = 300 if $retryAfter > 300;

    my $rlKey = ($flavor eq 'bundled')
        ? 'spoton_rate_limit_bundled'
        : 'spoton_rate_limit_own';
    $cache->set($rlKey, 1, $retryAfter);
    # DO NOT set RATE_LIMIT_CACHE_KEY here — it cross-contaminates flavors (D-01)
    $log->warn("Client: 429 rate limited [flavor=$flavor] for ${retryAfter}s on $cleanPath");
    $userCb->(undef, { error => 'rate_limited', code => 429 });
    return;
}

# In _request Step 1 — lines 274-283:
if ($isMeFamily) {
    if ($cache->get('spoton_rate_limit_own')) {
        $cb->(undef, { error => 'rate_limited', code => 429 });
        return;
    }
} else {
    # Check the start flavor to know which rate-limit key applies.
    # Since flavor resolution needs prefs, apply both checks here as the safe default.
    if ($cache->get('spoton_rate_limit_bundled') && $cache->get('spoton_rate_limit_own')) {
        $cb->(undef, { error => 'rate_limited', code => 429 });
        return;
    }
}
```

Note: the ideal fix moves the rate-limit check to after `_resolveStartFlavor` so the correct flavor key is known. The interim fix above at minimum removes cross-contamination.

### CR-03: Non-me paths do not enforce own-token rate limit at `_request` entry — own-429 not blocked

**File:** `Plugins/SpotOn/API/Client.pm:279-283`

**Issue:** The Step 1 rate-limit check for non-me paths (lines 279–283) only tests `spoton_rate_limit_bundled` and `RATE_LIMIT_CACHE_KEY`. Paths like `search`, `artists/*`, `albums/*`, and unknown paths are routed with own-token by `_resolveStartFlavor`, but `spoton_rate_limit_own` is **never consulted** for them. When own-token is rate-limited (e.g., after a 429 on a `search` call), subsequent `search` or `artists/*` calls proceed past Step 1, consume an inflight slot, and attempt an HTTP call that will receive another 429. This wastes inflight capacity and re-triggers rate-limit responses unnecessarily.

The rate-limit check is performed at Step 1, before `_resolveStartFlavor` runs at Step 5. The flavor of the actual request is therefore unknown at check time, which is the root cause.

**Fix:** Move the rate-limit check to after `_resolveStartFlavor`, or check both flavor keys for non-me paths until the resolution is known:

```perl
# After Step 5 (line 325), replace Step 1 with a post-resolution check:
my $startFlavor = $class->_resolveStartFlavor($cleanPath, $isMeFamily);

my $rlKey = ($startFlavor eq 'bundled') ? 'spoton_rate_limit_bundled' : 'spoton_rate_limit_own';
if ($cache->get($rlKey)) {
    $cb->(undef, { error => 'rate_limited', code => 429 });
    return;
}
```

This replaces the current Step 1 block entirely and correctly enforces the per-flavor rate limit for all paths (including the me/* case, since _resolveStartFlavor returns 'own' for me/*).

## Warnings

### WR-01: Step 2 cache check in `_request` is dead code

**File:** `Plugins/SpotOn/API/Client.pm:290-297`

**Issue:** The early cache check at lines 290–297 tests `$params->{_cacheKey}`. This key is only ever written **inside** `_doFlavouredRequest` (line 392), after the token is obtained. Every call to `_request` arrives with a fresh `$params` hashref from the public API methods (e.g., `search`, `getSavedTracks`). The concurrency-cap timer retry (Step 3) re-enters `_request` with the original `$params` reference, but since the timer fires before `$inflightCount` is incremented and before `_doFlavouredRequest` runs, `_cacheKey` is still unset at that point. The Step 2 check therefore never finds a cache hit and the `if (my $cacheKey = ...)` branch is unreachable in normal operation. The real cache check happens inside `_doFlavouredRequest` at line 388–399.

**Fix:** Remove the dead Step 2 block (lines 286–298) to eliminate the confusing double cache-check pattern.

### WR-02: `_onSuccess` and `_onError` named subs are dead code

**File:** `Plugins/SpotOn/API/Client.pm:553-628`

**Issue:** `_onSuccess` (line 556) and `_onError` (line 585) are standalone named subs left over from the pre-04.4 architecture. The phase-04.4 refactoring replaced them with inline closures inside `_doFlavouredRequest` that capture the `$flavor`, `$cleanPath`, `$isRetry`, `$isMeFamily`, and `$userCb` lexicals. Neither `_onSuccess` nor `_onError` is called from `_doFlavouredRequest` or anywhere else in the module. They carry outdated `$inflightCount` handling notes, and `_onError` reads `$params->{_flavor}` rather than the closed-over `$flavor`, which would be stale on the bundled retry path. Retaining dead code with slightly divergent logic is a maintenance hazard.

**Fix:** Delete the `_onSuccess` and `_onError` named subs. If test stubs rely on them, replace with thin wrappers that delegate to the inline closure logic.

### WR-03: `SPOTON_DEFAULT_CLIENT_ID` constant defined identically in two modules

**File:** `Plugins/SpotOn/API/TokenManager.pm:25`, `Plugins/SpotOn/API/Client.pm:32`

**Issue:** The constant `SPOTON_DEFAULT_CLIENT_ID => ''` is defined separately in both `TokenManager.pm` and `Client.pm`. When the real SpotOn app Client-ID is filled before release, a developer must update the value in two places. A mismatch — one updated, one not — would cause `_resolveStartFlavor` in `Client.pm` (which evaluates `SPOTON_DEFAULT_CLIENT_ID` to decide degraded mode) to disagree with `_fetchKeymasterToken` in `TokenManager.pm` (which uses the constant to choose the `--client-id` flag). The result would be a request routed as non-degraded but fetching a bundled-flavored token — a subtle logic inconsistency.

**Fix:** Define `SPOTON_DEFAULT_CLIENT_ID` once (e.g., in `TokenManager.pm` or a shared `Constants.pm`) and import/require it in `Client.pm`.

### WR-04: `Settings._autoSetupAccount` calls a private method on `TokenManager`

**File:** `Plugins/SpotOn/Settings.pm:214`

**Issue:** `_autoSetupAccount` calls `Plugins::SpotOn::API::TokenManager->_fetchDisplayName(...)` directly. Methods prefixed with `_` are by convention private to their module. This creates hidden coupling: if `TokenManager::_fetchDisplayName` is renamed, reordered, or its signature changes, `Settings.pm` will break silently (Perl does not enforce access control). Additionally, `_autoSetupAccount` duplicates the account-creation logic that already exists in `TokenManager::_setupAccountFromCredentials` — both parse `credentials.json`, derive `$accountId` via `md5_hex`, rename the `__DISCOVER__` dir, `chmod 0700`, and store prefs. The Settings variant stores the raw `$username` as display name instead of using the Spotify `/me` display name (the async update in the callback only writes the real name on the next request).

**Fix:** Replace `_autoSetupAccount` with a call to the existing public entry point (`TokenManager->_setupAccountFromCredentials`), or promote `_setupAccountFromCredentials` to a public method and call it from Settings.pm. Remove the duplicated logic.

### WR-05: `_request` concurrency-cap timer retry passes original `$path` with possible leading slash

**File:** `Plugins/SpotOn/API/Client.pm:304-307`

**Issue:** The concurrency-cap retry timer at lines 304–307 calls `$class->_request($method, $path, $params, $cb)` using the **original** `$path` argument, not `$cleanPath`. If the caller passes a path with a leading slash (e.g., `'/me/tracks'`), `$path` retains the slash. On re-entry, Step 0 strips it, so behaviour is correct, but the `$params` hashref is shared by reference across the retry. If in future code `$params` is mutated before the timer fires, the retry would see the mutated state. More concretely: any path that a public API method passes without a slash (the current convention) is not affected, but the code relies on an implicit convention that is not validated or documented.

**Fix:** Pass `$cleanPath` to the retry timer instead of `$path`:

```perl
Slim::Utils::Timers::setTimer(
    undef,
    Time::HiRes::time() + 0.1,
    sub { $class->_request($method, $cleanPath, $params, $cb) }
);
```

## Info

### IN-01: Missing `$accountId` validation in `_doFlavouredRequest`

**File:** `Plugins/SpotOn/API/Client.pm:357-358`

**Issue:** `$accountId = $params->{_accountId}` is extracted without a defined-or-default guard. If a caller omits `_accountId`, `$accountId` is `undef`. The downstream effects are: the `getToken` cache key becomes `spoton_token__own` (double underscore), `_fetchKeymasterToken` constructs a cache directory path with an empty component, and the 401-invalidation at line 467 uses the undef `$accountId` in a string interpolation resulting in `spoton_token__own`. All current callers correctly set `_accountId`, but the absence of a guard makes future callers fragile.

**Fix:** Add a guard and early error return:

```perl
my $accountId = $params->{_accountId};
unless (defined $accountId && length $accountId) {
    $log->error("Client: _doFlavouredRequest called without _accountId for $cleanPath");
    $userCb->(undef, { error => 'no_account' });
    return;
}
```
