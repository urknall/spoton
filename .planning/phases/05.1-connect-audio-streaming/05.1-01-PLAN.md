  <action>
    Three changes to connect.rs, all in the existing code — no new files:

    **Change A — LMS struct: add last_session_start_ns field (per D-03)**

    Add field `pub last_session_start_ns: Arc<AtomicU64>` to the LMS struct (after needs_position_sync). This follows the exact same pattern as seek_gen: Arc<AtomicU64>.

    In LMS::new(): initialize `last_session_start_ns: Arc::new(AtomicU64::new(0))`.

    In Clone impl: add `last_session_start_ns: Arc::clone(&self.last_session_start_ns)`.

    **Change B — Grace-timer logic in handle_player_event (per D-03)**

    In the TrackChanged arm, None branch (session start, line 222): store current SystemTime as nanoseconds-since-epoch in last_session_start_ns via Ordering::Release. Use `std::time::SystemTime::now().duration_since(UNIX_EPOCH).unwrap_or_default().as_nanos() as u64`. This ONLY triggers on None->Some (session start), NOT on track changes within an active session. Add eprintln! log: `[spoton] TrackChanged (session start): grace timer set`.

    In the Paused/Stopped arm (line 174): BEFORE `current_track.take()`, check the grace timer. Load last_session_start_ns with Ordering::Acquire. If the value is > 0 and `now_ns - last < 2_000_000_000` (2 seconds), log via eprintln! `[spoton] Paused/Stopped suppressed (grace timer, {elapsed_ms}ms)` and return early WITHOUT calling notify("stop") and WITHOUT clearing current_track. If grace period has elapsed, proceed with existing behavior.

    Per RESEARCH.md Pitfall 4: the grace timer ONLY fires on None->Some transition (first TrackChanged of a session). Track changes within an active session (Some->Some) do NOT reset the timer. This means user-initiated Pause during track-to-track transitions is never suppressed.

    **Change C — eprintln! debug logging throughout handle_player_event (per D-02)**

    Add named event logging at the entry of each match arm in handle_player_event. Use the `[spoton]` prefix for grep-filterability. Log the event name and relevant data:

    - Playing arm: `[spoton] Playing: track_id={new_id}, position_ms={position_ms}`
    - Paused arm: `[spoton] Paused: current_track={current_track:?}`
    - Stopped arm: `[spoton] Stopped: current_track={current_track:?}` (Paused and Stopped share an arm — log the discriminant to distinguish)
    - VolumeChanged arm: `[spoton] VolumeChanged: volume={volume}`
    - Seeked arm: `[spoton] Seeked: position_ms={position_ms}`
    - SessionConnected arm: `[spoton] SessionConnected`
    - TrackChanged arm: `[spoton] TrackChanged: track_id={new_id}`

    Note: Paused and Stopped share a match arm `PlayerEvent::Paused { .. } | PlayerEvent::Stopped { .. }`. To distinguish them in the log, use `std::mem::discriminant(event)` in the log line, e.g. `[spoton] Paused/Stopped (disc={:?}): ...`. The discriminant mapping from RESEARCH.md: 1=Stopped, 5=Paused.

    Also add sink-write logging in HttpStreamSink::write() after the frames_consumed update: log every 1000th frame with `[spoton] sink write: {frames_consumed} frames consumed`. This enables correlation of PCM data flow with event timing.

    All eprintln! lines are debug infrastructure per D-02; they will be removed in a future cleanup phase.
