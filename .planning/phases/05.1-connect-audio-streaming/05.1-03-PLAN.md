    Based on RESEARCH.md Open Question 1 and Pitfall 2: LMS sends `Range: bytes=0-` in the requestString. If the binary responds with `200 OK` and squeezelite rejects this (expecting `206 Partial Content`), apply the Range-aware response fix:

    In connect.rs http_stream_server, the GET /stream handler (around line 596): Check if the incoming request has a `Range` header. If so, respond with `206 Partial Content` and `Content-Range: bytes 0-` instead of `200 OK`. If no Range header, keep `200 OK`.

    Implementation: Before building the response, extract the Range header presence from the request. The request object `req` is available in the async closure. Add `let has_range = req.headers().contains_key("range");` BEFORE the pcm_rx drain block. Then in the response builder, conditionally set status and add Content-Range header.

    IMPORTANT: The `req` variable is consumed by the body read in the /control/* path. For /stream, the request is NOT consumed (no body read). So `req.headers()` is available at the point where the /stream response is built.

    If log analysis shows that squeezelite DOES accept 200 OK (connection is established, data flows), skip this fix — the problem lies elsewhere.
