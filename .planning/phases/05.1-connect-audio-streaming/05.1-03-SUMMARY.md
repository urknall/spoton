## Summary

Diagnosed and fixed Connect audio streaming. Three bugs resolved, one known gap remaining.

## What Was Done

### Bug 1 — Spurious Paused/Stopped (D-03): RESOLVED in Plan 01
Grace-timer in connect.rs suppresses Paused events within 2s of session-start TrackChanged. Confirmed working via binary stderr log: `[spoton] Paused/Stopped suppressed (grace timer, 0ms elapsed)`.

### Bug 2 — No Audio Output: RESOLVED
**Root Cause:** LMS selected `soc ogg * *` transcoding profile (codec 'o' / OGG) instead of `soc pcm * *` (codec 'p' / PCM). squeezelite was told to expect OGG data but the binary serves raw S16LE PCM, causing immediate disconnect (503 or format rejection).

**Discovery path:** Added relay debug logging to binary → found squeezelite disconnects immediately → added squeezelite debug logging (`-d slimproto=debug -d stream=debug`) → squeezelite log showed `codec: o` and `503 Service Unavailable` → identified wrong transcoding profile selection.

**Fix:** Removed `soc ogg * *` entry from `custom-convert.conf`. Connect mode always outputs raw PCM via HttpStreamSink, never OGG passthrough. LMS now correctly selects `soc pcm * *` → codec 'p' → squeezelite accepts the stream.

### Bug 2b — Daemon.pm tied-STDERR fork failure: RESOLVED
Plan 02's stderr capture caused Proc::Background to fail because LMS ties `*STDERR` to `Slim::Utils::Log::Trapper` (no `OPEN` method). Fixed by untie/retie around Proc::Background::new.

### Bug 3 — Time stuck at 0:00 (CON-14): RESOLVED
Resolved automatically when Bug 2 was fixed (per RESEARCH.md assumption A4). PCM data flows, squeezelite plays audio, time advances.

### Additional: isRepeatingStream added
Without `isRepeatingStream`, LMS treated the Connect stream as finite and sent `strm q` (stop) on pause, killing the TCP connection. With `isRepeatingStream` returning true for Connect mode, LMS sends only `strm p` (pause) and keeps the stream alive.

## Known Gaps

### Connect Resume (Play-Again after Pause) — NOT WORKING
When user pauses via Spotify app and then resumes, squeezelite stays paused. The binary fires `Playing` event on resume but does not send a new `start` notification to LMS. LMS has no trigger to unpause squeezelite.

**Diagnosis:** squeezelite debug log shows only `strm t` (timer) ticks after pause — no `strm u` (unpause) or `strm s` (new stream) arrives. Binary log shows `TrackChanged` + `Playing` but these map to LMS 'change' event (metadata update only), not 'start' event.

**Required fix:** Binary needs to send a resume/start event to LMS when Spotify resumes playback after a pause. This likely requires changes to connect.rs handle_player_event Playing arm to detect pause→play transitions and notify LMS.

### Connect Skip/Seek — NOT TESTED
Skip and seek from Spotify app were not tested in this session.
