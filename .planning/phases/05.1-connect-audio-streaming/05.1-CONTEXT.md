# Phase 05.1: Connect Audio Streaming Bugfix - Context

**Gathered:** 2026-06-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix des Audio-Datenpfads im Connect-Mode. Alle Event-, Metadaten- und Control-Pipelines sind verifiziert funktional (Phase 5 UAT). Nur der PCM-Audio-Stream kommt nicht beim squeezelite-Player an. Drei spezifische Bugs sind identifiziert:

1. squeezelite empfängt strm-Befehl von LMS, öffnet aber nicht konsistent eine TCP-Verbindung zum Binary-Stream-Port
2. Binary's Spirc feuert Stopped-Event kurz nach TrackChanged→Playing Sequenz, was die Session vorzeitig beendet
3. Auch bei bestehender Verbindung: time=0, PCM-Relay liefert möglicherweise keine Daten an den spezifischen squeezelite-Client

</domain>

<decisions>
## Implementation Decisions

### Debug-Methodik
- **D-01:** squeezelite mit `-d all` starten für vollen SLIMproto-Debug-Output. Zeigt genau welche strm-Befehle ankommen, welches Format/Codec gewählt wird, und ob die HTTP-Verbindung aufgebaut wird.
- **D-02:** Binary bekommt persistentes Debug-Logging via `eprintln!` nach stderr. Daemon.pm captured stderr in ein Log-File. Bleibt im Code bis eine spätere Phase es rückbaut.

### Spirc Stop-Event
- **D-03:** Stop-Unterdrückung binary-seitig — in `handle_player_event`: nach TrackChanged einen Grace-Timer setzen, während dessen Stopped/Paused nicht als "stop" an LMS gesendet wird. Problem wird an der Quelle gelöst, nicht durch Perl-seitiges Filtern.

### Spotty-NG Referenz
- **D-04:** Gezielter Diff an den drei Schmerzpunkten gegen Spotty-NG: (1) strm Format-Byte für spc/soc, (2) requestString für DirectStream-URLs, (3) Event-Sequenz TrackChanged→Playing. Kein vollständiger 1:1-Abgleich.

### Stream-Architektur
- **D-05:** DirectStream-Ansatz beibehalten — canDirectStream gibt HTTP-URL zurück, squeezelite verbindet sich direkt zum Binary. Das ist der bewährte Spotty-NG-Ansatz. Der Bug liegt im Setup, nicht im Konzept.

### Claude's Discretion
- Reihenfolge der Debug-Schritte (squeezelite-Log zuerst, dann Binary, dann Spotty-NG Diff)
- Ob stderr-Capture in Daemon.pm via Proc::Background `stderr` Option oder separatem Pipe-fd
- Konkreter Grace-Timer-Wert in der Binary (2s vorgeschlagen, kann angepasst werden)
- Ob die Perl-seitige Grace-Period (CONNECT_START_GRACE) zusätzlich vereinfacht wird

</decisions>

<code_context>
## Existing Code Insights

### Bekannte funktionierende Komponenten
- Binary HTTP-Server: `/stream` gibt HTTP/1.0 200 mit `audio/L16;rate=44100;channels=2` zurück, PCM-Daten fließen (614KB in 2s bei manuellem curl-Test)
- JSON-RPC Notify: `handle_player_event` → `notify()` sendet Events korrekt an LMS (verifiziert via /tmp/spoton-debug.log)
- spirc_active Watcher: Setzt Flag auf Playing/TrackChanged/SessionConnected (verifiziert)
- canDirectStream: Gibt korrekte HTTP-URL mit dynamischem Port zurück
- formatOverride: Gibt 'soc' nur für spotify://connect-* URLs zurück (Browse-Regression gefixt)

### Bekannte Problemstellen
- `Squeezebox.pm:581-587` — `getFormatForURL` wird auf DirectStream-URL aufgerufen, nicht auf Original-URL. Unser Fix gibt 'pcm' für `:PORT/stream` zurück, aber `$params->{'format'}` ist möglicherweise schon gesetzt
- `connect.rs:148-170` — Playing-Handler: None→Some setzt nur Cursor, sendet kein "start" mehr (TrackChanged ist authoritative). Aber Stopped zwischen TrackChanged und Playing cleared Cursor
- `Connect.pm:614-636` — Stop-Handler: Grace-Period jetzt ohne isPlaying-Check, aber CONNECT_START_GRACE=12s könnte zu lang sein für echte User-Stops

### Integration Points
- squeezelite ↔ LMS: SLIMproto Port 3483, strm-Kommando mit Format-Byte + Server-IP:Port + HTTP-Request-String
- LMS ↔ Binary: HTTP DirectStream auf dynamischem Port, Content-Type audio/L16

</code_context>

<specifics>
## Specific Ideas

- squeezelite `-d all` Log ist der primäre Debug-Vektor — dort sehen wir exakt welches strm-Kommando ankommt und was squeezelite damit macht
- Binary stderr-Logging persistent einbauen (eprintln für Events + Sink-Writes), später rückbauen
- Spotty-NG spc-Pfad nur an den drei Schmerzpunkten vergleichen, nicht komplett nachbauen
- Debug-Logging Rückbau als Ticket/Memory für spätere Phase merken

</specifics>
