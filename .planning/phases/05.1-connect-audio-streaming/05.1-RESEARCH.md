## Summary

Diese Phase fixiert drei spezifische Bugs im Connect-Audio-Datenpfad. Alle anderen Connect-Subsysteme (Event-Dispatch, Metadata, Control-Endpoints) sind verifiziert funktional.

**Bug 1 — Spirc feuert Paused nach TrackChanged:** Im `spoton-debug.log` ist die Sequenz sichtbar: Discriminant(13)=TrackChanged → notify "start" OK → Discriminant(5)=Paused → notify "stop" OK. Paused (disc=5) feuert aus der librespot-Spirc-Implementierung kurz nach dem TrackChanged-Event während des Übergangszustands zwischen Sessions. Fix: Grace-Timer in `handle_player_event` in `connect.rs` — Paused/Stopped ignorieren für 2 Sekunden nach dem letzten TrackChanged-Event.

**Bug 2 — squeezelite öffnet TCP-Verbindung nicht konsistent:** Der primäre Debug-Vektor ist squeezelite mit `-d slimproto:debug`. Aus der Code-Analyse: Die DirectStream-Kette ist korrekt implementiert (formatOverride→soc, transcoder command="-", canDirectStream→HTTP-URL, getFormatForURL→pcm, formatbyte='p'). Der Verdacht liegt auf dem `requestString`: LMS ruft `HTTP.pm::requestString` mit der DirectStream-URL auf. Diese Methode fügt einen `Range: bytes=0-` Header hinzu. Der Binary-HTTP-Server antwortet mit HTTP/1.0 200, ohne Range-Unterstützung. squeezelite könnte einen Range-Mismatch erhalten und die Verbindung verwerfen — oder der Header fehlt komplett im Binary-Response. Ohne squeezelite `-d slimproto` Log ist dies nicht eindeutig zu bestätigen.

**Bug 3 — time=0 / PCM-Relay liefert keine Daten:** Symptom ist wahrscheinlich sekundär zu Bug 2 (keine TCP-Verbindung = keine Daten = kein Zeitfortschritt). Falls Bug 2 behoben ist, löst sich Bug 3 mit hoher Wahrscheinlichkeit auf.

**Primäre Empfehlung:** Zuerst squeezelite-Logging aktivieren und Binary-stderr-Logging einbauen um klare Evidenz zu sammeln, dann Bug 1 (Grace-Timer) und Bug 2 (HTTP-Response-Headers) parallel fixen.

### Pattern 1: Tokio-basierter Grace-Timer in handle_player_event

**Was:** Nach TrackChanged einen Zeitstempel speichern; Paused/Stopped-Events für die Dauer des Grace-Zeitfensters ignorieren.
**Wann:** Immer wenn ein spurious Stop/Pause direkt nach TrackChanged feuert.

```rust
// Source: Eigenes Muster; tokio::time::Instant ist [ASSUMED] die beste Wahl
// (alternativ: std::time::Instant, kein async nötig da handle_player_event async ist)
use std::time::Instant;

// Zustand im LMS-Struct oder als separater Arc<AtomicU64>:
// Option A: AtomicU64 als UNIX-Timestamp-Nanos
use std::sync::atomic::{AtomicU64, Ordering};
let last_track_changed_ns: Arc<AtomicU64> = Arc::new(AtomicU64::new(0));

// Bei TrackChanged:
last_track_changed_ns.store(
    std::time::SystemTime::now()
        .duration_since(std::time::UNIX_EPOCH)
        .unwrap_or_default()
        .as_nanos() as u64,
    Ordering::Release,
);

// Bei Paused/Stopped:
let grace_ns = 2_000_000_000u64; // 2 Sekunden
let last = last_track_changed_ns.load(Ordering::Acquire);
let now_ns = std::time::SystemTime::now()
    .duration_since(std::time::UNIX_EPOCH)
    .unwrap_or_default()
    .as_nanos() as u64;
if now_ns.saturating_sub(last) < grace_ns {
    return; // Spurious Stop/Paused nach TrackChanged — ignorieren
}
```

**Saubere Alternative mit Instant (kein UNIX-Timestamp):**

```rust
// Verwende std::time::Instant in Arc<Mutex<Option<Instant>>>
// oder AtomicU64 mit elapsed()-Nanos relativ zu einem fixen Startpunkt.
// Einfachste Form: separates Arc<Mutex<Option<std::time::Instant>>>
```

### Pattern 2: Proc::Background stderr-Capture in Daemon.pm

**Was:** Neben dem bestehenden stdout-Pipe (für Port-Capture) einen zweiten stderr-File-Handle öffnen. LMS-Bundle-Proc::Background unterstützt `stderr => $fh`. [VERIFIED: /usr/share/squeezeboxserver/CPAN/Proc/Background.pm Zeile 104]

```perl
# Source: [VERIFIED: /usr/share/squeezeboxserver/CPAN/Proc/Background.pm]
# %Proc::Background::_available_options = (cwd=>1, stdin=>1, stdout=>1, stderr=>1, ...)

my $stderrFile = catfile($serverPrefs->get('cachedir'), 'spoton', $self->id . '-stderr.log');
open(my $stderr_fh, '>>', $stderrFile)
    or do { $log->warn("Cannot open stderr log $stderrFile: $!"); undef };

$self->_proc( Proc::Background->new(
    { 'die_upon_destroy' => 1, stdout => $port_w, ($stderr_fh ? (stderr => $stderr_fh) : ()) },
    $helperPath,
    @helperArgs,
) );

# Hinweis: $stderr_fh muss NACH dem close($port_w) offen bleiben (kein close notwendig
# da Proc::Background den Handle übernimmt, aber für Safety explizit offen lassen)
```

### Anti-Patterns to Avoid

- **Perl-seitiges Event-Filtern statt binary-seitigem:** CONNECT_START_GRACE in Connect.pm ist kein Ersatz für den Grace-Timer im Binary. Der Perl-seitige Grace existiert für den Fallback, aber der spurious Stop entsteht im Binary und muss dort unterdrückt werden.
- **HTTP/1.1 Keep-Alive im Binary-Server:** hyper verwendet HTTP/1.1 by default. Der Binary-Server verwendet explizit `http1::Builder` — sicherstellen dass squeezelite sich mit HTTP/1.0-Anfragen (via HTTP.pm::requestString) versteht. hyper's http1 serviert beide Varianten korrekt. [ASSUMED]
- **Grace-Timer zu lang:** Wert >3s riskiert echte User-initiated Pauses zu verzögern. Wert <1s löst den Bug möglicherweise nicht für langsame Netzwerkverbindungen. 2s ist der empfohlene Kompromiss (aus CONTEXT.md).

## Common Pitfalls

### Pitfall 1: spoton-debug.log zeigt Discriminant-Nummern, nicht Event-Namen

**Was schiefläuft:** Das bestehende Debug-Log schreibt `[event] Discriminant(5)` statt `[event] Paused`. Ohne Mapping ist das schwer lesbar.

**Warum:** Das Log wurde in Phase 5 mit generischem `discriminant()` implementiert, nicht mit Pattern-Matching.

**Wie vermeiden:** Im neuen eprintln!-Logging in D-02 nach Pattern matchen und Eventnamen ausschreiben.

**PlayerEvent-Discriminant-Mapping (librespot 0.8.0):** [VERIFIED: /home/sti/.cargo/registry/src/.../librespot-playback-0.8.0/src/player.rs]
```

1 = Stopped
2 = Loading
3 = Preloading
4 = Playing
5 = Paused         ← DER BUG: Paused kurz nach TrackChanged(13)
6 = TimeToPreloadNextTrack
7 = EndOfTrack
8 = Unavailable
9 = VolumeChanged
10 = PositionCorrection
11 = PositionChanged
12 = Seeked
13 = TrackChanged  ← Authoritative "start"-Quelle
14 = SessionConnected
15 = SessionDisconnected

```

**Beweis aus spoton-debug.log (Zeile 169-176):**

### Pitfall 2: Range-Header im requestString blockiert squeezelite

**Was schiefläuft:** `HTTP.pm::requestString` fügt automatisch `Range: bytes=0-` hinzu (Zeile 971 in Squeezebox.pm). [VERIFIED: /usr/share/perl5/Slim/Player/Protocols/HTTP.pm:971]

**Warum:** LMS sendet Range für alle HTTP-Streams. squeezelite erwartet möglicherweise `HTTP/1.0 206 Partial Content` wenn ein Range-Header gesendet wurde. Der Binary-Server antwortet mit `200 OK` ohne Content-Range, da es ein unendlicher Stream ist.

**Wie vermeiden:** Zwei mögliche Fixes:
1. Binary antwortet auf GET-Requests mit Range-Header trotzdem mit `200 OK` (RFC 2616-konform: Server darf Range ignorieren). Squeezelite muss 200 akzeptieren. [ASSUMED: squeezelite akzeptiert 200 bei Range-Request]
2. Alternativ: ProtocolHandler.pm überschreibt `requestString` um Range-Header wegzulassen. Spotty-NG tut dies NICHT (kein requestString-Override in ProtocolHandler.pm) — also ist Option 1 der richtige Ansatz.

**Squeezelite Verhalten:** squeezelite `-d slimproto:debug` zeigt das exakte strm-Kommando und ob die HTTP-Verbindung aufgebaut wird. Ohne dieses Log ist nicht verifizierbar ob Range das Problem ist. [ASSUMED: squeezelite loggt HTTP-Verbindungsfehler im slimproto-Log]

### Pitfall 3: Proc::Background stderr-Filehandle muss nach port_w-close offen bleiben

**Was schiefläuft:** Der Port-Capture-Code schliesst `$port_w` unmittelbar nach dem spawn (korrekt). Der `$stderr_fh` darf jedoch nicht vor dem Binary-Tod geschlossen werden.

**Wie vermeiden:** `$stderr_fh` im Daemon-Objekt als Accessor speichern (`mk_accessor`), damit Perl den Handle nicht vorzeitig garbage-collected.

### Pitfall 4: Grace-Timer im Binary kollidiert mit echtem User-Pause

**Was schiefläuft:** Wenn der User innerhalb der Grace-Periode (2s nach TrackChanged) manuell pausiert, wird dieses Event unterdrückt.

**Warum:** Spotify-App sendet Pause via Spirc, Binary empfängt PlayerEvent::Paused — und das wird jetzt ignoriert.

**Wie vermeiden:** Grace-Timer nur für den Übergang `current_track == None → Some` (d.h. erster TrackChanged, bevor irgendetwas gespielt wird), nicht für alle TrackChanged-Events. Oder: Grace nur triggern wenn `current_track` vorher `None` war (Session-Start), nicht bei Track-Wechseln.

**Empfehlung:** Den Grace-Timer nur setzen wenn `current_track` vorher `None` war (d.h. Session-Start-Übergang). Bei bereits aktiver Session (change-Event) keinen neuen Grace-Timer setzen.

### Pitfall 5: Two Player-Event-Channels im Binary — nur einer hat LMS-Notifier

**Was schiefläuft:** `run_connect` spawnt zwei `get_player_event_channel()`-Consumer: einen für den LMS-Notifier (Zeile 821), einen für den spirc_active-Watcher (Zeile 834). Der spirc_active-Watcher setzt den AtomicBool, aber hat keinen Grace-Timer-Zustand.

**Wie vermeiden:** Den Grace-Timer-Zustand (`last_track_changed` Timestamp) als Arc-Atomic implementieren, der von beiden Tasks shared werden kann — oder ihn nur in der LMS-Notifier-Task pflegen (empfohlen, da nur dort die Notify-Logik lebt).

## Common Pitfalls (Zusammenfassung für Planner)

1. **Spirc Paused/Stopped nach TrackChanged** — Binary-seitiger Grace-Timer nötig, nur bei None→Some-Transition (Session-Start), nicht bei Track-Change
2. **Range-Header in HTTP.pm::requestString** — Binary antwortet mit 200 OK (RFC-konform), squeezelite muss 200 bei Range-Request akzeptieren (verifizierbar via squeezelite `-d stream:debug`)
3. **Zwei Player-Event-Channels** — Grace-Timer-State nur in LMS-Notifier-Task, nicht im spirc_active-Watcher
4. **stderr_fh Garbage Collection** — Als Accessor im Daemon-Objekt speichern
5. **port_w close-Reihenfolge** — `close($port_w)` nach Binary-Spawn, stderr_fh offen lassen

## Open Questions

1. **Akzeptiert squeezelite 200 OK bei Range-Request?**
   - Was wir wissen: RFC 2616 erlaubt Server Range-Header zu ignorieren und 200 OK zu antworten
   - Was unklar ist: squeezelite 1.9.9-1449 tatsächliches Verhalten bei 200 OK statt 206
   - Empfehlung: squeezelite `-d stream:debug` Log nach Connection-Reset oder Retry auswerten; alternativ Binary-Server auf `206 Partial Content` anpassen (Response-Header ergänzen) falls 200 das Problem ist
