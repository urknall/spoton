# Phase 05.1: Code Review Report

**Reviewed:** 2026-06-01
**Depth:** standard
**Files Reviewed:** 4
**Status:** issues_found

## Summary

Phase 05.1 fixes the Connect audio streaming pipeline: squeezelite DirectStream connection timing, spurious Spirc Paused/Stopped suppression via a grace timer (D-03), and PCM relay data flow. The structural changes are sound and the core architecture (HTTP streaming, per-connection relay channel, relay_active guard) is correct.

The critical finding is a grace-timer over-suppression: the 2-second suppression window applies to both `Paused` and `Stopped`, but only `Paused` is documented as the spurious event. `Stopped` (end-of-track) fired within 2 seconds of session start is also swallowed, leaving LMS stuck in a playing state for short tracks or at playlist end. Four warnings cover a missing `require` in a new method, silent LMS connectivity failure, relay_active deadlock on panic, and a dead `_checkStreamStartTimes` function. Three info items flag notify error swallowing, a non-standard device_id hash, and the stderr log file never being explicitly closed on daemon restart.

## Critical Issues

### CR-01: Grace timer suppresses legitimate `Stopped` events for short tracks

**File:** `librespot-spoton/src/connect.rs:193-210`

**Issue:** The grace timer (D-03) suppresses **both** `Paused` and `Stopped` events within 2 seconds of a `None→Some` `TrackChanged`. The comment and rationale describe only the spurious `Paused` (discriminant 5) that Spirc fires immediately after `TrackChanged`. The `Stopped` event is different: it is the authoritative end-of-track signal.

For any track shorter than 2 seconds (jingles, short sound effects, podcast end-of-episode silence), the legitimate `Stopped` event fires within the grace window and is silently dropped. `current_track` remains `Some(id)`, `was_paused` stays `false`, and LMS never receives a `stop` command. LMS is then stuck in a playing state until the next `TrackChanged` or user action. If the playlist ends with a short track and no subsequent `TrackChanged` follows, LMS stays in playing state permanently for that session.

**Fix:** Restrict grace suppression to `Paused` only. `Stopped` must always be forwarded:

```rust
// Both Paused and Stopped collapse into `stop`. Only fire if we had an active track.
PlayerEvent::Paused { .. } => {
    log::debug!("[spoton] Paused (disc={:?}): current_track={:?}",
        std::mem::discriminant(event), current_track.as_deref());
    // D-03: Grace-timer — suppress spurious Paused within 2s of session start.
    // Spirc fires Paused immediately after TrackChanged at session start.
    let grace_ns: u64 = 2_000_000_000;
    let last = self.last_session_start_ns.load(Ordering::Acquire);
    if last > 0 {
        let now_ns = std::time::SystemTime::now()
            .duration_since(std::time::UNIX_EPOCH)
            .unwrap_or_default()
            .as_nanos() as u64;
        if now_ns.saturating_sub(last) < grace_ns {
            log::debug!("[spoton] Paused suppressed (grace timer)");
            return;
        }
    }
    if current_track.is_some() {
        self.was_paused.store(true, Ordering::Release);
        self.notify("stop", "", "").await;
    }
}
PlayerEvent::Stopped { .. } => {
    // NOT subject to grace timer — Stopped is the authoritative end-of-track signal.
    log::debug!("[spoton] Stopped: current_track={:?}", current_track.as_deref());
    if current_track.is_some() {
        self.was_paused.store(true, Ordering::Release);
        self.notify("stop", "", "").await;
    }
}
```

### WR-01: `isRepeatingStream` calls `Plugins::SpotOn::Connect` without `require`

**File:** `Plugins/SpotOn/ProtocolHandler.pm:151-154`

**Issue:** The newly added `isRepeatingStream` method calls `Plugins::SpotOn::Connect->isSpotifyConnect($song->master())` without a preceding `require Plugins::SpotOn::Connect`. Every other call site in the same file (`new`, `canSeek`, `canTranscodeSeek`) guards the call with `require`. If LMS invokes `isRepeatingStream` before any code path that loads `Connect.pm` has run — which can happen during abnormal or edge-case startup sequences — the call dies with `Can't locate object method "isSpotifyConnect" via package "Plugins::SpotOn::Connect"`.

**Fix:**

```perl
sub isRepeatingStream {
    my (undef, $song) = @_;
    require Plugins::SpotOn::Connect;
    return $song && Plugins::SpotOn::Connect->isSpotifyConnect($song->master());
}
```

### WR-02: `relay_active` flag is never cleared if the relay task panics

**File:** `librespot-spoton/src/connect.rs:596-651`

**Issue:** `relay_active_clone.store(false, Ordering::Release)` at line 650 is the only place the flag is cleared. It executes only on normal relay exit (when `conn_tx.send()` returns `Err`). If the relay task panics — for example, because `pcm_rx.lock().unwrap()` at lines 584, 622, or 633 unwinds a poisoned mutex — the store never runs. Subsequent `GET /stream` requests return `503 relay-busy` indefinitely: the daemon is alive and the sink is producing data, but the HTTP endpoint is permanently locked.

Mutex poisoning is triggered if a prior holder of the lock panicked while holding it. The relay task itself holds the lock during `try_recv()`, so a panic anywhere in that block (including OOM in downstream code called by `try_recv`) would poison the mutex.

**Fix:** Use `catch_unwind` or replace `unwrap()` with graceful handling:

```rust
// Replace all pcm_rx.lock().unwrap() in the relay task with:
let chunk = pcm_rx_clone.lock()
    .unwrap_or_else(|e| e.into_inner())  // recover from poisoned mutex
    .try_recv().ok();
```

And ensure `relay_active_clone` is always cleared by wrapping the relay body:

```rust
// At the end of the relay task, unconditionally clear the flag
relay_active_clone.store(false, Ordering::Release);
// This line already exists — it runs on normal exit.
// Add a panic hook or catch_unwind wrapper to cover the panic path.
```

### WR-03: `notify()` silently drops all LMS connectivity errors

**File:** `librespot-spoton/src/connect.rs:338-345`

**Issue:** Both error branches in `notify()` are completely empty — no logging, no metric, no debug output. A TCP connect failure or write failure is indistinguishable from success. During development and production debugging, LMS connectivity problems (wrong port, firewall, restart) produce no observable signal from the binary side. The current codebase uses `log::debug!` throughout; these paths should at minimum emit a debug or warn log.

```rust
// Current:
match TcpStream::connect(host_port).await {
    Ok(mut stream) => {
        if let Err(_e) = stream.write_all(request.as_bytes()).await {
        }
    }
    Err(_e) => {
    }
}
```

**Fix:**

```rust
match TcpStream::connect(host_port).await {
    Ok(mut stream) => {
        if let Err(e) = stream.write_all(request.as_bytes()).await {
            log::warn!("[spoton] notify({cmd}): write failed: {e}");
        }
    }
    Err(e) => {
        log::warn!("[spoton] notify({cmd}): connect to {host_port} failed: {e}");
    }
}
```

### WR-04: `_checkStreamStartTimes` is defined but never called — stream crash-backoff is dead code

**File:** `Plugins/SpotOn/Connect/Daemon.pm:211-233`

**Issue:** `_checkStreamStartTimes` implements crash-backoff for stream mode: after `MAX_STREAM_FAILURES` (5) stream crashes within `MAX_STREAM_INTERVAL` (2 minutes), it sets `_streamMode(0)` to disable streaming. However, the method is never called from `start()` or from any caller in `DaemonManager.pm`. The `_streamStartTimes` accessor is also never populated.

As a result, the stream crash-backoff protection is completely non-functional. The daemon will restart indefinitely after stream crashes, with only the general start-time backoff (`_checkStartTimes` → disable discovery) providing any protection.

**Fix:** Call `_checkStreamStartTimes()` from `start()` and return early if it signals disable:

```perl
sub start {
    my $self = shift;
    # ...existing code up to _checkStartTimes()...
    $self->_checkStartTimes();
    return if $self->_checkStreamStartTimes();   # <-- add this
    # ...rest of start()...
}
```

## Info

### IN-01: `notify()` does not read the HTTP response — may cause LMS "connection reset" log noise

**File:** `librespot-spoton/src/connect.rs:338-345`

**Issue:** The hand-rolled HTTP/1.0 request writes the request body and then drops the `TcpStream` without reading the response. HTTP/1.0 servers (LMS) may log a "connection reset by peer" or "broken pipe" when the client disconnects before reading the response. This is functionally correct (fire-and-forget) but generates diagnostic noise in LMS logs.

**Fix:** Either read and discard the response, or use `shutdown(Write)` before dropping to signal clean close:

```rust
use tokio::net::tcp::OwnedWriteHalf;
// After write_all succeeds, gracefully shut down the write side:
let _ = stream.shutdown().await;
```

### IN-02: `DefaultHasher` for `device_id` is not stable across Rust version upgrades

**File:** `librespot-spoton/src/connect.rs:957-961`

**Issue:** `DefaultHasher::new()` uses a fixed seed (not randomized), so the hash is stable across process restarts for a given binary. However, Rust documents `DefaultHasher` as an implementation detail that may change between compiler versions. If the binary is rebuilt with a new Rust version, `cache_dir` hashes to a different value, and Spotify/librespot registers a new device. This breaks ZeroConf discovery continuity.

**Fix:** Use a stable, version-independent hash (e.g., FNV or a simple deterministic function), or derive `device_id` from the player MAC address (already available as `player_mac`) rather than `cache_dir`:

```rust
// Use player_mac as device_id basis if available, otherwise fall back to cache_dir FNV
let device_id = if let Some(mac) = player_mac {
    mac.replace(':', "")
} else {
    // FNV-1a 64-bit — stable and not Rust-version-dependent
    let mut hash: u64 = 0xcbf29ce484222325;
    for byte in cache_dir.bytes() {
        hash ^= u64::from(byte);
        hash = hash.wrapping_mul(0x100000001b3);
    }
    format!("{hash:016x}")
};
```

### IN-03: Daemon stderr log file is never explicitly closed when daemon is restarted

**File:** `Plugins/SpotOn/Connect/Daemon.pm:158`

**Issue:** `$self->_stderrFh($stderr_fh)` stores the file handle to prevent premature GC. When `start()` is called again on an existing `Daemon` object (watchdog restart path via `DaemonManager.pm:140`), the accessor is overwritten with a new file handle. The old file handle is closed by Perl's garbage collector when the reference count drops to zero. This is reliable in CPython but less predictable under Perl's reference-count GC when circular references exist. In practice the handle will be closed promptly, but the sequence is implicit.

More importantly, the binary process inherits the file descriptor from the parent's open file handle. After parent reassigns `_stderrFh`, the old binary process still holds the file descriptor open (its own copy from `dup2`). No immediate bug, but a `stop()` + `start()` cycle leaves the old FD in the old process until it dies.

**Fix:** Explicitly close the old handle in `start()` before overwriting:

```perl
# Before: $self->_stderrFh($stderr_fh) if $stderr_fh;
if (my $old_fh = $self->_stderrFh) {
    close($old_fh);  # explicit, not relying on GC
}
$self->_stderrFh($stderr_fh) if $stderr_fh;
```
