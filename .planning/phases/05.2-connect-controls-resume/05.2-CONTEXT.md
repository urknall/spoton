<decisions>
## Implementation Decisions

### Resume-Mechanismus
- **D-01:** Neues Wire-Kommando `resume` — Binary sendet `notify("resume", track_id, position)` wenn Playing-Event für den selben Track nach Pause/Stopped feuert. Wire-Vokabular wird von 5 auf 6 Kommandos erweitert: start, change, stop, volume, seek, **resume**.
- **D-02:** Perl-seitiger `resume`-Handler in `_connectEvent` führt `['pause', 0]` (Unpause) aus UND synchronisiert Position via `startOffset`-Anpassung basierend auf der im Resume-Event mitgesendeten Position. Fortschrittsbalken steht nach Resume korrekt.

### Volume-Fix
- **D-03:** Root-Cause-Analyse für den 404 auf `/control/volume` und 411 auf Web API Fallback — Claude entscheidet den Fix-Ansatz (Logs-First oder Spotty-NG-Diff).
- **D-04:** Alle PUT-Endpoints in Client.pm prüfen ob Content-Length korrekt gesetzt wird (playerPause, playerPlay, playerVolume, playerSeek). Systemische Korrektur statt nur Volume-Fix.

### Seek-Deaktivierung im Connect-Modus
- **D-05:** LMS-seitiger Seek muss im Connect-Modus deaktiviert/wirkungslos sein. Infrastruktur existiert bereits (`Connect.pm:88` Kommentar, ProtocolHandler canSeek). Verifikation dass dies korrekt implementiert ist.

### Debug-Logging Rückbau
- **D-06:** Debug-Logging-Strategie für `eprintln!`-Zeilen in connect.rs — Claude entscheidet: Komplett entfernen, hinter `--verbose` Flag gaten, oder durch Rust `log`-Crate (`log::debug!`/`trace!` + `env_logger`) ersetzen.
- **D-07:** Perl-seitiges stderr-Capture in Daemon.pm — Claude entscheidet ob es als Crash-Diagnostik-Sicherheitsnetz beibehalten oder aufgeräumt wird.

### Claude's Discretion
- Ob `current_track` bei Paused/Stopped weiterhin via `take()` gecleard wird oder beibehalten bleibt — Trade-off-Analyse durch Researcher/Planner
- Debug-Logging-Ansatz: Entfernen vs. `log`-Crate vs. `--verbose` Flag — basierend auf Binary-Komplexität und Dependency-Budget
- stderr-Capture in Daemon.pm: Beibehalt vs. Aufräumen — basierend auf unabhängigem Nutzen für Crash-Diagnostik
- Volume 404 Root-Cause-Analyse: Logs-First vs. Spotty-NG-Diff — Researcher wählt basierend auf verfügbarer Diagnostik

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `Connect.pm::_sendControlCommand()` (Zeile 152-189) — HTTP-Control an Binary, mit Fallback. Vollständig implementiert, für Resume-Rückkanal wiederverwendbar.
- `Connect.pm::_connectEvent()` (Zeile 428-641) — Wire-Dispatch mit Handlern für start/change/stop/volume/seek. Muss um `resume`-Handler erweitert werden.
- Source-Marking Pattern (T-05-13) — Loop-Prevention auf allen bidirektionalen Pfaden etabliert.
- Volume-Debounce (`VOLUME_DEBOUNCE = 0.5s`) und Seek-Debounce (`SEEK_DEBOUNCE = 0.3s`) — bereits implementiert.

### Established Patterns
- Wire-Vokabular: `notify(cmd, p1, p2)` — einfaches 3-Parameter-Protokoll über JSON-RPC. Neues `resume`-Kommando folgt dem selben Pattern.
- Grace-Timer (D-03): `last_session_start_ns` AtomicU64 — suppresses spurious events. Muss nicht für Resume angepasst werden.
- CON-11 Volume Grace: Dual-Layer (Binary `suppress_next_volume` + Perl `VOLUME_GRACE_PERIOD`).

### Integration Points
- `connect.rs::handle_player_event()` → Neuer `resume`-Notify im Playing-Handler (None→Some oder Same-Track nach Paused)
- `Connect.pm::_connectEvent()` → Neuer `resume`-Branch: `['pause', 0]` + startOffset-Anpassung
- `Client.pm::player*()` → Content-Length auf allen PUT-Requests
- `connect.rs` → eprintln! Cleanup/Migration

</code_context>

<specifics>
## Specific Ideas

- Resume-Handler muss Position synchronisieren (`startOffset`-Anpassung), nicht nur unpausen — Fortschrittsbalken soll nach Resume korrekt stehen
- Seek in LMS muss im Connect-Modus deaktiviert sein — existierende Infrastruktur in Connect.pm/ProtocolHandler.pm verifizieren
- Alle PUT-Endpoints in Client.pm systematisch auf Content-Length prüfen — nicht nur Volume, auch Play/Pause/Seek
- Live-UAT-Session am Ende der Phase mit dem User für alle Kontrollpfade
- Kontrollpfad-Erwartungen exakt wie ROADMAP: Volume+Pause/Resume = bidirektional, Skip = semi-bidirektional, Seek = unidirektional (App→LMS only, LMS-Seek deaktiviert)
- Gapless Next Song: Wenn Spotify automatisch zum nächsten Track übergeht, soll das Audio nahtlos weiterlaufen (Binary streamt PCM kontinuierlich, TrackChanged→`change` Event aktualisiert nur Metadaten). Als UAT-Testfall verifizieren.

</specifics>
