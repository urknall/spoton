#### Full resume handler to insert after seek handler, before stop handler

```perl
# resume: same track after Pause/Stopped — unpause + position-sync
# Wire: p1=track_id(base62), p2=position_secs (3 decimals)
if ($cmd eq 'resume') {
    my $trackId  = $request->getParam('_p2');
    my $position = $request->getParam('_p3');   # NOTE: p3 not p2 for position

    return unless __PACKAGE__->isSpotifyConnect($client);

    main::INFOLOG && $log->is_info && $log->info(
        "Resume event: trackId=$trackId position=$position"
    );

    # Unpause — source-marked to prevent _onPause echo (T-05-13)
    my $unpauseReq = Slim::Control::Request->new($client->id, ['pause', 0]);
    $unpauseReq->source(__PACKAGE__);
    $unpauseReq->execute();

    # CON-13: adjust startOffset so songTime shows correct position
    # NEVER use ['time', N] in stream mode — would trigger _JumpToTime → stream restart
    if (defined $position && $position ne '') {
        my $song = $client->playingSong();
        if ($song) {
            my $elapsed = $client->songElapsedSeconds() || 0;
            $song->startOffset(int($position) - $elapsed);
            main::INFOLOG && $log->is_info && $log->info(
                "Resume: startOffset=" . $song->startOffset()
            );
        }
    }
    return;
}
```

**notify() parameter mapping:** `notify("resume", track_id, position_secs)` in Rust → LMS receives `_cmd=resume`, `_p2=track_id`, `_p3=position_secs`. The `getParam('_p3')` for position is intentional (p2 is track_id, p3 is position).

### Grace-Timer Guard Ordering

**Source:** `librespot-spoton/src/connect.rs`, Paused/Stopped handler (lines 187–200)

**Apply to:** New `was_paused.store(true)` in Paused/Stopped handler

The grace-timer `return` at line 199 MUST come before `was_paused.store(true)`. A spurious Paused event suppressed by the grace-timer must not set the flag, otherwise the next Playing event would send a false `resume` notify.

```rust
// Order is critical:
// 1. Grace-timer check (may return early)
if elapsed_ns < grace_ns {
    log::debug!("[spoton] Paused/Stopped suppressed (grace timer)");
    return;  // ← was_paused.store() must NOT execute before this return
}
// 2. Only AFTER grace-timer passes:
if current_track.is_some() {
    self.was_paused.store(true, Ordering::Release);  // ← only real pauses
    self.notify("stop", "", "").await;
    // KEIN take() — current_track bleibt gesetzt für Resume-Detection
}
```

### Logging Pattern (Perl)

**Source:** `Plugins/SpotOn/Connect.pm`, used throughout (e.g. lines 139–141, 254–255, 348–350)

```perl
main::INFOLOG && $log->is_info && $log->info("message $var");
```

Double-guard pattern (`main::INFOLOG &&` before `$log->is_info`) is the established SpotOn convention for zero-cost logging when log level is above threshold.

## Key Anti-Patterns (from RESEARCH.md — do not introduce)

| Anti-Pattern | Why Avoided | Correct Pattern |
|---|---|---|
| `['play']` for resume in stream mode | Opens new `/stream` HTTP connection, breaks audio | `['pause', 0]` — unpauses without reconnect |
| `current_track.take()` in Paused handler | Loses track_id context, blocks resume detection | Keep `current_track` set; use `was_paused` flag |
| `['time', N]` for position sync | Triggers `_JumpToTime` → `_Stop + _Stream` | `startOffset = int(position) - elapsed` |
| `was_paused.store()` before grace-timer return | Grace-suppressed events would falsely trigger resume | Store only after grace-timer passes |
| Missing `Content-Length` on PUT | Spotify API returns 411 Length Required | `push @headers, 'Content-Length' => 0` |
| 404 for volume/seek parse failure | Misleading — looks like endpoint not found | 204 (add volume/seek to the 204 condition) |
