## Summary

Phase 05.2 ist eine reine Bug-Fix- und Cleanup-Phase. Es gibt keine neuen Features. Drei unabhängige Bugs wurden in Phase 05.1 UAT identifiziert, und alle Code-Stellen sind bekannt.

**Bug 1 — Resume nach Pause:** Das Binary feuert `PlayerEvent::Playing` wenn Spotify Resume auslöst, aber der bestehende Playing-Handler in `connect.rs` behandelt den Fall `Some(prev) if prev == new_id` als No-Op (Zeile 158-165): er sendet keinen LMS-Notify. LMS hört kein Signal zum Fortsetzen und squeezelite bleibt pausiert. Fix: neues `resume` Wire-Kommando im Playing-Handler, das `notify("resume", track_id, position)` sendet wenn Playing für denselben Track nach einem Paused/Stopped-Zustand eintrifft. Perl-seitig: neuer `resume`-Branch in `_connectEvent` der `['pause', 0]` (Unpause) ausführt und `startOffset` anpasst.

**Bug 2 — Volume 411:** Der Spotify Web API Fallback in `_sendControlFallback` ruft `Plugins::SpotOn::API::Client->playerVolume()` auf. Dieser delegiert an `_doFlavouredRequest`, das am Ende `$http->put($url, @headers)` aufruft. `SimpleHTTP::Base::put` → `_createHTTPRequest` mit Methode `PUT`. Das `HTTP::Request`-Objekt wird ohne Body erstellt — kein `Content-Length`-Header. Die Spotify API antwortet mit 411 Length Required. Fix: `Content-Length: 0` explizit bei allen PUT-Calls ohne Body setzen.

**Bug 3 — Volume 404:** Das `/control/volume`-Endpoint im Binary ist implementiert (Zeile 672-683 in `connect.rs`) und parst `{"volume": 0-100}`. Die 404 deutet auf ein Path-Matching-Problem oder auf einen leeren Body-Parse-Fehler hin. Wenn der Body leer ist (`body_bytes.is_empty()`) gibt die JSON-Parse-`if`-Kette `None` zurück, und der `result`-Wert ist `None`. Der Status-Code-Entscheid bei Zeile 710-716 gibt 404 zurück für `volume` wenn `result.is_none()`. Das ist der tatsächliche Root-Cause: ein leerer oder fehlerhafter JSON-Body führt zu 404 statt zu einem hilfreichen Fehler.

Die Seek-Deaktivierung in `ProtocolHandler.pm::canSeek` / `canTranscodeSeek` ist bereits korrekt implementiert (gibt `0` zurück wenn `isSpotifyConnect($client)` true ist). Verifikation reicht aus.

**Primary recommendation:** Resume-Fix binary-seitig mit State-Tracking (`was_paused` Flag oder letzter Event-Typ), dann Perl-seitiger resume-Handler; Volume-Fix durch `Content-Length: 0` in `_doFlavouredRequest` für alle PUT-Requests; eprintln! durch `log::debug!` mit `env_logger` ersetzen für sauberes Logging ohne Dependency-Explosion.

## Bug-Analyse: Resume nach Pause

### Root Cause

**File:** `librespot-spoton/src/connect.rs`, Zeile 154-177

```rust
// [VERIFIED: Codebase grep]
PlayerEvent::Playing { track_id, position_ms, .. } => {
    let new_id = track_id.to_id().unwrap_or_default();
    match current_track.as_deref() {
        Some(prev) if prev == new_id.as_str() => {
            // PROBLEM: Nur position_sync — kein resume-Notify
            if self.needs_position_sync.load(Ordering::Acquire) {
                self.needs_position_sync.store(false, Ordering::Release);
                let secs = f64::from(*position_ms) / 1000.0;
                if secs > 1.0 {
                    self.notify("seek", &format!("{secs:.3}"), "").await;
                }
            }
            // Nach Pause/Stopped → Playing für denselben Track: keine Aktion → BUG
        }
        // ...
    }
}
```

Wenn der User in der Spotify App auf Play drückt (nach Pause), feuert librespot `PlayerEvent::Playing` mit derselben `track_id`. Der bestehende Code erkennt `Some(prev) if prev == new_id.as_str()` — das ist die Resume-Situation. Aber es wird weder `notify("resume", ...)` noch `notify("start", ...)` gesendet. LMS hört nichts, squeezelite bleibt pausiert.

**Zusätzliches Problem:** Bei `Paused/Stopped` (Zeile 201) wird `current_track.take()` aufgerufen — `current_track` wird auf `None` gesetzt. Wenn dann Playing kommt, trifft der `None`-Branch (Zeile 173-177), der nur den Cursor setzt und kein Notify sendet. In diesem Fall wird tatsächlich weder `resume` noch `start` gesendet.

**Konsequenz:** Der Resume-Bug hat zwei Trigger-Pfade:
1. `Paused` setzt `current_track = None` via `take()` → Playing trifft `None`-Branch → kein Notify
2. Falls `current_track` beibehalten würde (durch einen möglichen Fix) → `Some(prev) == new_id`-Branch → nur seek-sync, kein resume-Notify

### Fix-Strategie für D-01

**Empfehlung zum Claude's-Discretion-Punkt:** `current_track` bei Paused/Stopped NICHT via `take()` clearen — stattdessen einen `was_paused: AtomicBool` State-Flag in `LMS` einführen. `Paused/Stopped` setzt `was_paused = true`. Playing-Handler prüft: wenn `Some(prev) == new_id AND was_paused == true` → sendet `notify("resume", track_id, position_secs)` und setzt `was_paused = false`.

**Alternative (einfacher, aber funktional):** `current_track` bleibt `None` nach Paused/Stopped wie bisher. `None`-Branch in Playing sendet `notify("resume", new_id, position_secs)` wenn `last_stop_ns` (neues AtomicU64) innerhalb der letzten ~60 Minuten gesetzt wurde. Bei echtem Session-Start (TrackChanged None→Some) sendet TrackChanged den `start` — Playing im None-Branch ist Fallback. Mit `was_paused` ist klarer was gemeint ist.

**Empfohlene Implementierung (war_paused-Ansatz):**

```rust
// Neues Feld in LMS struct:
pub was_paused: Arc<AtomicBool>,

// In Paused/Stopped handler (nach Grace-Timer-Check):
if current_track.is_some() {
    self.was_paused.store(true, Ordering::Release);
    self.notify("stop", "", "").await;
}
// KEIN take() — current_track bleibt gesetzt

// In Playing handler, Some(prev) == new_id Branch:
Some(prev) if prev == new_id.as_str() => {
    if self.was_paused.swap(false, Ordering::AcqRel) {
        // Resume: selber Track nach Pause
        let secs = f64::from(*position_ms) / 1000.0;
        self.notify("resume", &new_id, &format!("{secs:.3}")).await;
    } else if self.needs_position_sync.load(Ordering::Acquire) {
        // ...seek-sync wie bisher...
    }
}
```

**Abweichung von D-01:** D-01 sagt `notify("resume", track_id, position)`. Der `notify()`-Code baut `params = [spottyconnect, cmd, p1, p2]`. Das passt zu: `notify("resume", track_id, position_secs)` → LMS empfängt `_cmd=resume`, `_p2=track_id`, `_p3=position_secs`. [VERIFIED: Codebase grep connect.rs:279-330]

### Perl-seitiger Resume-Handler (D-02)

**File:** `Plugins/SpotOn/Connect.pm`, `_connectEvent()`

Neuer Branch nach dem `seek`-Handler, vor dem `stop`-Handler:

```perl
# resume: selber Track nach Pause/Stopped — unpause + Position-Sync
if ($cmd eq 'resume') {
    my $trackId  = $request->getParam('_p2');
    my $position = $request->getParam('_p3');  # Sekunden (3 Dezimalstellen)

    return unless __PACKAGE__->isSpotifyConnect($client);

    main::INFOLOG && $log->is_info && $log->info(
        "Resume event: trackId=$trackId position=$position"
    );

    # Unpause (source-marked um _onPause-Echo zu verhindern)
    my $unpauseReq = Slim::Control::Request->new($client->id, ['pause', 0]);
    $unpauseReq->source(__PACKAGE__);
    $unpauseReq->execute();

    # CON-13: Position-Sync via startOffset (NICHT ['time', N] in stream mode)
    if (defined $position && $position ne '') {
        my $song = $client->playingSong();
        if ($song) {
            my $elapsed = $client->songElapsedSeconds() || 0;
            $song->startOffset(int($position) - $elapsed);
            main::INFOLOG && $log->is_info && $log->info(
                "Resume: startOffset=" . $song->startOffset()
            );
        }
    }
    return;
}
```

**Wichtig:** `['pause', 0]` ist das korrekte LMS-Kommando für "Unpause/Resume" (Parameter `_newvalue=0` bedeutet "nicht pausieren" = Play). Dies ist Source-marked um die `_onPause`-Subscriber-Schleife zu verhindern (T-05-13 Loop-Prevention bereits implementiert).

**Wire-Protokoll-Erweiterung:** Der Docstring in `_connectEvent` (Zeile 420-427) und der `LMS`-Struct-Docstring in connect.rs (Zeile 62) müssen aktualisiert werden.

## Bug-Analyse: Volume 404 und 411

### Root Cause: 404 auf /control/volume

**File:** `librespot-spoton/src/connect.rs`, Zeile 666-716

```rust
// [VERIFIED: Codebase grep]
"volume" => {
    if let Ok(json_val) = serde_json::from_slice::<serde_json::Value>(&body_bytes) {
        if let Some(vol) = json_val.get("volume").and_then(|v| v.as_u64()) {
            let vol_clamped = vol.min(100) as u32;
            let vol_u16 = (vol_clamped * 65535 / 100) as u16;
            spirc.set_volume(vol_u16).ok()
        } else {
            None  // ← Body hat JSON aber kein "volume" key
        }
    } else {
        None  // ← Body ist kein valides JSON (inkl. leerer Body)
    }
}
```

Status-Code-Entscheid (Zeile 710-716):
```rust
let status = if result.is_some() || cmd == "pause" || cmd == "play" || cmd == "next" || cmd == "prev" {
    StatusCode::NO_CONTENT
} else {
    StatusCode::NOT_FOUND  // ← volume mit result=None → 404
};
```

**Root Cause:** Die Status-Code-Logik gibt 404 zurück wenn `volume`/`seek` einen None-Result haben (Body-Parse-Fehler oder fehlender Key). Das ist korrekt für echte Fehler — aber der Caller (Perl `_sendControlCommand`) sieht 404 und logged "endpoint not found", was irreführend ist. **Der eigentliche Fehler liegt im Body, nicht im Endpoint.**

Zusätzlicher möglicher Trigger: Wenn `_sendControlCommand` den Body als `'{}'` sendet (leeres JSON-Objekt — Zeile 168 in Connect.pm): `my $jsonBody = $body ? eval { to_json($body) } : '{}'`. Wenn `$body` ein HashRef ohne `volume`-Key ist, wird `{}` serialisiert → JSON parse ok, aber `json_val.get("volume")` ist `None` → result=None → 404.

**Fix:** Entweder:
1. Status-Code-Logik erweitern: `volume` und `seek` bei fehlerhaftem Body → 400 Bad Request (nicht 404)
2. Oder: volumenspezifisches `result` als `Some(())` behandeln auch bei None (akzeptiere immer 204 für volume-Endpoint) — weniger korrekt semantisch

**Empfehlung:** Fix 1 — 400 Bad Request bei Parse-Fehler, 204 bei Erfolg. Macht Debugging einfacher.

### Root Cause: 411 auf Web API Fallback

**File:** `Plugins/SpotOn/API/Client.pm`, `_doFlavouredRequest()`, Zeile 535

```perl
# [VERIFIED: Codebase grep]
$http->$method($url, @headers);
```

Für PUT-Requests: `$method = 'put'` → `$http->put($url, @headers)`.

**File:** `/usr/share/perl5/Slim/Networking/SimpleHTTP/Base.pm`, Zeile 105-111

```perl
# [VERIFIED: Codebase grep LMS source]
$request = HTTP::Request->new( $type => $url );

if ( @_ % 2 ) {
    $request->content( pop @_ );  # Body nur wenn ungerade Anzahl von @_
}
```

Mit `$http->put($url, @headers)`: `@_` nach dem URL enthält nur Header-Paare (gerade Anzahl) → kein Body wird gesetzt → `HTTP::Request` hat keinen Content-Length. Die Spotify API reagiert mit 411 Length Required.

**Spotty-NG Fix (Zeile 1907-1909 in Spotty API.pm):** [VERIFIED: Codebase grep]
```perl
if ( $type eq 'PUT' || $type eq 'POST' ) {
    push @headers, 'Content-Length' => length($content || '');
}
```

Spotty-NG fügt Content-Length explizit in die Header-Liste ein, bevor der HTTP-Request erstellt wird. Das ist der korrekte Ansatz.

**Fix für SpotOn `_doFlavouredRequest`:**

```perl
# Vor dem $http->$method()-Aufruf:
if (uc($method) eq 'PUT' || uc($method) eq 'POST') {
    push @headers, 'Content-Length' => 0;  # Leerer Body für player control PUT-Requests
}
$http->$method($url, @headers);
```

**Systemische Prüfung (D-04):** Alle vier PUT-Endpoints:
- `playerPause`: `PUT /me/player/pause` — kein Body, kein `_method` in params (aber `_method => 'PUT'` gesetzt, wird aber nicht genutzt — der `method`-Parameter kommt von `$method = 'put'` aus dem `_request()`-Aufruf). Content-Length fehlt.
- `playerPlay`: `PUT /me/player/play` — identisches Problem
- `playerVolume`: `PUT /me/player/volume?volume_percent=N` — Parameter als Query-String, kein Body. Content-Length fehlt.
- `playerSeek`: `PUT /me/player/seek?position_ms=N` — Parameter als Query-String, kein Body. Content-Length fehlt.

Alle vier betroffen. Der Fix in `_doFlavouredRequest` heilt alle vier auf einmal.

**Hinweis zu `_method`-Param:** Die `_method`-Keys in playerPause/playerPlay/playerVolume/playerSeek sind Redundanz-Marker die nicht ausgewertet werden — sie landen via `next if $key =~ /^_/` im Filter. Die tatsächliche Methode kommt aus dem ersten Parameter von `_request('put', ...)`. Das ist korrekt und muss nicht geändert werden.

## Seek-Deaktivierung im Connect-Modus (D-05)

### Verifikation: Bereits korrekt implementiert

**File:** `Plugins/SpotOn/ProtocolHandler.pm`, Zeile 159-177 [VERIFIED: Codebase grep]

```perl
sub canSeek {
    my ($class, $client) = @_;
    if ($client) {
        require Plugins::SpotOn::Connect;
        return 0 if Plugins::SpotOn::Connect->isSpotifyConnect($client);
    }
    return Slim::Utils::Versions->compareVersions($::VERSION, '7.9.1') >= 0;
}

sub canTranscodeSeek {
    my ($class, $client) = @_;
    if ($client) {
        require Plugins::SpotOn::Connect;
        return 0 if Plugins::SpotOn::Connect->isSpotifyConnect($client);
    }
    return Slim::Utils::Versions->compareVersions($::VERSION, '7.9.1') >= 0;
}
```

Beide Methoden geben `0` zurück wenn `isSpotifyConnect` true ist. Das verhindert `_JumpToTime` → `_Stop + _Stream` in LMS.

**`_onSeek`-Subscriber in Connect.pm:** [VERIFIED: Codebase grep]
```perl
sub _onSeek {
    # ...
    return unless __PACKAGE__->isSpotifyConnect($client);
    # ...sendet seek an Binary
}
```

`_onSeek` leitet LMS-Seeks ans Binary weiter. Da `canSeek = 0`, ruft LMS `_JumpToTime` nicht auf — der `['time', N]`-Befehl wäre der Trigger für `_onSeek`. Aber LMS-UI-Seeks könnten `['time', N]` direkt senden (über CLI oder Web). Der Subscriber fängt das ab und leitet es ans Binary.

**Fazit:** Die Implementierung ist korrekt. Verifikation durch UAT-Testfall (Seek im LMS-Squeezelite-UI im Connect-Modus) ausreichend — kein Code-Fix nötig.

**Spotty-NG Vergleich:** In Spotty-NG Connect.pm `_bufferedSeek` (Zeile 441-446):
```perl
sub _bufferedSeek {
    # Stream mode: seek is disabled via canSeek → this should not be reached.
    # Guard against stale timer callbacks.
    main::INFOLOG && $log->is_info && $log->info(
        "Ignoring LMS seek in stream mode"
    );
}
```
SpotOn hat dasselbe Pattern, leitet aber ans Binary weiter. In SpotOn Connect mode ist LMS-seek → Binary→Spirc theoretisch erlaubt (die binary hat `/control/seek`). Aber der User hat entschieden: Seek im Connect-Modus ist unidirektional (nur App→LMS). LMS-seitiger Seek soll deaktiviert/wirkungslos sein. Deshalb ist der aktuelle `canSeek=0` Ansatz korrekt — LMS kann keinen Seek initiieren.

### Empfehlung zu D-07: stderr-Capture beibehalten

**Empfehlung:** Daemon.pm stderr-Capture beibehalten.

Nach D-06 wird das stderr-Capture weniger störend, weil Produktions-Binaries (ohne RUST_LOG) nichts mehr nach stderr schreiben.

## current_track take() — Entscheidung

**Claude's-Discretion-Punkt:** Ob `current_track` bei Paused/Stopped via `take()` gecleard wird oder beibehalten bleibt.

**Empfehlung: NICHT clearen — `was_paused` Flag stattdessen.**

Wenn `current_track` auf `None` gesetzt wird:
- `Playing` nach Pause trifft `None`-Branch → sendet kein Notify (bestehender Bug)
- Mit `was_paused`-Flag in der None-Branch-Logik: `notify("resume", ...)` möglich, aber semantisch unklar ob es wirklich ein Resume ist (könnte auch ein komplett neuer Session-Start sein)

Wenn `current_track` beibehalten wird + `was_paused`-Flag:
- `Playing` trifft `Some(prev) == new_id`-Branch → klarer Resume-Pfad
- `TrackChanged`-basierte echte Starts bleiben korrekt (TrackChanged setzt `current_track`)
- Session-Start nach komplettem Stop: librespot sendet TrackChanged vor Playing → TrackChanged setzt `current_track` neu, Playing trifft `Some(prev) == new_id` oder `Some(_) != new_id` (change) → korrekt

**Risiko von "nicht clearen":** Wenn nach einem echten Stop (Player wird weggenommen, Session endet) kein TrackChanged kommt und stattdessen direkt Playing für eine andere Session feuert → false resume. Dieses Szenario ist unwahrscheinlich weil librespot immer TrackChanged vor Playing sendet.

**Fazit:** `was_paused`-Flag ist robuster als `None`-Branch-Logik.

### Anti-Patterns to Avoid

- **`['time', N]` in Connect-Modus:** Triggert `_JumpToTime` → `_Stop + _Stream` → HTTP-Verbindung zum Binary wird getrennt und neu aufgebaut. Immer `startOffset` verwenden (CON-13).
- **`current_track.take()` nach Pause:** Verliert den Track-ID-Kontext, macht Resume-Detection ohne zusätzlichen State unmöglich.
- **404 vs 400 verwechseln im Binary:** 404 = Endpoint nicht gefunden, 400 = Bad Request (falsche Parameter). `/control/volume` mit leerem Body sollte 400 sein, nicht 404.
- **Content-Length weglassen bei PUT:** HTTP/1.1 erfordert Content-Length oder Transfer-Encoding für alle Requests mit potenziell leerem Body — Spotify API erzwingt das.

## Common Pitfalls

### Pitfall 1: [`pause`, 0] vs [`play`] für Unpause

**What goes wrong:** Man könnte annehmen `['play']` startet Playback neu (was HTTP-Stream neu öffnen würde).
**Why it happens:** LMS hat zwei Kommandos: `['play']` = neue Stream-Verbindung; `['pause', 0]` = Unpause ohne neue Verbindung.
**How to avoid:** Immer `['pause', 0]` für Resume in Connect-Modus. `['play']` würde eine neue HTTP-Verbindung zum Binary aufbauen, was den laufenden Stream unterbricht.
**Warning signs:** Audio stoppt kurz und springt dann nach Resume.

### Pitfall 2: was_paused-Flag bei Grace-Timer-suppressierten Paused-Events

**What goes wrong:** Grace-Timer supprimiert Paused-Events innerhalb von 2s nach Session-Start. Wenn das `was_paused`-Flag aber dort vor dem `return` gesetzt wird, würde ein spurious Paused den Flag setzen — und das nächste Playing sendet fälschlich ein `resume` statt einem `start`.
**Why it happens:** Grace-Timer `return` kommt vor der `current_track.is_some()`-Prüfung.
**How to avoid:** `was_paused.store(true)` NUR NACH dem Grace-Timer-Check — also nur bei wirklich weitergeleiteten Paused/Stopped-Events.

### Pitfall 3: notify()-Parameter-Reihenfolge für resume

**What goes wrong:** `notify("resume", track_id, position)` → LMS empfängt `_cmd=resume`, `_p2=track_id`, `_p3=position`. Wenn versehentlich `notify("resume", position, track_id)` — Perl liest falsche Werte.
**Why it happens:** `notify(cmd, p1, p2)` — p1 ist `_p2`, p2 ist `_p3` in LMS-Dispatch.
**How to avoid:** Docstring und Implementation synchron halten; Unit-Test mit korrekter Param-Reihenfolge.

### Pitfall 4: Loop-Prevention beim Unpause

**What goes wrong:** `['pause', 0]` ohne Source-Marking → `_onPause`-Subscriber feuert (Zeile 289: `return if ... && !$request->getParam('_newvalue')`) — aber `_newvalue=0` bedeutet "nicht pausieren", daher würde `_onPause` es korrekt NICHT weiterleiten. Dennoch: Source-Marking ist defensive Coding.
**How to avoid:** Source-Marking als Standard auf allen LMS-Request-Objekten die von Connect.pm ausgelöst werden (T-05-13 Pattern bereits überall etabliert).

### Pitfall 5: Spotify-API 411 nur beim Fallback-Pfad

**What goes wrong:** Binary-Control-Endpoints (/control/pause, /control/play, /control/volume) haben kein Content-Length-Problem (Hyper baut HTTP-Requests intern korrekt). Nur der Web-API-Fallback hat das Problem.
**Why it happens:** Zwei verschiedene HTTP-Stacks: Rust Hyper (Binary) vs. Perl SimpleAsyncHTTP.
**How to avoid:** Content-Length-Fix nur in `_doFlavouredRequest`, nicht in Binary-Code.
