# Code Review: Phase 05.2 — Connect Controls & Resume

**Scope:** All source changes in phase 05.2 (Connect.pm, Client.pm, connect.rs, main.rs)
**Effort:** high (7 finder angles, 1-vote verify)
**Result:** No critical or high-severity issues. 2 medium edge cases, 2 low-severity notes.

## Findings

### 1. [MEDIUM] was_paused blocks needs_position_sync via else-if

**File:** librespot-spoton/src/connect.rs:164-175
**Summary:** When both `was_paused` and `needs_position_sync` are true, the else-if structure means position sync is silently dropped in favor of resume.
**Scenario:** TrackChanged sets needs_position_sync=true, then Paused fires (sets was_paused=true), then Playing fires — resume is emitted but position sync is lost.
**Risk:** Low in practice — the two flags being simultaneously true requires Paused to fire after TrackChanged but before the first Playing event, which the 2s grace timer mostly prevents.
**Fix:** After resume notify, also check and consume needs_position_sync (send seek if > 1s).

### 2. [MEDIUM] TrackChanged same-ID no-op leaves stale was_paused on repeat

**File:** librespot-spoton/src/connect.rs:262
**Summary:** TrackChanged for the same track_id (repeat/loop) is a no-op that doesn't clear was_paused.
**Scenario:** User pauses → track repeats (same ID via Spotify repeat mode) → was_paused stays true → next Playing event sends false "resume" with position=0 instead of treating it as a fresh start.
**Risk:** Only affects repeat-one mode after explicit pause. Audio works but LMS gets an incorrect startOffset.
**Fix:** Clear was_paused in the same-track TrackChanged branch.

### 3. [LOW] 204 on JSON parse failure masks volume/seek errors

**File:** librespot-spoton/src/connect.rs:726
**Summary:** /control/volume and /control/seek return 204 even when JSON body parsing fails, because the condition includes `|| cmd == "volume" || cmd == "seek"`.
**Note:** This is by design (D-03 comment) — the Perl caller doesn't retry on 4xx, so 204 prevents false error cascades. The real fix (int() cast) is already in place on the Perl side.

### 4. [LOW] connectPauseTs echo window (1s) is empirically sufficient but not guaranteed

**File:** Plugins/SpotOn/Connect.pm:300
**Summary:** The 1s echo suppression window covers observed LMS internal event timing (10-300ms) but isn't formally bounded.
**Note:** Observed worst-case in logs was ~300ms. The 1s window provides 3x margin. If LMS event processing ever exceeds 1s (very slow hardware, heavy plugin load), a legitimate echo could leak through as /control/pause.
