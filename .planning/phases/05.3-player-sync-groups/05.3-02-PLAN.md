<objective>
Add relay_active timeout recovery and Spirc reconnect 'ready' notification to the binary.

Purpose: Two binary-side improvements for sync-group robustness and session handover:
(1) If relay_active gets stuck true (e.g., tokio task cancellation skips RelayGuard::drop),
a timeout health-check inside http_stream_server clears it after 60s of no data delivery.
This keeps relay_active scoped entirely within http_stream_server (per PATTERNS.md recommendation)
and avoids threading Arc into run_connect scope.
(2) After successful Spirc reconnect, notify LMS with a 'ready' command so the Perl side
can re-issue playlist play and resume streaming without user intervention (D-04 session handover).

Output: Modified connect.rs with both fixes + rebuilt binary.
</objective>

    Add a shared last_data_time: Arc<Mutex<Instant>> (or Arc<AtomicU64> storing epoch millis)

    In the main http_stream_server accept loop, before checking relay_active.swap(true) on a

    This allows the new connection to proceed instead of returning 503.

    Use std::time::Instant for the timestamp (already imported or add to use block).
    Use std::sync::Mutex for the shared Instant (lightweight, no async needed for a simple read/write).

    (line 894), clone lms: `let lms_for_reconnect = lms.clone();`. The original `lms` is still
    moved into the event dispatcher spawn. `lms_for_reconnect` stays in run_connect scope for
    the reconnect branches.

    In the "Reconnect with new credentials" branch (after line 1039 `connecting = false;`):
        lms_for_reconnect.notify("ready", "", "").await;
        log::debug!("[spoton] Spirc reconnected (new credentials) — notified LMS 'ready'");

    The "Spirc task completed — reconnect with backoff" branch sets connecting=true (line 1069)
    and actual reconnect happens in the next loop iteration in the "Reconnect with new credentials"
    arm. So the ready notify in the reconnect-success arm covers both cases.

    Do NOT thread relay_active into run_connect scope. Do NOT change http_stream_server's
    function signature. relay_active stays scoped within http_stream_server.
