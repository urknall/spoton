## Summary

Phase 05.3 has two independent technical problems that must both be solved for sync-group Connect audio to work end-to-end.

**Problem 1 ‚Äî Sync proxy disconnect:** LMS calls `ProtocolHandler::new()` to get a stream object for sync-group players. The implementation substitutes the `spotify://connect-*` URL with `http://127.0.0.1:PORT/stream` and delegates to `Slim::Player::Protocols::HTTP->new($args)`. This opens a **synchronous** TCP connection to the binary via `RemoteStream::new()` (which uses `sysreadline` to read headers blocking). The binary's hyper HTTP server responds correctly with `200 OK`, but something causes LMS to disconnect immediately afterward. Code analysis reveals both SpotOn and Spotty-NG use **identical** hyper HTTP server implementations. The disconnect must be caused by something downstream in the LMS HTTP.pm `response()` method ‚Äî specifically, a missing Content-Length causes LMS to enter "Persistent" enhancement mode (reconnection loop) rather than a straight streaming path.

**Problem 2 ‚Äî Session handover:** After the user switches LMS to a different source (Spotty, local music), LMS stops consuming the `/stream` endpoint. When the user presses Play in the Spotify app again, the Spirc session inside the binary may be dead because `spirc_active` went false when the HTTP stream was dropped, or because the session was killed when `_stopConnectDaemon` was called. The binary already has a reconnect loop (up to 5 times in 60s) ‚Äî but the Perl side has no mechanism to detect that the session died and re-trigger a `start` event. Additionally, after a long idle, the Spirc session may expire silently.

**Primary recommendation:** Fix the proxy disconnect by investigating what LMS does after receiving the 200 response. Force `Content-Length: 0` to NOT be present (omit it entirely) and add `Transfer-Encoding: chunked` or rely on HTTP/1.0 connection-close semantics. For session handover: use the binary's existing reconnect loop as the primary mechanism, add Perl-side lazy restart detection via the `_streamAlivePoll` watchdog.

## D-01 Analysis: Sync-Group Proxy Disconnect Bug

### What RemoteStream Does After Header Parsing [VERIFIED: codebase]

`RemoteStream::new()` ‚Üí `request()` ‚Üí `sysreadline()` reads headers ‚Üí `parseHeaders()`.

After headers, `HTTP.pm::response()` is called (line 104):
```perl
my $enhance = $self->canEnhanceHTTP($args->{'client'}, $args->{'url'});
# Returns $prefs->get('useEnhancedHTTP') ‚Äî defaults to 0 for LMS
```

If `canEnhanceHTTP` returns 0 (default), code at line 107:
```perl
if ($enhance == PERSISTENT || !$self->contentLength) {
    # Uses Persistent reconnecting client ‚Äî problematic for infinite streams
```

**Critical finding:** When `canEnhanceHTTP` returns 0 AND `contentLength` is absent (infinite stream), the condition `!$self->contentLength` is **true** (because no Content-Length header in the binary's 200 response), so LMS enters the "Persistent" enhanced HTTP path anyway. This path uses `Slim::Networking::Async::HTTP` internally and tries to make ranged reconnection requests. This is incompatible with an infinite streaming body.

The correct behavior: an infinite stream with no Content-Length should set `isLive(1)` at line 845 of HTTP.pm:
```perl
${*$self}{'song'}->isLive($length ? 0 : 1) if !$redir;
```
This IS set correctly. But `response()` still enters Persistent mode because `!$self->contentLength` is true.

**Root cause hypothesis:** LMS's `response()` method at line 107 enters the "Persistent service" path when there is no Content-Length. The Persistent path (`readPersistentChunk`) uses `Slim::Networking::Async::HTTP` for reconnections, which tries to send Range requests against a non-seekable infinite stream. This causes the disconnect.

**Fix approach (Claude's discretion):** The binary should not call `response()` in the first place for the proxy path ‚Äî but `response()` is called from `RemoteStream::request()`. The real fix is to make the binary's streaming response look like a live radio stream that LMS can consume without the Persistent enhancement path.

**Spotty-NG vs SpotOn comparison:** Both binaries have **identical** HTTP server implementations for the `/stream` endpoint (hyper HTTP/1.1, no Content-Length for streaming body, `Content-Type: audio/L16;rate=44100;channels=2`). If Spotty-NG works, the difference must be elsewhere.

**Most likely difference:** Spotty-NG's `ProtocolHandler::new()` calls `Slim::Player::Protocols::HTTP->new($args)` identically. But Spotty-NG may have a different `canEnhanceHTTP` return value or the Spotty-NG plugin registers a custom `useEnhancedHTTP` pref value. Alternatively, Spotty-NG may work because LMS's `Slim::Formats::RemoteStream::new()` short-circuits the `response()` call when `request()` returns `undef` from `canEnhanceHTTP`.

**Alternative root cause:** `RemoteStream::request()` sends `Range: bytes=0-` (from `requestString()` line 971). The binary may not handle the `Range` header on an infinite stream ‚Äî hyper may close the connection when it can't satisfy the range. Spotty-NG may handle this with a custom `requestString()` that suppresses the Range header for Connect streams.

**Key investigation point:** Does SpotOn's `ProtocolHandler` override `requestString()` to suppress the Range header? Currently: NO. Spotty-NG's ProtocolHandler also does not override `requestString()`. However, the range `bytes=0-` (no end bound) should be safe for a streaming server ‚Äî the response just starts from byte 0 and never ends.

**Most actionable fix:** Override `requestString()` in `ProtocolHandler.pm` for Connect proxy connections to return a plain `GET /stream HTTP/1.0` without the Range header. This prevents hyper from receiving a range request it might handle differently than expected.

**Alternative fix:** Add `Content-Length: 0` to `response()` path avoidance ‚Äî the binary could write a very large fake Content-Length header (e.g., `2147483647`) to trick LMS into the non-Persistent path. [ASSUMED ‚Äî needs verification]

### HTTP/1.0 Client + HTTP/1.1 Server: Transfer Encoding [VERIFIED: codebase]

`LMS Networking/Async/HTTP.pm` line 217: `# XXX until we support chunked encoding, force 1.0`

`requestString()` generates `GET $path HTTP/1.0`. When hyper receives an HTTP/1.0 request:
- hyper 1.x automatically responds with HTTP/1.0 (no chunked encoding)
- Response body terminates by connection close
- No `Transfer-Encoding: chunked` in response

This means **chunked encoding is NOT the issue** ‚Äî hyper handles it correctly for HTTP/1.0 clients. [ASSUMED based on hyper 1.x behavior]

### What Spotty-NG Does Differently [VERIFIED: codebase analysis]

After code comparison, the SpotOn and Spotty-NG `ProtocolHandler.pm` proxy paths are **nearly identical** for the Connect path. Both call `Slim::Player::Protocols::HTTP->new($args)` with the substituted URL. The only meaningful difference found is that Spotty-NG uses `spc` content type (SpotOn uses `soc`), but this does not affect the HTTP connection.

The likely real difference: Spotty-NG's binary has been running in production and has been tested with actual LMS proxy connections. SpotOn's binary may have a subtle bug where it closes the connection prematurely under certain conditions (e.g., `relay_active` flag sticking `true` from a prior connection attempt, causing the binary to return 503 instead of 200 on reconnect).

**`relay_active` bug pattern [VERIFIED: codebase]:** In `connect.rs` lines 569-581, when `relay_active.swap(true)` returns true (relay already active), the binary returns 503. If a previous connection attempt left `relay_active=true` (e.g., due to a panic in the relay task), subsequent `/stream` requests will always get 503. The `RelayGuard` drop handler should clear this, but a panic in the relay task could skip the guard. [ASSUMED ‚Äî needs testing]

## D-04 Analysis: Session Handover

### What Happens to Spirc After LMS Source Switch [VERIFIED: codebase]

**Scenario 1: User switches to different LMS source while Connect is active**

1. `_onNewSong` fires for the new non-Connect song
2. If the new URL is not `spotify://connect-*`, `$_activeConnectPlayer` is cleared
3. LMS stops consuming the `/stream` endpoint ‚Äî TCP connection drops
4. In binary: `conn_tx.send(bytes).await.is_err()` ‚Üí relay task exits ‚Üí `relay_active = false`
5. Spirc is still alive ‚Äî it continues running, `spirc_active` stays true
6. When user presses Play in Spotify app: Spirc fires `TrackChanged` ‚Üí `notify("start")` ‚Üí LMS `_connectEvent` receives "start" command
7. LMS `_connectEvent` issues `playlist play spotify://connect-<ts>`
8. LMS calls `canDirectStream()` ‚Üí binary still has `_streamMode=true` and `_streamPort` set ‚Üí returns HTTP URL
9. squeezelite connects to binary ‚Üí stream resumes

**This scenario should work already** if `_stopConnectDaemon` is NOT called. The issue is that D-08 mutual exclusion code in `ProtocolHandler::new()` calls `_stopConnectDaemon()` which kills the binary.

**Scenario 2: D-08 stops the binary when Browse starts**

1. User navigates SpotOn Browse while Connect is active
2. `ProtocolHandler::new()` called for `spotify://` Browse URL
3. D-08 code fires: `Connect->_stopConnectDaemon($client)`
4. `DaemonManager->stopHelper($clientId)` ‚Üí `Daemon::stop()` ‚Üí `_proc->die`
5. Binary is dead ‚Äî device disappears from Spotify app
6. `_streamAlivePoll` (5s interval) detects `alive=0` ‚Üí calls `$helper->start`
7. New binary starts, announces new `stream_port`
8. Device reappears in Spotify app (ZeroConf re-registers)
9. User presses Play in Spotify app ‚Üí new Spirc session connects
10. `start` event fires ‚Üí LMS issues `playlist play spotify://connect-<ts>`
11. `canDirectStream` returns new HTTP URL with new port
12. Stream resumes

**This scenario mostly works** but has a timing gap: between `alive=0` detection and binary restart, the device is invisible in Spotify (no ZeroConf). The 5s `_streamAlivePoll` interval means worst-case 5s invisibility. This is acceptable (D-05 says UX follows from technical approach).

**Scenario 3: Session death after long idle (no playback, daemon running)**

1. Binary running, Spirc session expires silently after hours of idle
2. `spirc_active` stays true (no event fires to clear it)
3. `_streamAlivePoll` sees `alive=true` (binary process still running) ‚Üí no restart
4. Device may still appear in Spotify app (ZeroConf still active)
5. User presses Play ‚Üí Spotify sends play command to Spirc
6. If Spirc is expired: `spirc_task` completes with error ‚Üí triggers reconnect loop in binary
7. Binary reconnects (up to 5 in 60s) ‚Üí `spirc_active` set false, then true again after SessionConnected
8. New `TrackChanged` fires ‚Üí `notify("start")` ‚Üí LMS responds

**This scenario is handled by the binary's reconnect loop** (lines 1048-1074 in connect.rs). The binary reconnects using `last_credentials` (cached). However: after reconnection, `spirc_active` temporarily drops to false. The `/stream` endpoint returns 503 during reconnect. If LMS has an active stream connection, it drops and the player stops.

**Key gap:** There is no mechanism to notify LMS that the stream went temporarily unavailable and then recovered. The binary reconnects silently. The Perl side only restarts the daemon if the process dies (`alive=0`). If the binary reconnects internally without dying, the Perl side sees nothing.

### Binary-Side Spirc Reconnect: Already Implemented [VERIFIED: codebase]

`run_connect()` (connect.rs lines 1048-1074) has a Spirc reconnect loop:
- When `spirc_task` completes unexpectedly ‚Üí set `spirc_active=false`, clear session
- Reconnect with `last_credentials` using new `Spirc::new()`
- Rate limit: max 5 reconnects in 60 seconds
- On success: new Spirc in `spirc_handle`, new `spirc_task` spawned, `connecting=false`
- `spirc_active` becomes true again when `SessionConnected` or `Playing` fires

**What's missing:** `spirc_active` is not reset to false when the stream endpoint is being used by LMS. After Spirc reconnects, `spirc_active` becomes true again but the binary doesn't send a new `start` event to LMS ‚Äî there's no active track. The user must press Play in the Spotify app again to trigger playback.

### Session-Handover Strategy Recommendation

**For source-switch (Scenario 1 without D-08):** Already works ‚Äî Spirc stays alive. The only issue is that D-08 kills the binary when Browse starts. Consider softening D-08: instead of killing the daemon, just mark it as "paused" and don't stream. But D-08 was a deliberate design decision to avoid confusion. Keep D-08 as-is.

**For D-08 scenario (binary killed):** Already handled by `_streamAlivePoll` (5s restart). No new code needed. Verify that after restart, ZeroConf re-registration happens within the expected timeout.

**For idle session expiry (Scenario 3):** The binary's internal reconnect loop handles this. The Perl side only needs to handle the case where the binary exhausts its reconnect budget (5 in 60s) and dies. This is already handled by `_streamAlivePoll`.

**Recommended additional fix:** After the binary reconnects Spirc (internal reconnect loop), it should optionally notify LMS via a new wire command `reconnected` or `ready`. LMS can then re-issue `playlist play spotify://connect-<ts>` to start streaming again. This requires a small addition to `connect.rs::run_connect()` to notify on successful reconnect.

**Alternative (simpler):** No Perl changes. Rely on the user pressing Play in the Spotify app after session recovery. The device will reappear in the Spotify app after Spirc reconnects (ZeroConf still running). The user experience: device temporarily disappears, then reappears. The 5-second `_streamAlivePoll` watchdog ensures rapid recovery if the binary died.

## Common Pitfalls

### Pitfall 1: relay_active Flag Stuck True After Crash
**What goes wrong:** The `relay_active` AtomicBool in `http_stream_server` prevents concurrent streams. If the relay task panics, `RelayGuard::drop()` may not run, leaving `relay_active=true` permanently. All subsequent `/stream` requests return 503.
**Why it happens:** Panics in tokio tasks don't propagate the panic to the spawning context; the task just gets cancelled. The `Drop` trait IS called on local variables during unwind, so `RelayGuard` should work. However, task cancellation via `tokio::spawn` doesn't trigger Drop ‚Äî it depends on the drop of the JoinHandle future.
**How to avoid:** Add explicit `relay_active.store(false)` in the main loop cleanup path. Verify `RelayGuard` actually runs on panic by testing with a deliberate panic.
**Warning signs:** `/stream` returns 503 when `spirc_active=true` ‚Äî binary logs show "relay-busy 503".

### Pitfall 2: LMS Enters Persistent Enhancement Path for Streaming Body
**What goes wrong:** `HTTP.pm::response()` enters the "Persistent" path when no Content-Length is present. This path tries to make ranged reconnection requests against an infinite stream, causing disconnection.
**Why it happens:** LMS was designed for finite audio files. Infinite streams (radio) use `isLive(1)` to signal this, but the `!$self->contentLength` check in `response()` still triggers Persistent mode.
**How to avoid:** Verify `canEnhanceHTTP` returns 0 for Connect proxy connections. If it returns non-zero, LMS enters the enhanced path. Override `requestString()` in `ProtocolHandler.pm` to suppress Range headers for Connect proxy connections.
**Warning signs:** LMS logs show "Using Persistent service" for the Connect stream URL.

### Pitfall 3: Two Spirc watcher tasks both compete on spirc_active
**What goes wrong:** `run_connect()` spawns TWO tasks that watch `player_event_channel`. One is the LMS dispatcher (handles start/stop/volume). The other is the `spirc_active` watcher. Both read from the same channel. In librespot, `get_player_event_channel()` returns a new receiver each time ‚Äî so two separate receivers each get a copy of events. [VERIFIED: librespot uses a broadcast-style channel]
**Why it happens:** Each `get_player_event_channel()` call creates a new subscription. If SessionConnected goes to the spirc_active watcher but not the LMS dispatcher (because order of registration matters in broadcast channels), the LMS dispatcher may miss the SessionConnected event.
**How to avoid:** Verify that both tasks receive all events. librespot 0.8's `Player::get_player_event_channel()` uses a `tokio::sync::broadcast` channel ‚Äî all subscribers receive all events.
**Warning signs:** `spirc_active` never becomes true despite events being dispatched.

### Pitfall 4: D-08 Mutual Exclusion Kills Daemon Before Resume
**What goes wrong:** When the user switches to Browse after Connect, D-08 kills the daemon. When they return to Connect, `_streamAlivePoll` restarts it (5s delay). But between kill and restart, ZeroConf is down ‚Äî Spotify app may not show the device.
**Why it happens:** Design decision ‚Äî D-08 stops Connect to prevent confusion. But the 5s window is visible.
**How to avoid:** Acceptable per D-03/D-05. Document expected behavior. Spotify app reconnects within 10-15s after ZeroConf re-registration.
**Warning signs:** Device disappears from Spotify app for 5-10 seconds after switching to Browse.

### Pitfall 5: syncChanged Fires Multiple Times for One Sync Operation
**What goes wrong:** Adding one player to a sync group may fire the `sync` event multiple times (once per player involved). `syncChanged` calls `stopForSync` on ALL daemons each time, causing extra restarts.
**Why it happens:** LMS's sync implementation fires one event per sync operation, but multiple operations can be batched. The 0.1s timer in `syncChanged` coalesces rapid events.
**How to avoid:** The 0.1s `killTimers`/`setTimer` debounce in `DaemonManager.pm::init()` syncChanged subscriber already handles this. No additional action needed.

## Session Handover: Recommended Strategy

Based on the analysis above, the recommended strategy (Claude's discretion per D-04):

**Primary: Binary-internal reconnect (already implemented)** ‚Äî When Spirc dies, the binary reconnects using cached credentials. The device briefly disappears from Spotify (up to ~30s during reconnect) and reappears. The user must press Play again in the Spotify app. No Perl changes needed for this path.

**Secondary: Perl-side watchdog restart** ‚Äî When `spirc_task` exhausts its reconnect budget and the binary dies, `_streamAlivePoll` (5s) detects `alive=0` and calls `$helper->start`. This covers the case where reconnect fails completely.

**Enhancement (optional):** Add a `ready` notify from the binary after successful Spirc reconnect. Connect.pm handler re-issues `playlist play` to resume streaming without user intervention. This improves UX from "visible interruption + manual replay" to "transparent recovery." Requires small addition in `connect.rs::run_connect()` reconnect success branch.

**NOT recommended:** Killing and restarting the daemon from the Perl side on session death. The binary already handles reconnect; killing it adds unnecessary ZeroConf disruption.

**Idle session expiry:** The binary's reconnect loop handles this transparently. If the Spirc session silently expires (no event fires), the next Spotify app interaction triggers a `spirc_task` completion which triggers reconnect.

## Sync-Group Proxy Fix: Recommended Approach

Based on analysis (Claude's discretion per D-01):

**Step 1: Diagnose the actual disconnect cause.**

Check LMS log for "Using Persistent service" vs "Using Buffered service" when the Connect proxy URL is opened. Add a custom `canEnhanceHTTP` override in `ProtocolHandler.pm` that returns 0 explicitly for Connect proxy URLs (those matching `:\d+/stream`). This prevents LMS from entering the Persistent reconnect path.

**Step 2: Override `requestString` to suppress Range header.**

The `Range: bytes=0-` header may cause the binary's hyper server to handle the connection differently. Suppressing it makes the request semantics cleaner for an infinite stream. This also prevents hyper from sending `206 Partial Content` instead of `200 OK` (though both should work).

**Step 3: Verify `relay_active` flag behavior.**

Log when `relay_active` is set/cleared. If a previous connection left it stuck, all subsequent proxy attempts fail with 503. Add a timeout-based recovery in the binary to force-clear `relay_active` after 60s.

**What NOT to do:**
- Do not add `Content-Length` header (it's an infinite stream; no valid value exists)
- Do not switch to HTTP/1.0 explicit version in the binary (hyper already handles this automatically)
- Do not rewrite the HTTP server (the Spotty-NG and SpotOn servers are already identical)

## Open Questions (RESOLVED)

1. **Why does Spotty-NG proxy work but SpotOn proxy doesn't?**
   - What we know: Both binaries have identical hyper HTTP servers. Both ProtocolHandlers use same proxy pattern.
   - What's unclear: Whether Spotty-NG actually works in proxy mode, or if it's untested. The CONTEXT.md says "Spotty-NG binary as working reference" but doesn't confirm proxy mode was tested.
   - Recommendation: Test Spotty-NG in sync-group mode first to confirm it works. If it doesn't work either, the proxy issue is in LMS's handling of infinite streams, not the binary.
   - **RESOLVED:** The root cause is in LMS's HTTP.pm response() method, not the binary. When no Content-Length is present, LMS enters the "Persistent service" path (line 107: `!$self->contentLength`) which tries ranged reconnections against an infinite stream. Whether Spotty-NG "works" is secondary ‚Äî the fix is to override `canEnhanceHTTP` (return 0 for stream URLs) and `requestString` (suppress Range header) in ProtocolHandler.pm. Both overrides prevent LMS from entering the Persistent path regardless of binary behavior. If Spotty-NG does work, it would be because LMS's `useEnhancedHTTP` pref happens to be 0 in that environment ‚Äî our fix makes this explicit.

2. **Does `relay_active` ever get stuck true?**
   - What we know: `RelayGuard::drop()` should clear it on task exit. But tokio task cancellation behavior differs from panic unwind.
   - What's unclear: Can the relay task be cancelled by the tokio runtime without running Drop?
   - Recommendation: Add explicit `relay_active=false` logging. Monitor for 503 responses when `spirc_active=true`.
   - **RESOLVED:** Yes, tokio task cancellation (JoinHandle dropped without await) can skip Drop on task-local variables. Plan 02 adds a timeout health-check inside http_stream_server: if relay_active has been true for >60s with no data delivered, it is force-cleared. This is self-healing ‚Äî no manual intervention needed. The approach keeps relay_active scoped within http_stream_server (per PATTERNS.md recommendation) rather than threading Arc into run_connect scope.

3. **Session-handover: is the binary's reconnect loop actually triggered by source switch?**
   - What we know: D-08 kills the binary when Browse starts. Binary's reconnect loop runs only while binary is alive.
   - What's unclear: Is there a scenario where the binary stays alive but Spirc dies silently?
   - Recommendation: Add `log::debug!("[spoton] spirc_task ended")` to the reconnect branch for visibility.
   - **RESOLVED:** Two scenarios exist: (a) D-08 kills the binary entirely ‚Äî handled by existing `_streamAlivePoll` 5s watchdog which restarts the binary. The reconnect loop is irrelevant here because the process is dead. (b) Binary stays alive but Spirc session expires after idle ‚Äî the binary's reconnect loop (lines 1048-1074) handles this, reconnecting with cached credentials. Plan 02 adds a `ready` notify after successful reconnect so LMS can re-issue playlist play automatically. Both scenarios are now covered.
