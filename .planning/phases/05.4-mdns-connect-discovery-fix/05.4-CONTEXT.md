# Phase 05.4: mDNS Connect Discovery Fix - Context

**Gathered:** 2026-06-02
**Status:** Ready for planning

<domain>
## Phase Boundary

mDNS/ZeroConf-Discovery für Spotify Connect funktioniert zuverlässig — Geräte erscheinen in der Spotify-App via mDNS (nicht nur über „In anderen Netzwerken"), Verbindungen via Discovery starten Audio, Spirc-Registration ist stabil (keine wiederholten ready-Events), und der Crash-Loop-Schutz (`disableDiscovery`) setzt sich nach einem Cooldown automatisch zurück. Per-Player Discovery-Toggle in Settings UI.

Known Issues sind aus Code-Analyse identifiziert, nicht aus Livebetrieb — Phase beinhaltet sowohl Diagnose als auch Fixes.

</domain>

<decisions>
## Implementation Decisions

### Cooldown-Strategie
- **D-01:** `disableDiscovery` wird automatisch zurückgesetzt — sowohl über einen Timer-Cooldown ALS AUCH bei jedem LMS-Neustart. Doppelte Absicherung: Timer für automatische Recovery, LMS-Restart für sofortige manuelle Recovery.
- **D-02:** Cooldown-Timer-Dauer — Claude entscheidet. Researcher ermittelt typische mDNS-Recovery-Zeiten und wählt passend. Orientierungswert: 15-60 Minuten.
- **D-03:** Nach Cooldown-Ablauf wird `disableDiscovery` auf 0 gesetzt und Daemons werden neu gestartet (mit Discovery enabled). Bei erneutem Crash-Loop greift der Schutz wieder → neuer Cooldown-Zyklus.

### Discovery-Toggle UX
- **D-04:** Per-Player `disableDiscovery` als separate Checkbox in Settings UI — „mDNS Discovery" unter dem Connect-Toggle pro Player. Ermöglicht Connect OHNE Discovery (Gerät nur via „In anderen Netzwerken" erreichbar).
- **D-05:** Crash-Loop-Schutz überschreibt die per-Player Checkbox. Wenn Crash-Loop-Schutz Discovery deaktiviert, bleibt sie deaktiviert auch wenn die Checkbox an ist. Nach Cooldown oder LMS-Restart wird der Checkbox-Zustand wiederhergestellt. Schutz hat Priorität über manuelle Einstellung.

### Diagnose-Strategie
- **D-06:** Paralleler Ansatz — Code-Analyse für offensichtliche Bugs (disableDiscovery-Reset, Device-ID-Generierung, doppelte mDNS-Registrierung) UND UAT-Diagnose-Session für unklare Issues (Discovery-Connection, ready-Events). Fixes für klare Bugs sofort, für unklare nach Diagnose.
- **D-07:** Known Issues sind aus Code-Analyse, NICHT aus Livebetrieb. Planner muss eine Diagnose-Session als Task einplanen bevor finale Fixes für SC #1-3 geplant werden können.

### Discovery vs. Spirc Architektur
- **D-08:** Claude entscheidet ob Discovery Reconnect-Pfad bleibt (Status quo) oder architektonisch geändert wird. Researcher analysiert librespot-Internals: ob Spirc einen eigenen mDNS-Service laufen hat, ob Discovery und Spirc sich in die Quere kommen (doppelte mDNS-Announcements, Device-ID-Konflikte).
- **D-09:** Fix-Aggressivität basiert auf tatsächlichem Befund — kleiner Bug = kleiner Fix, architektonisches Problem = größerer Eingriff. Claude wägt ab.

### Claude's Discretion
- Timer-Cooldown-Dauer (D-02): Zwischen 15 und 60 Minuten, basierend auf typischen mDNS-Recovery-Mustern
- Discovery/Spirc-Architektur (D-08): Status quo + Fix vs. Discovery-First vs. anderer Ansatz — basierend auf librespot-Internals-Analyse
- Fix-Aggressivität (D-09): Minimalinvasiv vs. moderate Refaktoren — basierend auf Diagnose-Befund
- Spirc-Reconnect-Rate-Limiting: `connect.rs:1086-1098` erlaubt 5 Reconnects in 60s — möglicherweise zu aggressiv für mDNS-Stabilität

</decisions>

## Existing Code Insights

### Reusable Assets
- `Daemon.pm::_checkStartTimes()` (Zeile 191-213) — Crash-Loop-Detection-Logik. Muss um Auto-Reset-Timer erweitert werden.
- `DaemonManager.pm::initHelpers()` (Zeile 83-166) — Bereits 60s-Watchdog-Timer. Kann für Discovery-Cooldown-Check erweitert werden.
- `connect.rs` Discovery-Builder (Zeile 982-1001) — Existierender Discovery-Setup mit FNV-1a Device-ID. Basis für mDNS-Fixes.
- `Settings.pm` — `enableSpotifyConnect` per-Player Toggle als Pattern für neues `disableDiscovery` Checkbox.

### Established Patterns
- Per-Player Prefs: `$prefs->client($client)->get(...)` — bereits etabliert für `enableSpotifyConnect`, `activeAccount`.
- Pref-Change-Callback: Spotty-NG nutzt `disableDiscovery` Pref-Change → DaemonManager reinit. SpotOn hat noch keinen.
- Crash-Backoff: Sliding-Window (3 Starts in 5min) → globales Flag. Muss auf per-Player + Auto-Reset umgebaut werden.
- Timer: `Slim::Utils::Timers::setTimer()` — für Cooldown-Timer direkt nutzbar.

### Integration Points
- `Daemon.pm::_checkStartTimes()` → Cooldown-Timer statt permanentem `disableDiscovery=1`
- `Daemon.pm::start()` → Per-Player `disableDiscovery` statt global
- `DaemonManager.pm::initHelpers()` → Cooldown-Reset-Check im Watchdog
- `Settings.pm` → Neues per-Player `disableDiscovery` Checkbox
- `basic.html` → UI-Element für Discovery-Toggle
- `connect.rs` → Discovery/Spirc-Lifecycle je nach Diagnose-Befund

</code_context>

<specifics>
## Specific Ideas

- Crash-Loop-Schutz wird von global auf per-Player umgebaut — ein Player mit Discovery-Problemen soll nicht alle anderen blockieren
- Cooldown-Timer nutzt `Slim::Utils::Timers::setTimer()` — nach Ablauf setzt Timer `disableDiscovery` zurück und ruft `initHelpers()` auf
- Bei LMS-Start: `initPlugin()` oder `DaemonManager::init()` setzt alle `disableDiscovery` Prefs zurück (globale + per-Player)
- UAT-Diagnose-Session: Daemon mit Discovery starten, Spotify-App öffnen, Gerät suchen, verbinden, Logs auswerten — als konkreter Planungs-Task
- FNV-1a Device-ID in `connect.rs:988-991` muss validiert werden — möglicherweise Kollision mit Spirc's interner Device-ID

</specifics>
