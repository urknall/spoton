## Zusammenfassung

Phase 05.4 behebt drei zusammenhängende Probleme, die zuverlässige mDNS-Discovery für Spotify Connect verhindern.

**Hauptbefund (VERIFIED durch Quellcode-Analyse):** In `connect.rs` wird `SessionConfig::default()` aufgerufen, das bei jedem Start eine neue zufällige UUID als `device_id` generiert (`librespot-core-0.8.0/src/config.rs:36`). Gleichzeitig berechnet die Discovery-Instanz eine FNV-1a-Hash der `cache_dir` als ihre `device_id`. Das sind immer zwei verschiedene Device-IDs. Spirc registriert sich bei der Spotify-Cloud mit der UUID, Discovery annonciert per mDNS mit dem FNV-1a-Hash — Spotify sieht zwei verschiedene Geräte. Wenn der Nutzer via Discovery-Gerät verbindet, matcht die credentials-Antwort die Discovery-Device-ID, aber Spirc kennt diese ID nicht. Das erklärt sowohl das Discovery-Connection-Problem als auch möglicherweise die wiederholten `ready`-Events (Spirc verliert die Session und versucht Reconnect).

**Sekundärer Bug:** `_checkStartTimes()` in `Daemon.pm` setzt `disableDiscovery` permanent auf 1 ohne je zurückzusetzen — einmal ausgelöst, bleibt Discovery dauerhaft deaktiviert bis manueller Eingriff. Das ist global (nicht per-Player) und durch kein LMS-Restart zurückgesetzt, da `disableDiscovery` kein Default-Wert in `$prefs->init()` hat und persistiert.

**Empfehlung:** Fix-Reihenfolge: (1) Device-ID-Mismatch beheben (architektonisches Problem, klarer Fix), (2) Cooldown-Auto-Reset implementieren, (3) per-Player Discovery-Toggle in UI. UAT-Session nach Fix (1) um Spirc-Stabilität zu verifizieren.

### Pattern 1: Device-ID-Synchronisierung Discovery↔Spirc

**Was:** `SessionConfig::default()` erzeugt bei jedem Aufruf eine neue zufällige UUID als `device_id`. Um Discovery und Spirc auf dieselbe ID zu bringen, muss `session_config.device_id` explizit auf den FNV-1a-Hash der `cache_dir` gesetzt werden — genau dieselbe Berechnung wie in der Discovery-Builder-Instanz.

**Wenn zu nutzen:** In `run_connect()` vor `Session::new(session_config, ...)`.

```rust
// Source: librespot-core-0.8.0/src/config.rs:27,36 (analysiert)
// VORHER (Bug): session_config.device_id = UUID::new_v4() (zufällig)
let session_config = SessionConfig::default();

// NACHHER (Fix): session_config.device_id = FNV-1a(cache_dir) — gleiche ID wie Discovery
let device_id_shared = {
    let mut h: u64 = 14695981039346656037;
    for b in cache_dir.as_bytes() { h ^= *b as u64; h = h.wrapping_mul(1099511628211); }
    format!("{:016x}", h)
};
let mut session_config = SessionConfig::default();
session_config.device_id = device_id_shared.clone();
let session = Session::new(session_config.clone(), Some(cache.clone()));

// Discovery verwendet dann dieselbe device_id_shared
let discovery = if !disable_discovery {
    match librespot_discovery::Discovery::builder(device_id_shared, KEYMASTER_CLIENT_ID.to_string())
        .name(device_name.to_string())
        .device_type(DeviceType::Speaker)
        .launch()
    {
        Ok(d) => Some(d),
        Err(_) => None,
    }
} else { None };
```

**Wichtig für Reconnect:** Bei jedem `Session::new()` in der Reconnect-Branch (Zeile ~1035 in connect.rs) muss ebenfalls `session_config.device_id = device_id_shared` gesetzt werden. Die `session_config` muss also `device_id_shared` als Variable zugänglich haben — sie darf nicht mehr im `if !disable_discovery` Block lokal sein, sondern muss vor dem Event-Loop berechnet werden.

### Pattern 2: Per-Player Crash-Loop-Schutz mit Cooldown-Timer

**Was:** `_checkStartTimes()` setzt globales `disableDiscovery`-Flag permanent. Fix: (a) per-Player statt global, (b) nach Cooldown automatisch zurücksetzen via `Slim::Utils::Timers`, (c) bei LMS-Start alle `disableDiscovery`-Prefs zurücksetzen.

**Cooldown-Dauer-Empfehlung (D-02):** 30 Minuten. mDNS-Recovery nach Netzwerkproblemen braucht typisch 1–5 Minuten. 30 Minuten gibt genug Abstand um transiente Netzwerkfehler zu absorbieren, ohne dauerhaft offline zu bleiben.

```perl
# Source: Daemon.pm Zeile 191-213 (analysiert) + DaemonManager.pm Timers-Pattern
use constant DISCOVERY_COOLDOWN_SECONDS => 1800;  # 30 Minuten (D-02)

sub _checkStartTimes {
    my $self = shift;
    my $client = Slim::Player::Client::getClient($self->mac);

    if ( scalar @{$self->_startTimes} >= MAX_FAILURES_BEFORE_DISABLE_DISCOVERY ) {
        splice @{$self->_startTimes}, 0,
               @{$self->_startTimes} - MAX_FAILURES_BEFORE_DISABLE_DISCOVERY;

        # Per-Player disableDiscovery (D-05) — nur wenn Checkbox-Zustand es erlaubt
        my $alreadyDisabled = $client
            ? ($prefs->client($client)->get('disableDiscovery') || 0)
            : $prefs->get('disableDiscovery');

        if ( time() - $self->_startTimes->[0] < MAX_INTERVAL_BEFORE_DISABLE_DISCOVERY
            && !$alreadyDisabled
        ) {
            $log->warn("SpotOn daemon crashed too often - disabling discovery for 30 min");

            if ($client) {
                # Per-Player: "crash-loop-active" Flag — getrennt von User-Checkbox
                $prefs->client($client)->set('discoveryDisabledByCrashLoop', 1);
            } else {
                $prefs->set('disableDiscovery', 1);
            }

            # Cooldown-Timer: Auto-Reset nach DISCOVERY_COOLDOWN_SECONDS
            require Plugins::SpotOn::Connect::DaemonManager;
            Slim::Utils::Timers::killTimers($self->mac, \&_resetDiscoveryCooldown);
            Slim::Utils::Timers::setTimer(
                $self->mac,
                Time::HiRes::time() + DISCOVERY_COOLDOWN_SECONDS,
                \&_resetDiscoveryCooldown,
                $self->mac,   # timer arg: client MAC
            );
        }
    }
    push @{$self->_startTimes}, time();
}
```

**Architektur-Überlegung:** Ob das Cooldown-Flag als separates `discoveryDisabledByCrashLoop` per-Player-Pref oder als Override-Logic im `start()`-Pfad implementiert wird, ist Planner-Entscheidung (D-09). Das separate Flag ist klarer trennbar von der User-Checkbox (D-05).

### Anti-Patterns

- **Anti-Pattern:** `SessionConfig::default()` ohne explizites `device_id`-Setzen vor `Session::new()` — erzeugt immer neue UUID, Discovery-Mismatch.
- **Anti-Pattern:** `disableDiscovery` global setzen ohne per-Player-Scope — ein crashender Player deaktiviert Discovery für alle.
- **Anti-Pattern:** Permanentes `disableDiscovery = 1` ohne Auto-Reset — Discovery bleibt dauerhaft aus.
- **Anti-Pattern:** `disableDiscovery` pref ohne Default in `$prefs->init()` — bei Erststart ist Wert undefined, persistiert dann aber als 1.

## Common Pitfalls

### Pitfall 1: Spirc-Reconnect verwendet neue Session mit neuer UUID

**Was schiefläuft:** In der Reconnect-Branch (connect.rs Zeile ~1034) wird `Session::new(session_config.clone(), ...)` aufgerufen. Wenn `session_config` die Standard-UUID hat, erzeugt jede neue Session eine neue Device-ID. Spirc registriert sich mit einer anderen ID als Discovery.

**Warum es passiert:** `session_config.clone()` klont die `SessionConfig` inkl. der zufälligen UUID, die beim ersten `SessionConfig::default()` erstellt wurde. Das ist technisch konsistent — aber wenn `SessionConfig::default()` erneut aufgerufen wird (was aktuell nicht der Fall ist), wäre die UUID wieder neu.

**Wie vermeiden:** `device_id_shared` als Variable vor dem Event-Loop berechnen und `session_config.device_id = device_id_shared.clone()` setzen. Dann wird die gleiche ID beim Reconnect weiterverwendet.

**Warnsignal:** Logs zeigen `[spoton] Spirc reconnected — notified LMS 'ready'` kurz nach Start, oder `ready`-Events im 30s-Intervall.

### Pitfall 2: Discovery-Instanz und Spirc registrieren unterschiedliche ZeroConf-Namen

**Was schiefläuft:** Wenn Discovery und Spirc beide mDNS-Announcements machen würden, würden zwei `_spotify-connect._tcp`-Services annonciert. Aktuell macht nur Discovery mDNS. Spirc selbst macht kein mDNS — er registriert sich nur bei der Spotify-Cloud.

**Warum relevant:** Das erklärt, warum mDNS-Discovery das Gerät sichtbar macht, aber die Verbindung fehlschlägt: Die Cloud-Registrierung (Spirc mit UUID) passt nicht zur mDNS-Registrierung (Discovery mit FNV-1a). [VERIFIED: librespot-core-0.8.0/src/config.rs, librespot-connect-0.8.0/src/state.rs]

**Warnsignal:** Gerät in Spotify-App sichtbar via mDNS, aber Verbindung startet kein Audio.

### Pitfall 3: Crash-Loop-Schutz feuert bei `stopForSync()` fälschlicherweise

**Was schiefläuft:** `stopForSync()` ruft `_startTimes([])` auf (Zeile 263) — setzt Start-Zeiten korrekt zurück. Aber `_checkStreamStartTimes()` und `_checkStartTimes()` werden bei jedem `start()` aufgerufen, auch nach einem normalen `stopForSync`-Neustart.

**Prüfen in UAT:** Nach Sync-Gruppe-Änderung: Werden Crash-Loop-Zähler korrekt zurückgesetzt? [ASSUMED — `_startTimes([])` in stopForSync sieht korrekt aus, muss in UAT bestätigt werden]

### Pitfall 4: `disableDiscovery` Pref fehlt in `$prefs->init()`

**Was schiefläuft:** `Plugin.pm::initPlugin()` initialisiert Prefs ohne `disableDiscovery`. Das Pref wird erst gesetzt wenn Crash-Loop auslöst. Beim ersten Lesen (`$prefs->get('disableDiscovery')`) gibt `undef` zurück, was falsy ist — korrekt. Aber nach Recovery-Reset auf 0 kann es zu unerwartetem Verhalten kommen wenn Code `defined` statt Truthiness prüft.

**Wie vermeiden:** `disableDiscovery => 0` zu `$prefs->init()` in Plugin.pm hinzufügen. [CITED: Spotty-NG Plugin.pm Zeile 71]

### Pitfall 5: Cooldown-Timer und DaemonManager-Neustart race condition

**Was schiefläuft:** Nach Cooldown-Ablauf setzt `_resetDiscoveryCooldown()` das Flag zurück und ruft `DaemonManager::initHelpers()` auf. Wenn gleichzeitig der 60s-Watchdog läuft, könnte `initHelpers()` doppelt ausgeführt werden.

**Wie vermeiden:** `Slim::Utils::Timers::killTimers($class, \&initHelpers)` vor dem Aufruf, dann `setTimer` für initHelpers. DaemonManager.pm macht das bereits so (Zeile 86).

### Pitfall 6: Spirc-Reconnect-Rate-Limit zu aggressiv (5 in 60s)

**Was schiefläuft:** `connect.rs:1085-1097` erlaubt max. 5 Reconnects in 60 Sekunden. Nach dem fünften Reconnect bricht die Schleife ab mit `"Spirc shut down too often. Not reconnecting."`. Das könnte die wiederholten `ready`-Events erklären: Spirc startet, sendet `ready`, fällt kurz ab, sendet `ready` erneut — nach 5x Abbruch stirbt der Daemon und Perl-Watchdog startet neu.

**Wie vermeiden:** Nach Device-ID-Fix prüfen ob das Rate-Limit noch triggert (UAT-Session D-07). Wenn ja, Rate-Limit erhöhen (z.B. auf 10 in 120s) oder implementieren einen längeren Backoff.

// Beide müssen dieselbe device_id_shared bekommen.

pub async fn run_connect(
    cache_dir: &str,
    device_name: &str,
    // ... übrige Parameter ...
    disable_discovery: bool,

    // 1. Cache + Credentials (unverändert)
    let cache = Cache::new(Some(cache_dir), None::<&str>, None::<&str>, None)?;
    let credentials = cache.credentials().ok_or("No cached credentials")?;

    // (session_config.clone() enthält bereits device_id_shared)
    // ...

## State of the Art

| Alter Ansatz | Aktueller Ansatz | Geändert | Impact |
|--------------|-----------------|----------|--------|
| Statische Device-ID via `--device-id` CLI-Flag | FNV-1a Hash der cache_dir (intern) | Phase 05 | FNV-1a ist stabil und deterministisch, aber wurde nicht mit SessionConfig synchronisiert |
| Globales `disableDiscovery` Pref (Spotty-NG Pattern) | Per-Player + Cooldown-Reset | Phase 05.4 | Verhindert Cross-Player-Contamination |
| Spirc-Reconnect mit neuer Session (zufällige UUID) | Session mit fixer device_id_shared | Phase 05.4 | Löst Discovery↔Spirc-Mismatch |

**Veraltet/Problematisch:**
- `SessionConfig::default()` ohne explizites `device_id` — erzeugt immer neue UUID. Fix: vor `Session::new()` explizit setzen.
- Globales `disableDiscovery` ohne Auto-Reset — funktioniert als Notfall-Schutz, aber Recovery erfordert manuellen Eingriff.

## Diagnose-Strategie (D-06, D-07)

Die Phase benötigt eine UAT-Diagnose-Session als ersten Task (D-07). Die Code-Analyse hat den architektonischen Bug identifiziert (Device-ID-Mismatch), aber zwei Issues bleiben zu validieren:

1. **Repeated `ready`-Events (~30s Interval):** Ob diese von der Device-ID-Inkonsistenz stammen oder von einem separaten Spirc-Reconnect-Problem. Nach Device-ID-Fix beobachten.

- `ready`-Events zählen und Interval messen
- Ergebnis dokumentiert → bestimmt ob weitere Fixes in connect.rs nötig (D-09)

## Offene Fragen

1. **Repeated `ready`-Events: Ursache Device-ID-Mismatch oder Spirc-intern?**
   - Was wir wissen: Code-Analyse zeigt Device-ID-Mismatch als wahrscheinliche Ursache
   - Was unklar: Ob Spirc nach Discovery-Reconnect mit falscher Device-ID eine neue Session erzwingen, die zu wiederholten `SessionConnected`-Events führt
   - Empfehlung: Device-ID-Fix zuerst deployen, dann in UAT-Session `ready`-Events beobachten

2. **Port-Kollision bei Discovery-HTTP-Server und Binary-HTTP-Server?**
   - Was wir wissen: Binary bindet Port 0 (dynamisch) für den Streaming-HTTP-Server
   - Was unklar: `librespot-discovery` bindet ebenfalls einen HTTP-Port (für ZeroConf-Handshake). Bei mehreren Daemons könnte es Konflikte geben
   - Empfehlung: In UAT prüfen; `librespot-discovery` hat eigenen Port-Mechanismus (CON-16 gilt nur für Stream-Port) [ASSUMED]

3. **Spotty-NG's `canDiscovery()` Funktion — braucht SpotOn ein Äquivalent?**
   - Was wir wissen: Spotty-NG hat `canDiscovery()` in Plugin.pm (gibt immer `1` zurück) und nutzt es in DaemonManager.pm als Guard
   - Was unklar: SpotOn hat diese Funktion nicht — `disableDiscovery` Pref-Check reicht als Ersatz
   - Empfehlung: Kein `canDiscovery()` nötig; direkte Pref-Checks sind idiomatischer
