# Phase 05.4: Code Review Report

**Reviewed:** 2026-06-02T12:00:00Z
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

This phase implements three distinct fixes: (1) synchronizing `device_id` between Discovery (mDNS) and Spirc (Spotify Cloud) in the Rust binary, (2) upgrading crash-loop protection from global to per-player scope with a 30-minute cooldown timer, and (3) adding a per-player Discovery toggle to the Settings UI.

The core architectural fix (device_id synchronization) is correct and well-reasoned. The credential blob decryption in `Credentials::with_blob` uses `device_id` as the HMAC key material (librespot-core auth.rs:101), so the split-brain was indeed causing authentication failures. The fix is verified by tracing through the librespot source.

The Perl-side changes are mostly sound but contain one blocker: `Daemon.pm` calls `Slim::Utils::Timers` without importing it, which will produce a fatal runtime error the first time `_checkStartTimes()` triggers a crash-loop. There are also three warnings around logic correctness and security.

## Critical Issues

### CR-01: `Slim::Utils::Timers` not imported in `Daemon.pm` — fatal runtime crash on crash-loop trigger

**File:** `Plugins/SpotOn/Connect/Daemon.pm:237-243`
**Issue:** `_checkStartTimes()` calls `Slim::Utils::Timers::killTimers(...)` and `Slim::Utils::Timers::setTimer(...)` at lines 237–242, but `Slim::Utils::Timers` is never imported via `use` in `Daemon.pm`. Under `use strict`, calling a fully-qualified package function whose module is not yet loaded produces an "Undefined subroutine" fatal error. The module may happen to be loaded transitively (DaemonManager.pm pulls it in at line 9), but that is an implicit dependency — if load order changes or the module is loaded lazily, the timer call will die at the worst possible moment: exactly when a crash-loop is detected and the cooldown timer needs to be set. The entire crash-loop protection feature is dead until this import is added.

`Time::HiRes` is also used at line 240 (`Time::HiRes::time()`) without an explicit import, for the same reason.

**Fix:**
```perl
# Add to the use block at the top of Daemon.pm (after line 12):
use Slim::Utils::Timers;
use Time::HiRes;
```

## Warnings

### WR-01: LMS-start reset of global `disableDiscovery` is unconditional — wipes user's manual global setting

**File:** `Plugins/SpotOn/Connect/DaemonManager.pm:101-106`
**Issue:** The LMS-start reset at lines 100–106 resets the global `disableDiscovery` pref to 0 unconditionally whenever it is truthy. The intent (D-01) is to clear the flag that was set by the crash-loop fallback path (when `$client` was undef in `_checkStartTimes()`). However, the global `disableDiscovery` pref also serves as a user-controlled global fallback (see `Plugin.pm:49` comment "global discovery toggle"). If a user has deliberately set the global flag to 1 via a mechanism other than crash-loop protection, a LMS restart will silently wipe their preference.

The per-player `discoveryDisabledByCrashLoop` is correctly distinguished from `disableDiscovery` (user checkbox), but the global `disableDiscovery` conflates both uses. A separate `globalDiscoveryDisabledByCrashLoop` flag, or a comment noting that the global pref is crash-loop-only, would prevent unexpected behavior.

**Fix:** Introduce a companion global flag `disableDiscoveryByCrashLoop` that is set only by the crash-loop fallback, and clear only that one at LMS start. Leave the user-facing global `disableDiscovery` pref untouched. Or, at minimum, add a comment documenting that the global `disableDiscovery` pref is treated as crash-loop-only and will be reset on every restart:
```perl
# D-01 LMS-start reset — only clear if set by crash-loop fallback.
# NOTE: the global disableDiscovery pref is only set via the crash-loop
# fallback (no $client). A user-facing global toggle does not exist in
# the current UI; if added in a future phase, this reset must be scoped.
if ($prefs->get('disableDiscovery')) {
    ...
    $prefs->set('disableDiscovery', 0);
}
```

### WR-02: `was_paused` set to `true` on `Stopped` event — spurious `resume` after end-of-track

**File:** `librespot-spoton/src/connect.rs:212-217`
**Issue:** The `Stopped` handler (lines 212–217) sets `was_paused.store(true, Ordering::Release)` just like the `Paused` handler. This means that after a track ends (Stopped fires), if the next track's `Playing` event arrives with the same track ID (unlikely but possible in a replay scenario), `was_paused.swap(false)` returns `true` and a spurious `resume` notification is sent to LMS. More importantly, the flag name `was_paused` semantically means "player was paused, expect a resume", but a `Stopped` state at end-of-track is not a pause — it is track completion. Mapping Stopped to `was_paused=true` creates a latent logical inconsistency. The `Stopped` comment at line 195 explicitly calls it "the authoritative end-of-track signal", yet it then sets the same flag as `Paused`.

**Fix:** Do not set `was_paused=true` on `Stopped`. The `Stopped` handler should only emit `stop` and leave `was_paused=false`:
```rust
PlayerEvent::Stopped { .. } => {
    log::debug!("[spoton] Stopped: current_track={:?}", current_track.as_deref());
    if current_track.is_some() {
        // Do NOT set was_paused — Stopped is end-of-track, not a pause.
        self.notify("stop", "", "").await;
    }
}
```

### WR-03: Discovery toggle save in `Settings.pm` calls `initHelpers()` but the discovery flag is saved AFTER the `initHelpers()` call

**File:** `Plugins/SpotOn/Settings.pm:147-164`
**Issue:** The per-player settings save block (lines 145–165) calls `Plugins::SpotOn::Connect::DaemonManager->initHelpers()` at line 150 — BEFORE the `disableDiscovery` pref is saved at line 163. This means the daemon restart triggered by `initHelpers()` reads the OLD value of `disableDiscovery`, not the newly submitted value. The Discovery toggle change takes effect only on the NEXT `initHelpers()` call (the 60-second watchdog), not immediately on save.

The `enableSpotifyConnect` toggle is saved at line 147, before `initHelpers()` at line 150, so it takes effect immediately. The `disableDiscovery` pref at line 163 is after — the ordering is inconsistent.

**Fix:** Save the `disableDiscovery` pref before calling `initHelpers()`:
```perl
if ($client) {
    my $enableConnect = $paramRef->{'pref_enableSpotifyConnect'} ? 1 : 0;
    $prefs->client($client)->set('enableSpotifyConnect', $enableConnect);

    # OGG-passthrough override
    if (defined $paramRef->{'pref_connectOggOverride'}) {
        my $override = $paramRef->{'pref_connectOggOverride'};
        $override = 'auto' unless $override =~ /^(?:auto|ogg|pcm)$/;
        $prefs->client($client)->set('connectOggOverride', $override);
    }

    # Per-player Discovery toggle — save BEFORE initHelpers() reads it
    my $disableDiscovery = $paramRef->{'pref_enableDiscovery'} ? 0 : 1;
    $prefs->client($client)->set('disableDiscovery', $disableDiscovery);

    require Plugins::SpotOn::Connect::DaemonManager;
    Plugins::SpotOn::Connect::DaemonManager->initHelpers();
}
```

## Info

### IN-01: FNV-1a device_id is 16 hex chars; librespot default is a UUID (36 chars) — no validation but functionally safe

**File:** `librespot-spoton/src/connect.rs:857-864`
**Issue:** The FNV-1a hash produces a 16-character hex string (64-bit value, `{:016x}` format). The librespot default `SessionConfig::default()` generates a UUID v4 in hyphenated form (36 characters, e.g. `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`). Neither `SessionConfig` nor `Discovery::builder()` validate the format or length of the device_id string — it is passed through as-is to the Spotify protocol. The Spotify protocol does not formally require a UUID format (it accepts any opaque string), and the librespot-discovery source (`server.rs:94`) embeds it verbatim in the ZeroConf JSON response.

This is functionally safe and matches the fix objective. Documenting the deliberate format divergence from the librespot default would help future maintainers.

**Fix:** Add a comment at the device_id computation (line 854):
```rust
// FNV-1a 64-bit hash, formatted as 16 lowercase hex chars.
// Deliberately shorter than librespot's default UUID (36 chars) —
// Spotify accepts any opaque string here; no format is mandated.
```

### IN-02: `_isDiscoveryEnabled` helper in `DaemonManager.pm` is defined but never called

**File:** `Plugins/SpotOn/Connect/DaemonManager.pm:36-42`
**Issue:** `_isDiscoveryEnabled($client)` is added as a plan artifact (05.4-02-PLAN.md key_links) and defined at lines 36–42, but is not called anywhere in `DaemonManager.pm` or the rest of the codebase. The discovery flag evaluation happens directly in `Daemon.pm::start()` (lines 114–117) via inline logic, duplicating the helper's logic without using the helper. Dead code that duplicates logic is a maintenance hazard — if the priority rules change, two places must be updated.

**Fix:** Either call `_isDiscoveryEnabled` from `Daemon.pm::start()` (requires making the helper accessible, e.g., exporting it or calling it via `Plugins::SpotOn::Connect::DaemonManager::_isDiscoveryEnabled($client)`), or remove the helper and document that the flag evaluation lives only in `Daemon.pm::start()`.
