<task type="auto">
  <name>Task 2: Create GitHub release with plugin zip, finalize repo.xml, make repo public</name>
  <files>repo.xml</files>
  <action>
  1. Build the release zip. The zip must contain the directory structure `Plugins/SpotOn/` at the top level (LMS expects this layout when unpacking into InstalledPlugins/).

     From the repo root:
     ```
     zip -r SpotOn-v1.1.0.zip Plugins/SpotOn/ -x "Plugins/SpotOn/.git*"
     ```

     This creates a zip containing the full plugin tree including all Bin/ subdirectories with platform binaries.

  2. Compute SHA1 of the zip:
     ```
     sha1sum SpotOn-v1.1.0.zip
     ```
     Record the hex digest.

  3. Create a GitHub release v1.1.0 with the zip as an asset:
     ```
     gh release create v1.1.0 SpotOn-v1.1.0.zip --title "v1.1.0" --notes "v1.1 Hardening & Reach: Multi-arch binaries, stream metadata, EN code cleanup, production deployment."
     ```

  4. The download URL for the asset will be:
     `https://github.com/stiefenm/spoton/releases/download/v1.1.0/SpotOn-v1.1.0.zip`

  5. Update repo.xml: replace the PLACEHOLDER values in the plugin element:
     - `sha="PLACEHOLDER_SHA1_PHASE_6_1"` with the actual SHA1 hex digest from step 2
     - `url="PLACEHOLDER_URL_PHASE_6_1"` with `https://github.com/stiefenm/spoton/releases/download/v1.1.0/SpotOn-v1.1.0.zip`
     Also update the XML comment at the top to remove the placeholder notes — replace with a simple maintainer note about updating SHA and URL when publishing new versions.

  6. Commit the finalized repo.xml: "chore(09.5): finalize repo.xml with v1.1.0 release SHA and URL"

  7. Push to GitHub: `git push origin main`

  8. Make the repository public:
     ```
     gh repo edit stiefenm/spoton --visibility public
     ```

  9. Verify the raw repo.xml URL is accessible:
     ```
     curl -sI https://raw.githubusercontent.com/stiefenm/spoton/main/repo.xml
     ```
     Expect HTTP 200.

  10. Clean up the local zip file: `rm SpotOn-v1.1.0.zip`

  NOTE: The release zip SHA is computed BEFORE pushing, but the release is created from the current HEAD. The order is: build zip, compute SHA, create release (uploads zip), update repo.xml with SHA/URL, commit, push. The repo.xml commit happens after the release, which is fine because repo.xml references the release asset URL (which is stable once created).

  Actually, correct order to avoid issues:
  a. Push current state first (so the tag has a valid commit)
  b. Build zip from working tree
  c. Compute SHA1
  d. Create release, upload zip
  e. Update repo.xml with SHA and URL
  f. Commit and push repo.xml update
  g. Make repo public
  </action>
  <verify>
    <automated>cd /home/sti/spoton && gh repo view stiefenm/spoton --json isPrivate --jq '.isPrivate' | grep -q 'false' && echo "repo public" && gh release view v1.1.0 --json assets --jq '.assets[].name' | grep -q 'SpotOn-v1.1.0.zip' && echo "release asset ok" && grep -v '^#' repo.xml | grep -c 'PLACEHOLDER' | grep -q '^0$' && echo "no placeholders" && curl -sI https://raw.githubusercontent.com/stiefenm/spoton/main/repo.xml 2>/dev/null | head -1 | grep -q '200' && echo "repo.xml accessible"</automated>
  </verify>
  <done>GitHub release v1.1.0 exists with SpotOn-v1.1.0.zip asset. repo.xml has real SHA1 and download URL. Repo is public. Raw repo.xml URL returns HTTP 200.</done>
