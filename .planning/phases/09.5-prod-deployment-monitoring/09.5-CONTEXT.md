<decisions>
## Implementation Decisions

### Repository Visibility
- Repo `stiefenm/spoton` goes public
- Internal files excluded via `.gitignore` + `git rm --cached` (pragmatic approach — history stays, current state is clean)
- Files to exclude: `.planning/`, `.claude/`, `CLAUDE.md`, `REQUIREMENTS.md`, `claude-spotty.sudoers`, `test_keymaster_*.py`, `test_pkce_token.py`, `librespot-spoton/target/`

### Plugin Packaging
- `install.xml` must be created (does not exist yet)
- `repo.xml` exists with placeholders — needs real SHA and download URL
- Version: 1.1.0 (aligns with v1.1 milestone)
- Download mechanism: GitHub release zip or raw GitHub tarball

### Target Environment
- Raspberry Pi at 192.168.13.5, SSH as user `sti`
- aarch64 Debian 12 bookworm, hostname `lyrion`
- LMS runs as `squeezeboxserver` user, plugins in `/var/lib/squeezeboxserver/cache/InstalledPlugins/`
- Logs at `/var/log/squeezeboxserver/server.log`

### Monitoring
- Reuse existing cron infrastructure from spotty-monitor (`/etc/cron.d/`, daily log files, 30-day retention)
- Update grep patterns from Spotty keywords to SpotOn keywords
- Script at `/usr/local/bin/spoton-monitor.sh`, logs at `/var/log/squeezeboxserver/spoton-monitor/`

### Cleanup
- Remove `/home/sti/Spotty-Plugin/` directory
- Remove `/usr/local/bin/spotty-monitor.sh`
- Remove `/etc/cron.d/spotty-monitor`
- Remove `/var/log/squeezeboxserver/spotty-monitor/` (old logs)
- Leave Spotty plugin in LMS installed state alone (user can uninstall via UI)

### Claude's Discretion
- install.xml field values beyond what's specified
- Exact grep patterns for SpotOn monitoring
- Whether to create a GitHub release or use raw tarball URL
- Order of operations for deployment steps

</decisions>
