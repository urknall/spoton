<objective>
Add change detection to the CI release workflow so that Rust cross-compilation is
skipped when a tag push contains only plugin file changes (Perl, strings, HTML, config).
When Rust source has not changed, the workflow downloads binaries from the previous
GitHub release and repackages them into the new release zip. Full Rust builds still
trigger for Rust source changes and for manual workflow_dispatch runs.

Purpose: Reduce plugin-only release time from ~8 minutes to under 1 minute.
Output: Updated `.github/workflows/build-librespot.yml` with conditional build logic.
</objective>

Modify `.github/workflows/build-librespot.yml` to add conditional Rust build skipping. The changes add three new jobs and modify two existing ones. Here is the exact design:

**1. New job: `detect-changes` (runs first, before everything else)**

This job determines whether the Rust source changed between this tag and the previous tag.

- runs-on: ubuntu-latest
- outputs: `rust_changed` (string "true" or "false")
- Steps:
  a. Checkout repository with `fetch-depth: 0` (need full history for tag comparison)
  b. "Detect Rust source changes" step with `id: check`:
     - For `workflow_dispatch` events, always output `rust_changed=true` (force full build)
     - For tag push events:
       - Get the previous tag: `PREV_TAG=$(git tag --sort=-v:refname | grep '^v' | sed -n '2p')`
       - If no previous tag exists, output `rust_changed=true`
       - Otherwise run: `CHANGED=$(git diff --name-only "$PREV_TAG" "$GITHUB_REF_NAME" -- librespot-spoton/ | head -1)`
       - If CHANGED is non-empty, output `rust_changed=true`, else `rust_changed=false`
       - Log the decision with: `echo "Previous tag: $PREV_TAG"`, `echo "Rust changed: $RUST_CHANGED"`
  c. Set the output: `echo "rust_changed=$RUST_CHANGED" >> $GITHUB_OUTPUT`

**2. Modify existing `build` job:**
- Add `needs: [detect-changes]`
- Add `if: needs.detect-changes.outputs.rust_changed == 'true'`
- Everything else stays identical

**3. Modify existing `build-macos` job:**
- Add `needs: [detect-changes]`
- Add `if: needs.detect-changes.outputs.rust_changed == 'true'`
- Everything else stays identical

**4. Existing `lipo` job — unchanged:**
- Already has `needs: [build-macos]`, keep it
- The `lipo` job will be skipped automatically when `build-macos` is skipped (GitHub Actions skips downstream jobs when `needs` jobs are skipped)

**5. New job: `reuse-binaries` (runs only when Rust did NOT change)**

This job downloads all 8 platform binaries from the previous GitHub release and uploads them as artifacts with the exact same names as the build jobs produce.

- runs-on: ubuntu-latest
- needs: [detect-changes]
- if: `needs.detect-changes.outputs.rust_changed == 'false'`
- permissions: contents: read (needs to read releases)
- env: same BINARY_NAME as top-level
- Steps:
  a. "Find previous release tag" step:
     - `PREV_TAG=$(gh release list --limit 10 --json tagName --jq '.[].tagName' | grep '^v' | head -1)`
     - `echo "Downloading binaries from $PREV_TAG"`
     - `echo "prev_tag=$PREV_TAG" >> $GITHUB_OUTPUT`
     - id: prev
     - env: GH_TOKEN: `${{ github.token }}`
  b. "Download binaries from previous release" step:
     - `gh release download "${{ steps.prev.outputs.prev_tag }}" -p 'spoton-*' -D prev-binaries/`
     - env: GH_TOKEN: `${{ github.token }}`
  c. "Upload as build artifacts" step — a series of upload-artifact steps, one per platform, to match the exact artifact names the build jobs produce. The release assets have platform suffixes, so map them back:
     - For each Linux platform and Windows:
       Create the directory, copy the binary, upload with matching artifact name.
       The mapping (release asset name -> artifact name -> local path):
       - `spoton-x86_64-linux` -> artifact `spoton-x86_64-linux` -> `spoton`
       - `spoton-aarch64-linux` -> artifact `spoton-aarch64-linux` -> `spoton`
       - `spoton-armhf-linux` -> artifact `spoton-armhf-linux` -> `spoton`
       - `spoton-arm-linux` -> artifact `spoton-arm-linux` -> `spoton`
       - `spoton-i386-linux` -> artifact `spoton-i386-linux` -> `spoton`
       - `spoton-x86_64-win64.exe` -> artifact `spoton-x86_64-win64` -> `spoton.exe`
       - `spoton-darwin` -> artifact `spoton-darwin` -> `spoton`

     Use a single "Prepare artifacts" step that creates directories for each platform and copies/renames:
     ```
     mkdir -p artifacts/{x86_64-linux,aarch64-linux,armhf-linux,arm-linux,i386-linux,x86_64-win64,darwin}
     cp prev-binaries/spoton-x86_64-linux artifacts/x86_64-linux/spoton
     cp prev-binaries/spoton-aarch64-linux artifacts/aarch64-linux/spoton
     cp prev-binaries/spoton-armhf-linux artifacts/armhf-linux/spoton
     cp prev-binaries/spoton-arm-linux artifacts/arm-linux/spoton
     cp prev-binaries/spoton-i386-linux artifacts/i386-linux/spoton
     cp prev-binaries/spoton-x86_64-win64.exe artifacts/x86_64-win64/spoton.exe
     cp prev-binaries/spoton-darwin artifacts/darwin/spoton
     chmod 755 artifacts/*/spoton*
     ```

     Then use 7 separate `actions/upload-artifact@v4` steps, one per platform:
     - name: spoton-x86_64-linux, path: artifacts/x86_64-linux/spoton
     - name: spoton-aarch64-linux, path: artifacts/aarch64-linux/spoton
     - name: spoton-armhf-linux, path: artifacts/armhf-linux/spoton

- The rest of the release job stays identical — it downloads artifacts by name, and the artifact names are the same whether produced by `build`+`lipo` or by `reuse-binaries`.

- The `reuse-binaries` job uploads `spoton-darwin` directly (the Universal Binary from the previous release), skipping the per-arch intermediates entirely
- Keep all existing env vars at the top level unchanged

|----------|-------------|
| Tag push -> CI | Untrusted tag content could manipulate change detection |
| Previous release -> Current release | Reused binaries come from a prior release we authored |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16.1-01 | Tampering | detect-changes git diff | accept | Tags are signed/pushed by repo owner; CI runs in GitHub-controlled environment. An attacker who can push tags already controls the repo. |
| T-16.1-02 | Tampering | reuse-binaries gh release download | accept | Binaries come from our own GitHub releases in the same repo. GitHub verifies release integrity. No third-party source. |
| T-16.1-03 | Denial of Service | detect-changes prev tag lookup | mitigate | If no previous tag exists, force full build (rust_changed=true). Prevents release failure on first-ever tag. |
| T-16.1-04 | Information Disclosure | workflow logs | accept | Tag names and changed file paths are logged but contain no secrets. Standard CI log visibility. |
