<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OPML client -> Plugin.pm | showUri arrives via OPML passthrough from _showItem; user cannot directly inject but tampered LMS state could send arbitrary strings |
| Plugin.pm -> Spotify API | showUri is forwarded to PUT/DELETE/GET /me/library via Client.pm; malformed URI would cause 400 from Spotify |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-01 | Tampering | SpotOnManageFollow showUri param | mitigate | Validate showUri against regex `^spotify:show:[A-Za-z0-9]+$` before any API call (same pattern as SpotOnManageLike trackUri validation on Plugin.pm line 430) |
| T-20-02 | Information Disclosure | checkShows response | accept | Boolean array only reveals follow state for the requesting user's own library; no cross-user data exposure |
| T-20-03 | Denial of Service | Rapid follow/unfollow toggling | accept | Each action makes one API call; Spotify rate limiting (429 + Retry-After) and existing Client.pm throttle handle burst protection. Cache prevents redundant checkShows calls within 60s window. |
| T-20-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this phase — zero supply chain surface |
</threat_model>
