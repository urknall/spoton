<objective>
Add rapid-skip debounce to unified.rs so that when LMS sends multiple
GET /track/{id} requests in rapid succession (user pressing Next quickly),
the previous in-flight Player::load is aborted before the new one begins.

Purpose: When a user skips through tracks rapidly, LMS issues a new
/track/{id} request before the previous track's audio-key fetch completes.
Without debounce, multiple Player::load() calls pile up concurrently against
the Spotify AP — triggering the audio-key throttle (HTTP 429) that causes
the "no audio after rapid skip" bug. The unified daemon already serialises
Browse/Connect mode via ActiveMode mutex, but concurrent Browse-on-Browse
requests slip through because each spawns its own task.

Mechanism: Add an Arc<AtomicU64> browse_abort_gen counter to unified_http_server.
Each /track/{id} request captures the current generation value before spawning
serve_track_request. serve_track_request checks the generation after Player::load
returns (or on the first PlayerEvent) — if the generation has advanced, it drops
pcm_tx early (causing EOF to LMS) instead of streaming audio.

Output: unified.rs with browse_abort_gen AtomicU64 shared across /track/ requests.
</objective>

Current /track/{id} handler flow (unified.rs lines ~438-561 before this plan):

1. Validate track_id ([A-Za-z0-9]+)
2. Parse ?start_position=N
3. Lock mode_state, transition to Browse(track_id), optionally pause Spirc
4. Create (pcm_tx, pcm_rx) channel
5. Clone browse_cancel notify, spawn cancel_task to drop pcm_tx on notification
6. Spawn async task: call serve_track_request(&track_id, session, pcm_tx, start_position_ms)
   - serve_track_request() creates a new Player, calls player.load(), streams PCM via pcm_tx
7. tokio::time::timeout(500ms) for early 404 from status_tx
8. Build StreamBody from pcm_rx, return 200 response

serve_track_request() signature (browse.rs, exported, used by unified.rs):
  pub async fn serve_track_request(
      track_id: &str,
      session: Session,
      pcm_tx: mpsc::Sender<Bytes>,
      start_position_ms: u32,
  ) -> StatusCode

The abort mechanism: adding an Arc<AtomicU64> browse_abort_gen to unified_http_server
shared state. Each /track/ request increments the counter (fetch_add) before spawning
its task. The spawned task captures the pre-increment value. After Player::load() is
called (or when the first PlayerEvent arrives), check if current gen != captured gen.
If yes: another request arrived — drop pcm_tx immediately and return without streaming.

Implementation options for the check point:
  Option A: Pass abort_gen into serve_track_request as an additional parameter.
            serve_track_request checks it after player.load() in the event loop.
  Option B: Check abort_gen in the spawned task in unified.rs BEFORE calling
            serve_track_request (pre-load abort), and AFTER it returns (post-load abort).

Option B is simpler — serve_track_request signature stays unchanged. The check
in the spawned task after serve_track_request returns (or before it starts) is
sufficient because Player::load() completes async after the audio-key fetch;
if a new request arrived while we were fetching the key, gen will have advanced.

For the pre-call check (before Player::load begins): check gen immediately after
acquiring mode_state. If already advanced, drop pcm_tx and return without calling
serve_track_request at all.

For the mid-stream check: the cancel mechanism via browse_cancel.notify_waiters()
already handles Connect preemption. For Browse-on-Browse preemption, incrementing
gen and checking it in the cancel_task is the cleanest approach — when a new
/track/ request arrives and increments gen, the old cancel_task sees gen changed
and drops pcm_tx_for_cancel, causing EOF on the old connection.
-->

    Before calling serve_track_request inside the tokio::spawn, check if another
    request has already superseded us:
      let current_gen = browse_abort_gen_task.load(Ordering::SeqCst);
      if current_gen != my_gen + 1 {
          // Another request arrived; drop pcm_tx and exit without streaming
          drop(pcm_tx);
          let _ = status_tx.send(StatusCode::NOT_FOUND);
          return;

    (Note: fetch_add returns the OLD value, so after MY increment the counter is my_gen+1.
    If another request has arrived and incremented again, current_gen > my_gen+1.)

    only goes up and our own increment set it to my_gen+1.

    Step 5 — Post-load EOF on supersession:
