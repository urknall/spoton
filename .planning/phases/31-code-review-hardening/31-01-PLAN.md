If that channel's recv loop exits (e.g., because the Player's internal event sender closed during the Spirc shutdown/reconnect cycle), LMS notifications stop permanently.

Output: unified.rs with event dispatcher respawn logic in the ZeroConf reconnect path.

3. In the ZeroConf reconnect success path (after the new Spirc is stored in spirc_handle, around line 1175), add event dispatcher respawn:
   a. Abort the old dispatcher: if let Some(ref h) = event_dispatcher_handle { h.abort(); }
   b. Get a fresh event channel from the existing Player: let mut event_chan = connect_player_opt.as_ref().unwrap().get_player_event_channel();
   c. Clone all the Arc handles needed by the dispatcher (mode_state, spirc_active, browse_cancel, and lms if configured).
   d. Spawn the new dispatcher task with the same logic as the initial spawn (lines 937-978 for LMS-configured case, 982-1007 for non-LMS case). Use lms_for_reconnect.is_some() to pick the branch.
   e. Store the new JoinHandle: event_dispatcher_handle = Some(tokio::spawn(...));
   f. Log: log::info!("[spoton/unified] Event dispatcher respawned after reconnect");

4. Also respawn in the "Spirc task died" reconnect path (around line 1250, after reconnect succeeds): same abort + respawn pattern.

5. Also respawn in the Browse-triggered session reconnect path (around line 1202): after the reconnect succeeds and a new Spirc is created, abort old dispatcher and spawn new one.

Key constraint: The LMS instance used inside the dispatcher closure must be cloned from the original lms (not lms_for_reconnect, which has flush_tx=None). Use the same lms variable that was used in the initial spawn. To make this work, clone the LMS instance into a variable that persists across the event loop iterations. Since LMS::clone() drops flush_tx, and the initial spawn at line 937 uses the original lms (with flush_tx), store a second clone of the original lms specifically for reconnect dispatchers. Name it lms_for_dispatcher. The flush_tx in the dispatcher is only used by handle_player_event for Seeked events — the cloned version still works because seek_gen and flush_tx are Arced in the original. Actually, reviewing LMS::clone() — flush_tx is set to None in the clone. The initial dispatcher uses the original lms which has flush_tx. For the respawned dispatcher, use lms.clone() which will have flush_tx=None — this means seek-flush signals will not fire from the reconnected dispatcher. This is acceptable because the UnifiedHttpStreamSink has its own flush_tx and the relay task watches flush_rx independently. The Seeked event still sends the "seek" notification to LMS (that goes through notify(), which is unaffected by flush_tx). So lms.clone() (flush_tx=None) is correct for the dispatcher.

Do NOT refactor the dispatcher into a separate function — the closure captures too many local variables. Instead, duplicate the spawn block inline in the reconnect path. This is the same pattern used for current_spirc_task.
