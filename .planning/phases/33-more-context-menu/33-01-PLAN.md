Extend the "More" context menu (Track Info menu) for SpotOn tracks and episodes to show Artist View, Album View, and Like/Unlike -- matching Spotty's feature set and resolving GitHub Issue #29.

Purpose: Currently the More menu only shows a single Like/Unlike item. Users expect navigation to artist/album pages from Now Playing, which is standard in Spotty and other LMS plugins.

Output: Updated Plugin.pm (trackInfoMenu rewrite + registerInfoProvider fix), ProtocolHandler.pm (cache extension + trackInfoURL update), Connect.pm (cache extension).
</objective>

  <name>Task 1: Extend metadata cache with artist/album/show IDs</name>
  <files>Plugins/SpotOn/Plugin.pm, Plugins/SpotOn/ProtocolHandler.pm, Plugins/SpotOn/Connect.pm</files>
  <action>
Add artistId, albumId, and (for multi-artist) an artistIds field to all metadata cache write sites. These fields enable trackInfoMenu to build navigation links without a live API call.

In Plugin.pm _trackItem (around line 737, the cache->set call): add three fields to the cache hash:
- artistId: $track->{artists}[0]{id} (primary artist, undef if missing)
- artistIds: a JSON-encoded string of [{id => $id, name => $name}, ...] from @{$track->{artists}} -- use JSON::XS::encode_json. Only include artists that have both id and name.
- albumId: $track->{album}{id} (undef if missing)

In ProtocolHandler.pm _cacheExplodedTrack (line 786): add the same three fields. The function receives $track which has $track->{artists} array and $track->{album}. Extract:
- artistId: $track->{artists}[0]{id}
- artistIds: JSON::XS::encode_json of [{id, name} pairs]
- albumId: the album ID is NOT directly available here (function receives $albumName, $albumCover but not album ID). Add a fifth parameter $albumId to the function signature. Update all callers of _cacheExplodedTrack to pass the album ID. There are two call sites: search grep for "_cacheExplodedTrack(" to find them -- one in explodePlaylist and one elsewhere. The album ID is available in $track->{album}{id} at those call sites.

In ProtocolHandler.pm _cacheExplodedEpisode (line 798): add two fields:
- showId: ($ep->{show} || {})->{id}
- showName: ($ep->{show} || {})->{name} (already computed as artist field but store explicitly)

In ProtocolHandler.pm _asyncRefetch callback (around line 894): add the same fields to the %new_meta hash. For tracks: artistId from $info->{artists}[0]{id}, artistIds from JSON-encoded artist pairs, albumId from ($info->{album} || {})->{id}. For episodes: showId from ($info->{show} || {})->{id}, showName from ($info->{show} || {})->{name}.

In Connect.pm _fetchTrackMetadata (around line 957, the cache->set call): add artistId from $trackInfo->{artists}[0]{id}, artistIds from JSON-encoded artist pairs, albumId from ($trackInfo->{album} || {})->{id}.

Import note: JSON::XS is already available (used elsewhere in the codebase). Add `use JSON::XS qw(encode_json)` to ProtocolHandler.pm if not already imported. Plugin.pm and Connect.pm should already have it -- verify with grep before adding.

<task type="auto">
  <name>Task 2: Rewrite trackInfoMenu and trackInfoURL for full context menu</name>
  <files>Plugins/SpotOn/Plugin.pm, Plugins/SpotOn/ProtocolHandler.pm</files>
  <action>
Three changes: fix registerInfoProvider positioning, rewrite trackInfoMenu to return a full array, and update trackInfoURL to match.

1. Plugin.pm line 167 -- registerInfoProvider: add `after => 'top'` to the registration so SpotOn items appear after standard LMS providers (not before or in arbitrary position):
   Slim::Menu::TrackInfo->registerInfoProvider( spotonTrackInfo => (
       after => 'top',
       func  => \&trackInfoMenu,
   ) );

2. Plugin.pm -- Rewrite trackInfoMenu (currently lines 464-482) to return an ARRAYREF of items instead of a single hash. The function signature stays the same: ($client, $url, $track, $remoteMeta, $tags).

   Guard logic: match both track and episode URIs. Extract the Spotify ID and type:
   - spoton://track:ID or spoton:track:ID -> type=track, id=ID
   - spoton://episode:ID or spoton:episode:ID -> type=episode, id=ID
   Return undef if neither matches.

   Get accountId via _getAccountId($client). Return undef if no account.

   Read the metadata cache: $cache->get('spoton_meta_' . md5_hex($url)). The cache now contains artistId, artistIds, albumId (for tracks) or showId, showName (for episodes).

   Build an @items array:

   For tracks (type eq 'track'):
   - If cache has artistId: push Artist View item with url => \&_artistFeed, passthrough => [{ artistId => $meta->{artistId} }], type => 'link'
   - If cache has albumId: push Album View item with url => \&_albumFeed, passthrough => [{ albumId => $meta->{albumId} }], type => 'link'
   - Push Like/Unlike item (same as current): name => cstring($client, 'PLUGIN_SPOTON_MANAGE_LIKE'), url => \&SpotOnManageLike, passthrough => [{ trackUri => "spotify:track:$id", accountId => $accountId }], favorites => 0

   For episodes (type eq 'episode'):
   - If cache has showId: push View Show item with name => cstring($client, 'PLUGIN_SPOTON_SHOW_VIEW'), url => \&_showFeed, passthrough => [{ showId => $meta->{showId}, showName => $meta->{showName} }], type => 'link'
   - Build show URI from showId: "spotify:show:$meta->{showId}"
   - Push Follow/Unfollow item: name => cstring($client, 'PLUGIN_SPOTON_MANAGE_FOLLOW'), url => \&SpotOnManageFollow, passthrough => [{ showUri => $showUri, accountId => $accountId }], favorites => 0

   Return \@items (arrayref). If @items is empty, return undef.

   IMPORTANT: LMS registerInfoProvider expects the func to return either a single hashref (one item) or an arrayref of hashrefs (multiple items). Returning an arrayref is the correct approach -- Spotty does this too.

3. ProtocolHandler.pm -- Update trackInfoURL (lines 758-784) to build the same set of items as trackInfoMenu. The function receives ($class, $client, $url).

   Keep the existing guard: extract trackId from URL. Also add episode matching.

   Read cache via getMetadataFor or directly via cache->get('spoton_meta_' . md5_hex($url)).

   Build @items the same way as trackInfoMenu above (Artist View if artistId, Album View if albumId, Like/Unlike for tracks; View Show + Follow for episodes).

   Return the existing OPML wrapper: { name => $meta->{title} || $url, type => 'opml', items => \@items }.

   Do NOT require Plugins::SpotOn::Plugin for _artistFeed/_albumFeed/_showFeed -- these are already in Plugin.pm's namespace. Use full package paths: \&Plugins::SpotOn::Plugin::_artistFeed, \&Plugins::SpotOn::Plugin::_albumFeed, \&Plugins::SpotOn::Plugin::_showFeed. Same for SpotOnManageLike and SpotOnManageFollow. Note: the existing code already does this for SpotOnManageLike (line 773).

   Update the comment block above trackInfoMenu to document the new return format.
