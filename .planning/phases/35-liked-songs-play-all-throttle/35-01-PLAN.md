<objective>
Fix GitHub Issue #51: Playing all Liked Songs (1633 tracks) causes Browse daemon 404s and song skips due to concurrent API load from Material Skin browse requests + _fetchAllPages play-all pagination hitting Spotify simultaneously.

Purpose: Eliminate ~42 redundant me/tracks API calls by populating $_playAllItemCache incrementally after each page, so concurrent browse requests are served from the growing cache instead of making independent API calls.

Output: Modified Plugin.pm with onPage callback in _fetchAllPages and incremental cache wiring in _savedTracksFeed.

  <action>
  Three code locations in Plugin.pm need modification:

  **Location 1: _fetchAllPages (line 1320) — add onPage callback support**

  After extracting $done from $args (line 1329), also extract an optional onPage callback:
  my $onPage = $args->{onPage};

  After the existing `push @accumulated, @{$items};` line (between line 1345 and the total check), insert:
  $onPage->($items) if $onPage;

  This passes the CURRENT PAGE's raw items (not the full accumulator) to the callback. The caller handles mapping and caching. No other changes to _fetchAllPages — done callback, recursion, and T-25-01 guard all remain unchanged.

  **Location 2: _savedTracksFeed play-all branch (line 1185) — wire onPage for incremental caching**

  Before the _fetchAllPages call (after the $cacheKey assignment at line 1183), initialize the cache entry:
  $_playAllItemCache{$cacheKey} = { items => [], ts => time(), loading => 1 };

  Add an onPage parameter to the _fetchAllPages call (after extractItems, before done):
  onPage => sub {
      my ($pageItems) = @_;
      my @mapped = map  { _trackItem($client, $_->{track}) }
                   grep { defined $_->{track} }
                   @{$pageItems};
      push @{ $_playAllItemCache{$cacheKey}{items} }, @mapped;
      $_playAllItemCache{$cacheKey}{ts} = time();
  },

  This incrementally maps each page's raw items to OPML items via _trackItem and appends them to the cache. The ts is refreshed each page so _evictPlayAllCache never evicts a loading entry.

  Modify the done callback (line 1195) to finalize from the cache instead of re-mapping all items:
  done => sub {
      my $cached = $_playAllItemCache{$cacheKey};
      if ($cached) {
          delete $cached->{loading};
          $cached->{ts} = time();
      }
      my @items = $cached ? @{ $cached->{items} } : ();
      if (!@items) {
          push @items, { name => cstring($client, 'PLUGIN_SPOTON_NO_RESULTS'), type => 'textarea' };
          $_playAllItemCache{$cacheKey} = { items => \@items, ts => time() };
      }
      $callback->({ items => \@items });
  },

  This avoids remapping all 1633 items a second time — the cache already has them from onPage. The loading flag is removed to signal completion.

  **Location 3: _savedTracksFeed cache-check branch (line 1207) — guard against eviction during loading**

  The current code at lines 1214-1216:
      delete $_playAllItemCache{$cacheKey};
      goto &_savedTracksFeed;

  Replace with a loading-aware guard. If loading is still true (play-all pagination in progress), do NOT delete the cache — instead fall through to a direct single-page API call for this specific offset. If loading is false or absent (fully loaded but expired/out-of-range), keep the existing delete + goto behavior:

  if ($cached->{loading}) {
      # Still loading — serve this request via direct API without evicting the growing cache
      Plugins::SpotOn::API::Client->getSavedTracks($accountId, {
          offset => $offset,
          limit  => $limit,
      }, sub {
          my $data = shift;
          unless ($data) {
              $callback->({ items => [{ name => cstring($client, 'PLUGIN_SPOTON_NO_RESULTS'), type => 'textarea' }] });
              return;
          }
          my @items = map { _trackItem($client, $_->{track}) } @{ $data->{items} || [] };
          $callback->({ items => \@items, offset => $offset, total => $data->{total} });
      });
  } else {
      delete $_playAllItemCache{$cacheKey};
      goto &_savedTracksFeed;
  }

  This ensures the one edge case — browse request for an offset not yet accumulated (user scrolled way ahead during loading) — gets served via a normal API call without destroying the incremental cache that _fetchAllPages is actively building.
